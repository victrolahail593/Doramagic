  Since v1.2.2, set_key() and unset_key() don't follow symlinks by default.
  This is a security fix to prevent writing to untrusted paths via symlinks.
context: |
  Previously, a symlink attack could modify arbitrary files. Now you must explicitly
  use follow_symlinks=True to restore old behavior.
do:
  - "Use follow_symlinks=True only if you trust the .env file location"
  - "Be aware of symlinks in your project directory"
dont:
  - "Assume symlinks are followed like in older versions"
  - "Create .env symlinks in untrusted directories"

  - "src/dotenv/main.py: set_key(), unset_key()"

## 真实场景
攻击者在 /tmp 创建一个指向 /etc/passwd 的符号链接 .env。你的程序调用 set_key()
想创建一个 .env 文件，但因为工作目录在 /tmp，它意外修改了 /etc/passwd。
升级到 v1.2.2 后，这个攻击被阻止了，但如果代码依赖旧的 symlink 行为就需要改代码。

## 影响范围
- 谁会遇到：使用符号链接的项目的开发者
- 什么时候：升级到 v1.2.2+ 后
- 多严重：旧代码可能失败，但安全风险被消除
