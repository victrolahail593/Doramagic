# WHY 提取改进策略 Brainstorm (Gemini)

## 核心理念

您当前的系统本质上是一个“代码考古”系统，通过分析代码（遗迹）来推断设计意图（文明）。这是一个非常好的起点。我的建议将围绕以下几个方面展开，旨在引入新的信息源和分析维度，将“考古”升级为“社会学+考古”的综合研究。

1.  **从“静态快照”到“动态演化”**：代码不是静态的，WHY 蕴含在变更历史和决策过程中。
2.  **从“代码优先”到“多源融合”**：代码是最终结果，但设计讨论、Issue、PR 提供了更直接的 WHY。
3.  **从“随机采样”到“智能导航”**：用启发式规则和图分析来定位“WHY 浓度”最高的代码区域。
4.  **从“单一模型”到“协同智能”**：系统化地利用不同模型的优势，构建“模型委员会”。

## 1. 我们可能遗漏了什么改进方向？

除了您提到的 P0-P3 优先级，我认为还有以下几个值得探索的盲区：

### a. **作者画像分析 (Author Persona Analysis)**

**思路**：不同的开发者有不同的设计哲学。一个项目的核心贡献者（尤其是创始人）的决策权重远高于外围贡献者。通过分析 `git blame` 和贡献者统计，我们可以构建核心作者的画像，并优先分析他们编写或重构的代码。

**价值**：
*   **高信噪比**：核心作者的代码更有可能体现项目核心的设计哲学。
*   **一致性**：可以发现特定作者贯穿始终的设计偏好（例如，偏好函数式编程、总是用特定设计模式等）。

### b. **演化轴分析 (Evolutionary Axis Analysis)**

**思路**：项目的 WHY 不是一成不变的。一个在 0.1 版本做出的决策，可能在 3.0 版本时因为技术栈升级或认知变化而被推翻。

**价值**：
*   **揭示“WHY NOT”**：通过对比主要版本（例如，v1.0 vs v2.0）的重大重构，我们可以发现“为什么我们不再那样做”，这和“为什么这样做”同样有价值。
*   **历史背景**：理解某个决策是受当时的技术限制（如 Python 2.x 语法），还是一个永恒的设计原则。

### c. **交互式/Agentic 提取**

**思路**：将单向的“Prompt -> WHYs”流程，升级为多步的交互式探索。LLM Agent 可以不仅仅是被动接收信息，而是可以主动“提问”。

**价值**：
*   **打破信息孤岛**：当 LLM 在分析 `A.py` 时发现一个奇怪的函数调用，它可以主动请求读取 `B.py`（被调用函数所在文件）或全局搜索该函数的用法，而不是在信息不足的情况下猜测。
*   **动态调整焦点**：Agent 可以根据初步发现，动态调整后续要分析的文件范围，模拟人类专家读代码的“顺藤摸瓜”过程。

## 2. Stage 0 采样策略具体怎么改？

“按文件大小 Top 3 × 前 150 行”的策略确实过于简单。以下是一些更智能的采样策略，可以组合使用，形成一个**分层采样漏斗**。

### a. **代码结构中心度分析 (Code Centrality Analysis)**

**思路**：一个文件被 import 的次数越多，它在项目中的“中心度”就越高，其设计决策影响力也越大。

**具体做法**：
1.  用 `grep` 或静态分析工具扫描整个项目，构建一个文件间的 import 依赖图。
2.  计算每个文件的“入度”（In-degree），即被多少其他文件 import。
3.  **优先采样入度最高的文件**，例如 `src/core/engine.py` 或 `vnpy/trader/engine.py`。这些通常是核心引擎或事件分发中枢。

### b. **代码变更热点分析 (Code Churn/Hotspot Analysis)**

**思路**：频繁变更的文件往往是业务逻辑的核心或正在被积极重构的地方，这两者都是 WHY 的富矿。

**具体做法**：
1.  使用 `git log --stat --since="1 year ago"` 等命令分析提交历史。
2.  **优先采样变更行数最多、提交次数最频繁的文件**。这些“热点”文件揭示了项目维护的重心和演进方向。

### c. **“非代码”资产优先采样 (Non-Code Asset First)**

**思路**：很多顶层设计决策体现在配置文件、CI/CD脚本和项目脚手架中。

**具体做法**：
*   **配置文件**：`pyproject.toml`, `setup.cfg` -> 依赖管理、项目打包策略。
*   **容器与部署**：`Dockerfile`, `docker-compose.yml` -> 运行时环境、服务依赖关系、伸缩性设计。
*   **CI/CD 流程**：`.github/workflows/*.yml` -> 测试策略、质量门禁、构建与发布流程，这些都反映了对软件质量和工程效率的理念。
*   **测试目录**：`tests/` 目录下的 setup 和 mock 文件，往往能揭示被测模块的边界和依赖假设。

### d. **语义块采样 (Semantic Chunking)**

**思路**：放弃“前 150 行”，改为提取完整的、有语义的逻辑单元。

**具体做法**：
*   **类和函数**：使用 AST（抽象语法树）分析器，提取文件中的顶级类和函数作为基本单元，而不是按行数切割。
*   **接口与实现**：优先提取抽象基类（ABC）和它的几个具体实现，这直接揭示了“接口与实现分离”的设计思想。

## 3. 多模型融合的最佳实践？

利用 Sonnet 的深度和 Gemini 的广度互补，可以设计以下几种融合策略：

### a. **流水线模式 (Pipeline Model)**

**思路**：让不同模型在提取流程的不同阶段各司其职。
1.  **广度探索 (Gemini)**：利用 Gemini 的广度优势，执行 Stage 0 的智能采样。例如，让 Gemini 读取目录结构和所有文件名，输出一个“最可能包含 WHY 的 Top 10 文件”列表。
2.  **深度分析 (Sonnet)**：将 Gemini 筛选出的文件列表，交给 Sonnet 进行深入的代码分析和 WHY 提取。

### b. **生成-辩论-综合模式 (Generate-Debate-Synthesize)**

**思路**：让模型之间形成一种制衡和启发关系。
1.  **独立生成 (Generate)**：Sonnet 和 Gemini 并行，基于相同的输入，各自生成一份 WHYs 列表。
2.  **交叉质询 (Debate)**：将 Sonnet 的输出作为上下文喂给 Gemini，反之亦然。Prompt 指示模型：“请评估对方提出的 WHYs，并与你自己的发现进行对比。哪些是重合的？哪些是对方独有但你也认可的？哪些是你认为不准确的？”
3.  **最终综合 (Synthesize)**：将两份原始列表和交叉质询的结果，喂给一个最终的“仲裁者”模型（可以是更强大的模型），让它生成一份去重、合并、筛选后的最终版 WHYs 列表。

### c. **专家委员会模式 (Committee of Experts)**

**思路**：给不同模型分配不同的“专家角色”。
*   **架构师 (Architect)**：专注于分析项目结构、核心目录（`core`, `engine`）、配置文件。
*   **算法工程师 (Algorithmist)**：专注于分析具体实现、工具函数、性能敏感的代码。
*   **运维工程师 (DevOps)**：专注于分析 `Dockerfile`, `.github/workflows`, `scripts/`。

让不同角色的模型只分析自己领域相关的 Stage 0 输入，最后将各专家意见汇总。

## 4. 如何让评分更客观？

人工评分是黄金标准，但自动化是规模化的必由之路。

### a. **基于 Rubric 的 LLM 自动评分**

这是最直接的方法，关键在于 Prompt 设计：
*   **提供清晰的评分标准**：将您的 4 维度 Rubric 完整给到 LLM。
*   **提供正反案例 (Few-shot Learning)**：
    *   “这是一个信息增量得 2 分的例子，因为它揭示了...”。
    *   “这是一个证据锚定得 0 分的例子，因为它只提到了文件名，没有代码和行号...”。
*   **强制要求 LLM 给出评分理由**：要求 LLM 不仅给出分数，还要解释“为什么在‘因果完整性’上给 1 分”，这使得结果更可解释，也迫使 LLM 思考更深入。

### b. **可度量的代理指标 (Measurable Proxy Metrics)**

对于某些维度，可以用确定性程序来辅助评分：
*   **证据锚定**：写一个脚本，自动检查 WHY 中引用的 `file:line_number` 是否真实存在于代码库中。如果引用不存在或文件已删除，自动扣分。
*   **信息增量**：
    1.  将项目的 `README.md`、官方文档、以及相关框架（如 FastAPI, React）的文档内容向量化，存入向量数据库，构成“常识库”。
    2.  将生成的 WHY 文本向量化。
    3.  在“常识库”中搜索与 WHY 最相似的内容。如果相似度非常高，说明这条 WHY 可能是常识，信息增量低，应自动降分。

### c. **成对比较排名 (Pairwise Comparison Ranking)**

**思路**：绝对评分（0-2）对 LLM 来说很难，但相对比较（A 是否比 B 好）则容易得多。
1.  从一次提取的 5-8 条 WHYs 中，随机抽取两条（A 和 B）。
2.  问 LLM：“在‘提供新洞察’方面，A 和 B 哪个更好？为什么？”
3.  重复多次，使用类似 ELO 等级的算法为每条 WHY 计算一个相对排名。
4.  最终保留排名最高的几条。

## 5. 有没有完全不同的方法论？

绝对有。当前的方法是“读代码”，但真正的 WHY 往往存在于代码之外的“沟通”中。

### a. **Git 历史挖掘 (Git History Mining)**

**思路**：Commit Message 是开发者为自己留下的“为什么”的备忘录。
*   **筛选关键提交**：过滤掉“fix typo”等无用信息，重点分析包含关键词（如 `Refactor`, `Improve`, `Change`, `Add support for`）且变更行数较多的提交。
*   **分析提交内容**：将 `git show <commit_hash>` 的输出（包含 commit message 和 diff）作为一个整体喂给 LLM。Prompt: "从这次提交中，提取作者的设计决策和背后的原因。"

### b. **社会性代码分析 (Social Code Analysis)**

**思路**：GitHub 的 Issues 和 Pull Requests 是设计决策的辩论场，是 WHY 的金矿。
*   **挖掘高价值讨论**：筛选那些被标记为 `feature`, `enhancement` 或 `discussion`，且评论数量多、涉及核心贡献者的 Issues/PRs。
*   **提取决策过程**：将 PR 的描述、评论区的讨论、Code Review 的意见汇总，喂给 LLM。Prompt：“总结这个 PR 引入这项变更的原因，以及在讨论中被否决的其他方案是什么？最终选择该方案的权衡是什么？”

### c. **动态运行时分析 (Dynamic Runtime Analysis)**

**思路**：静态代码无法揭示性能瓶颈和资源消耗，而这往往是很多“丑陋”但高效代码的 WHY。
*   **有目标的性能剖析 (Targeted Profiling)**：在测试环境中，使用 `cProfile` (Python) 等工具运行核心用例，找到性能热点函数。
*   **分析热点代码**：将热点函数的代码和其性能数据一起提供给 LLM。Prompt：“这个函数是性能瓶颈。分析其代码，解释作者可能为了优化性能做了哪些设计决策？”

这些新方法论可以作为对现有流程的补充，甚至在某些情况下（如 funNLP 这种代码少的项目），可能成为提取 WHY 的主要手段。
