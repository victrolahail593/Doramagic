# WHY 提取改进 Brainstorm

## 背景

我们在做一个从开源项目代码中提取设计哲学（WHY）的系统——不是代码做了什么（WHAT），不是怎么做的（HOW），而是"为什么这样做而不那样做"。

这是一个产品的核心引擎，目标是让用户在学习开源项目时，能快速获取项目作者的设计智慧。

## 当前流程

1. **Stage 0**（确定性提取）：用 shell 脚本从 git repo 中提取 repo_facts（语言分布、目录结构、README、依赖、核心代码片段——按文件大小 Top 3 × 前 150 行）
2. **Stage 1**（LLM 提取）：将 repo_facts 喂给 LLM，用精心设计的 prompt 提取 5-8 条 WHY
3. **评估**：用 4 维度评分 rubric 打分（证据锚定 0-2、因果完整性 0-2、信息增量 0-2、工具价值 0-2，满分 8）

## 实验数据

我们用 3 个开源项目（vnpy 量化交易、daily_stock_analysis 股票分析、funNLP 中文NLP资源库）× 2 个模型（Sonnet、Gemini）× 2 个 prompt 版本（v1、v2）进行了完整的对照实验。

### v1 → v2 改进结果

| 项目 | Sonnet v1 | Sonnet v2 | Gemini v1 | Gemini v2 |
|---|---|---|---|---|
| vnpy | 7.0 | 7.3 | 6.5 | 6.3 |
| daily_stock | 7.1 | 7.6 | 5.8 | 6.4 |
| funNLP | 5.3 | 6.6 | 4.2 | 5.0 |
| **Overall** | **6.5** | **7.1** | **5.5** | **5.9** |

### v2 做了什么改进（3 个 P0 修复）

1. **证据硬性规格**：每条 WHY ≥2 个 evidence，≥1 个 CODE 类型带行号
2. **常识过滤**：如果一个 claim 直接从框架官方 README 就能得出，那是 WHAT 不是 WHY
3. **历史遗留过滤**：区分刻意设计决策 vs 过时语法的历史遗留

### 关键发现

1. **跨模型互补**：Sonnet 擅长深度（事件引擎、monkeypatch），Gemini 擅长广度（配置目录、缓存 TTL）——互补而非竞争
2. **funNLP 始终最低**：两个模型、两个版本都给 funNLP 最低分——因为 funNLP 几乎没有代码，Stage 0 提取的代码片段很少
3. **信息增量是区分度最大的维度**：高分 WHY 和低分 WHY 的主要差距在"信息增量"——是否提供了读代码才能发现的洞察
4. **Prompt 优化的边际效益递减**：v2 的 3 个修复有效（+0.4~0.6），但继续微调 prompt 的收益已经很小
5. **Stage 0 输入质量是天花板**：funNLP 低分不是因为 prompt 不好，而是因为 Stage 0 给 LLM 的代码太少

## 当前优先级评估

- **P0**: Stage 0 采样策略升级（最高杠杆——输入质量决定输出上限）
- **P1**: 多轮提取 + 合并去重（解决单次提取的随机性/覆盖不全问题）
- **P2**: 自动化评分（让 LLM 自评，自动门控低质量 WHY）
- **P3**: 继续微调 prompt（边际效益递减）

## 你的任务

基于上面的实验数据和分析，请提出你自己的 WHY 提取改进策略。特别关注：

1. **我们可能遗漏了什么改进方向？** 不要只复述上面的优先级，提出我们没想到的方向。
2. **Stage 0 采样策略具体怎么改？** 目前是"按文件大小 Top 3 × 前 150 行"，这显然不够。什么样的采样策略能最大化 WHY 的提取质量？
3. **多模型融合的最佳实践？** 我们已经证明 Sonnet 和 Gemini 互补，如何系统化利用这种互补性？
4. **如何让评分更客观？** 目前是人工打分，如何自动化且保证质量？
5. **有没有完全不同的方法论？** 比如，不通过"读代码"来提取 WHY，而是通过 git history、PR review、issue 讨论等来源？

请大胆提出想法，不需要考虑实现成本。我们先发散，再收敛。
