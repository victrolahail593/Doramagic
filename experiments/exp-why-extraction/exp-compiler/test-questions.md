# Knowledge Compiler A/B 测试题

## 测试说明

将 condition-A 或 condition-B 作为上下文注入，然后问以下 7 个问题。
每个问题测试不同的知识访问模式。评分标准：每题 0-2 分。

---

## Q1: 快速查找（测试结构化信息的可检索性）

> 我要给 vnpy 的 alpha 模块添加一个新的 ML 模型，损失函数应该用什么？为什么？

**满分答案要点**：MSE，因为多因子策略核心是截面排序而非涨跌预测，回归保留相对强弱信息。
- 2 分：MSE + 截面排序的理由
- 1 分：MSE 但理由不完整或错误
- 0 分：错误答案

## Q2: 设计哲学理解（测试叙事信息的传递效果）

> 如果我要给 vnpy 新增一个风控模块（RiskManager），正确的集成方式是什么？

**满分答案要点**：
1. 创建独立 pip 包
2. 通过 `main_engine.add_app(RiskManagerApp)` 运行时注册
3. 通过 EventEngine 订阅事件（如 EVENT_ORDER），不直接调用 Gateway
- 2 分：三点都提到
- 1 分：提到注册但遗漏事件驱动或独立包
- 0 分：建议直接改 MainEngine 源码或硬编码

## Q3: 反面知识（测试"不要做什么"的传递效果）

> 我想用 SQLAlchemy 给 vnpy 写一个统一的数据库抽象层，你怎么看？

**满分答案要点**：不推荐。tick 写入高频，ORM 对象映射是性能瓶颈；时序库(TDengine)和关系型库的查询模式差异大，ORM 无法统一。应该用 get_database() 工厂 + 每个适配器专项优化。
- 2 分：明确反对 + 两个具体理由（性能+查询模式差异）
- 1 分：反对但理由模糊
- 0 分：支持这个方案

## Q4: 跨知识点推理（测试多条知识的融合理解）

> vnpy 的可选依赖管理策略（Polars 在 alpha extra，DB 驱动独立包）背后的统一原则是什么？

**满分答案要点**：核心框架保持最小依赖，所有非核心功能通过外部机制组装（extra/独立包/工厂注入）。这与 App 注册、EventEngine 解耦是同一设计哲学："框架提供骨架和协议，具体实现由使用者组装"。
- 2 分：识别出统一原则 + 关联到整体设计哲学
- 1 分：只描述了依赖管理本身，未关联
- 0 分：答非所问

## Q5: 实战场景（测试操作性知识的传递）

> 我要部署一个 vnpy 策略到生产环境（无 UI），为什么不能用 systemd 来管理进程？正确方式是什么？

**满分答案要点**：systemd 不了解交易时段逻辑，需额外 cron；正确方式是父子双进程——父进程监控生命周期，子进程跑策略，非交易时段 sys.exit(0) 干净退出（保证 CTP ReqUserLogout），父进程等待后重新 fork。
- 2 分：systemd 缺陷 + 双进程方案 + CTP 断连细节
- 1 分：知道双进程但缺少细节
- 0 分：建议用 systemd

## Q6: 技术选型辨析（测试 contrast 信息的传递）

> 为什么 vnpy 的 RPC 不用 gRPC streaming 而用 ZeroMQ？

**满分答案要点**：gRPC Python 实现在高频场景下序列化开销大；双向 streaming 错误恢复复杂；ZeroMQ 是 C 库绑定，序列化通过 pickle 在应用层控制。且 ZeroMQ 的 REP+PUB 组合天然匹配两类通讯需求。
- 2 分：序列化开销 + 错误恢复 + ZeroMQ 优势
- 1 分：只提到其中一个原因
- 0 分：答非所问

## Q7: 全局理解（测试整体心智模型的建立）

> 用一句话概括 vnpy 框架的核心设计原则。

**满分答案要点**：总线解耦 + 按需组装（或等价表述：事件驱动解耦 + 运行时插拔 + 最小核心依赖）
- 2 分：准确概括核心原则
- 1 分：部分正确但不完整
- 0 分：泛泛而谈或错误
