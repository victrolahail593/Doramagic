基于 Superpowers 的 skill 描述，完整流程如下：

---

## Superpowers 规定的完整开发流程

### 1. 需求探索 — `superpowers:brainstorming`
**触发时机：** 任何创建功能之前，**必须**先用
- 探索用户意图、需求边界、设计方案
- 确定认证方式（JWT/Session/OAuth）、存储、安全要求等

### 2. 制定实现计划 — `superpowers:writing-plans`
**触发时机：** 有了需求/规格，在碰代码之前
- 产出分步实现计划，写入 plan 文件
- 识别关键文件、架构决策、依赖关系

### 3. （可选）隔离工作区 — `superpowers:using-git-worktrees`
**触发时机：** 需要与当前工作区隔离时
- 为认证功能创建独立的 git worktree

### 4. 执行计划 — `superpowers:executing-plans` 或 `superpowers:subagent-driven-development`
**触发时机：** 有了书面计划，开始执行
- `executing-plans`：带 review checkpoint 的单步执行
- `subagent-driven-development`：独立任务并行执行

### 5. 编写代码 — `superpowers:test-driven-development`
**触发时机：** 实现任何功能或修复 bug 之前
- 先写测试，再写实现代码（TDD）

### 6. 完成前验证 — `superpowers:verification-before-completion`
**触发时机：** 声称"完成了"或"测试通过"之前，**必须**先用
- 运行验证命令，用实际输出作为证据，不能凭假设声明成功

### 7. 请求代码审查 — `superpowers:requesting-code-review`
**触发时机：** 完成实现后、合并之前
- 验证工作是否满足需求

### 8. 接收并处理审查意见 — `superpowers:receiving-code-review`
**触发时机：** 收到 code review 反馈之后
- 技术性评估反馈，不盲目接受，验证后再实施

### 9. 合并收尾 — `superpowers:finishing-a-development-branch`
**触发时机：** 所有测试通过、实现完整
- 决定 merge / PR / cleanup 策略，完成分支集成

---

## 流程图

```
需求 → brainstorming → writing-plans → [git-worktrees]
     → executing-plans/subagent-driven-development
     → test-driven-development (循环)
     → verification-before-completion
     → requesting-code-review → receiving-code-review (循环)
     → finishing-a-development-branch → 合并
```

**核心原则：** 每一步都有对应 skill 约束，不能跳步——尤其是 brainstorming（动手前必做）和 verification（完成前必做）。
