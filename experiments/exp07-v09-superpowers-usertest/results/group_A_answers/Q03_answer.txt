两阶段分开是有意为之，核心原因是**关注点分离**——合并会让两种不同的判断互相干扰。

## 为什么不能合并？

**Spec compliance review 问的是：**
> "做了该做的事吗？没多做也没少做？"

**Code quality review 问的是：**
> "做法对吗？代码写得好吗？"

这两个问题的**评判标准完全不同**：

| | Spec Compliance | Code Quality |
|---|---|---|
| 参考基准 | 需求/计划文档 | 工程最佳实践 |
| 评判维度 | 有/无、符合/不符合 | 好/差、健壮/脆弱 |
| 典型问题 | "漏了 `--force` 参数" | "魔数 100 应该提取为常量" |
| 修复标准 | 对齐 spec | 提升代码质量 |

## 合并的实际问题

**问题1：顺序耦合**。Spec 不过关时根本不应该评 quality——代码都还不满足需求，讨论代码质量是浪费。Skill 明确禁止："Start code quality review before spec compliance is ✅"。

**问题2：认知负载混淆**。同一个 reviewer 同时检查"做没做"和"做得好不好"，极易发生遗漏。专注单一维度的 reviewer 更可靠。

**问题3：修复循环定位模糊**。发现问题后，implementer 不知道该对齐 spec 还是改善质量，两者优先级冲突时容易做出错误取舍。

## 一句话总结

先确认**做对了什么**（spec），再确认**做得怎么样**（quality）。这是两个串行门控，不是同一个检查的两个维度。合并会破坏"spec 通过才值得评 quality"这个前提假设。
