这是个很好的架构设计问题。核心原因是：**软件开发任务的执行路径天然是非线性的。**

## 固定 pipeline 的问题

如果强制线性执行，比如：

```
brainstorm → plan → TDD → implement → review → finish
```

会出现：
- **修一个 typo**，也要走完整个流程 → 摩擦过高
- **调试 bug**，不需要 brainstorm，但需要先 debug 再 TDD
- **接收 code review 反馈**，要从中间插入，破坏线性顺序

## 动态激活的本质

每个 skill 有明确的**触发条件**，本质上是一组 `if-then` 规则：

| 触发条件 | 激活的 skill |
|---|---|
| 遇到 bug / 测试失败 | `systematic-debugging` |
| 要实现新功能 | `brainstorming` → `writing-plans` |
| 任务有 2+ 独立子任务 | `dispatching-parallel-agents` |
| 即将说"完成了" | `verification-before-completion` |

## 设计意图

**1. 组合性**：不同 skill 可以组合，路径由任务形态决定，而非预定义。

**2. 认知门槛匹配**：简单任务激活少量 skill，复杂任务激活更多——成本与收益匹配。

**3. 防止遗漏，而非规定顺序**：`verification-before-completion` 的设计目标不是"第 N 步"，而是"无论何时，只要你准备宣布完成，就必须触发"。这是**状态驱动**，不是**步骤驱动**。

**4. 并发支持**：`dispatching-parallel-agents` 本身就是打破线性的——独立任务可以并行，固定 pipeline 无法表达这个语义。

## 类比

这更像**规则引擎**或**有限状态机**，而不是流水线。每个 skill 是一个带守卫条件（guard）的规则，系统在每个决策点扫描所有规则，激活匹配的那个。

```
当前状态 + 触发条件 → 激活对应 skill → 产生新状态
```

这样设计使得 skill 系统对任意任务类型都具有**适应性**，而不是为特定工作流硬编码顺序。
