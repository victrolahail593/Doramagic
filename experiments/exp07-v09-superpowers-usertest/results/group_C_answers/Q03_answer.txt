两阶段审查的根本原因是**认知干扰**——两类审查的判断标准会互相污染。

## 为什么不能合并成一次？

**Spec compliance review** 问的是："这段代码做了需求要求的事吗？"  
**Code quality review** 问的是："这段代码写得好不好？"

这两个问题用的是完全不同的评判框架：

- Spec compliance 是**外部视角**：对着需求文档逐条核对行为是否符合预期，不在乎实现方式。
- Code quality 是**内部视角**：看代码结构、可读性、SOLID 原则、测试覆盖——不管需求。

如果合并成一次，审查者（无论是人还是 AI subagent）会不自觉地用"代码写得好"来替代"需求是否满足"的判断。一个实现优雅但功能偏离需求的代码，很容易在混合审查中"通过"——因为代码质量的光环会掩盖 spec 违反。

## 结合 Superpowers 的上下文

知识包里强调 subagent 会在执行过程中悄悄偏离计划（2-5 分钟粒度的原因也是如此）。两阶段审查是这个问题的补救机制：

1. **先做 spec compliance**：强制先回到需求文档，不允许代码质量的判断介入，确认"做了正确的事"。
2. **再做 code quality**：在确认方向正确的前提下，才评价"做得正不正确"。

顺序不能倒——如果先审代码质量再对 spec，当代码写得很漂亮时，审查者会倾向于"说服自己"需求其实也满足了。

**一句话总结**：两阶段是为了防止"代码好看"污染"需求达成"的判断，强制分离两类完全不同的评判框架。
