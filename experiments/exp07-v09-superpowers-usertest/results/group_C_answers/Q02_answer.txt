动态激活的核心原因是：**不同任务的需求结构根本不同，强制线性顺序会造成大量无效开销和错误约束。**

具体来说：

**1. 任务类型决定所需 skill 的组合**

- "修一个 bug" → `systematic-debugging` 优先，不需要 `brainstorming`
- "实现新功能" → `brainstorming` → `writing-plans` → `test-driven-development`
- "收到 code review" → `receiving-code-review`，不需要 planning 流程

线性 pipeline 意味着修一个 typo 也要走完整个 14 步流程，这显然荒谬。

**2. Skill 之间存在条件依赖，而非时序依赖**

知识注入里明确了优先级规则：
- **Process skills 先于 implementation skills**（brainstorming 先于 frontend-design）
- 但这是"如果两者都适用时的顺序"，不是"所有任务都要经历两者"

**3. 强制触发 + 动态激活 = 纪律而不是官僚**

设计哲学里说得很清楚：skill 是"在每个关键节点说'慢着，先想想'"。关键在于**节点**——每个触发条件对应一个决策点，而不是时间轴上的固定位置。

如果是线性 pipeline：
- 代理会机械地走完流程，即使某些 skill 对当前任务毫无意义
- 反而削弱了"纪律感"——变成了走程序，不是真正的判断

动态激活让每次触发都是**真实需要**的判断，而不是形式主义的勾选。

**本质上**：superpowers 模拟的是一个有经验的 sensei 的行为——sensei 不会对每个问题都用同一套讲解，而是根据学生当前的错误模式精准介入。固定 pipeline 是教科书，动态激活是真正的师徒制。
