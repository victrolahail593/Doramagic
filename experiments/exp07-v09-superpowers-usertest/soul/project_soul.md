# superpowers — AI 知识注入

## 设计哲学
代理的创造力需要框架来引导，而不是被压制。superpowers 的核心信念是：AI 代理就像一个"热情过度、品味不佳、缺乏项目上下文、讨厌写测试的初级工程师"。他们有无限精力和能力，但需要结构化的工作流把能量转化为高质量代码。所有设计——强制 TDD、任务粒度 2-5 分钟、subagent 两阶段审查——都是为了让代理的创造力在正确的轨道上奔驰，而不是脱缰野马。

## 心智模型
把 superpowers 想象成代理的 sensei + 训练轮。它不阻止代理行动，而是在每个关键节点说"慢着，先想想""写测试了吗""让我看看代码"。理解了这一点，你就明白为什么每个 skill 都是强制触发而非建议——这是纪律，不是选项。

## 为什么这样设计

**1. 为什么 Skill 触发是强制的？**
因为 AI 代理太擅长"自己搞定"——当你说"帮我修这个 bug"，它立刻就开始调试，完全跳过了 systematic-debugging skill 的系统化分析方法。如果 skill 是可选的，代理会永远选择更快的路径。强制触发确保它不会在错误的方向上跑太远。

**2. 为什么 Subagent 必须 fresh？**
因为上下文污染。代理连续执行多个任务时，之前任务的代码会"污染"后面的思考——它会不自觉地复用之前的数据结构、变量名、测试策略。fresh subagent 保证每个任务从零开始，质量不受任务数量影响。

**3. 为什么任务粒度必须是 2-5 分钟？**
因为 subagent 没有人类在旁边实时指导。一个 30 分钟的任务，足够它偏离计划 3 次而不被发现。2-5 分钟的粒度意味着每个任务都在人类可接受的审查周期内完成，即使出错也能快速发现和纠正。

## 你一定会踩的坑

**1. 安装成功但 Skill 静默失效**
你按 README 安装了 superpowers，然后像平常一样让 Claude 写代码。它确实写了，也确实用了 Plan Mode——但它用的是 Claude Code 原生的 Plan Mode，不是 superpowers 的 brainstorming → writing-plans → TDD 流程。你完全不知道 skill 根本没被触发，因为它没有明显的视觉反馈。后来你看了 Issue #189（22 comments）才发现是 disable-model-invocation 配置问题。解决方法是正确配置 marketplace：先 `/plugin marketplace add obra/superpowers-marketplace`，再 `/plugin install superpowers@superpowers-marketplace`，然后重启 Claude Code。

**2. 安装路径错误导致 Skills 不可用**
你把 superpowers clone 到了自己觉得"合理"的目录，但 Claude Code 只在 ~/.claude/plugins/ 里找 skills。你反复检查 README，确认每一步都做了，但 skill 永远说找不到。Issue #58（23 comments）和 Issue #76（17 comments）里大量用户反馈这个问题——不同平台的安装路径完全不同，Claude Code、Cursor、Codex、OpenCode 各有各的约定。解决方法是严格按照对应平台的 INSTALL.md 操作，不要自己发挥。

**3. 无法确认 superpowers 是否被激活**
安装完成后，你在一次普通对话中问 Claude 一个问题，它确实做了计划，也确实写了代码——但它用的是标准行为，不是 superpowers 工作流。你没有办法判断它是否真的在用 brainstorming skill，还是只是表现得"有点像"而已。Issue #446（2 👍, 20 comments）专门讨论这个问题，至今没有完美的解决方案。暂时只能通过问它"你在用哪个 skill？"来确认。
