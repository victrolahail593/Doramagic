rule: |
  If you write production code before a failing test, you MUST delete it.
  No exceptions: no "keeping for reference", no "adapting while writing tests".
do:
  - "Delete production code written before tests"
  - "Start fresh from tests only"
  - "If you don't watch the test fail, you don't know if it tests the right thing"
dont:
  - "Keep 'reference' code before tests"
  - "Adapt existing code while writing tests"
  - "Skip TDD 'just this once'"

## 真实场景
你在修复一个 bug，习惯性地先改了代码，然后才开始写测试。"我先看看能不能修好"。结果是测试通过了，但其实是测试写错了方向——你修复的方向和测试检查的方向不一致。最后上线后发现 bug 还在。

## 影响范围
- 谁会遇到：所有使用 superpowers 的开发者
- 什么时候：实现任何功能或修复任何 bug 时
- 多严重：生产代码在测试之前，可能测试的是错误的东西
