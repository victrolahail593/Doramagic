rule: |
  After each subagent task, review in two stages:
  1. Spec compliance: Does the code match the spec?
  2. Code quality: Does the code meet quality standards?
  Skipping either stage risks bugs or technical debt.
do:
  - "Always do spec review before code quality review"
  - "Fix spec gaps before checking quality"
  - "Use spec-reviewer-prompt.md and code-quality-reviewer-prompt.md"
dont:
  - "Skip spec review and go straight to code review"
  - "Approve code that matches spec but has quality issues"
  - "Approve code that has great quality but doesn't match spec"

## 真实场景
你只做了代码质量审查，觉得代码写得很漂亮就通过了。结果上线后发现功能完全不对——代码实现的是一个"用户列表按字母排序"，而 spec 要求的是"用户列表按活跃度排序"。

## 影响范围
- 谁会遇到：使用 subagent-driven-development 的开发者
- 什么时候：每个任务完成后
- 多严重：先质量后 spec 或跳过任一阶段都会导致问题漏过
