rule: |
  NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST. Write the test first.
  Watch it fail. Write minimal code to pass. If you wrote code before the test,
  delete it. Start over. Implement fresh from tests.
do:
  - "Write failing test (RED phase)"
  - "Watch test fail before writing code"
  - "Write minimal code to pass (GREEN phase)"
  - "Refactor if needed while tests pass"
  - "Commit after each RED-GREEN-REFACTOR cycle"
dont:
  - "Write code before writing test"
  - "Keep 'reference' code for later"
  - "Adapt existing code while writing tests"
  - "Skip RED phase for 'quick' tasks"

## 真实场景
你在修复一个 bug，顺手写了修复代码，然后才开始写测试。
测试通过了，你觉得代码是对的。实际上测试没有覆盖真正的问题场景。
上线后 bug 依然存在，你排查半天发现测试写错了，但因为之前有"参考代码"，
你一直沿着错误思路走。

## 影响范围
- 谁会遇到：所有使用 superpowers 的开发者
- 什么时候：任何实现、bug修复、重构时
- 多严重：CRITICAL — 违反会导致测试无效，无法验证代码正确性
