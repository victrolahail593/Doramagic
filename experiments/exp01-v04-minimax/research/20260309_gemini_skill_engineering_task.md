# 研究任务：Soul Extractor 的产品策略与跨模型适配

## 项目背景

### 我们在做什么
AllInOne 是一个以"开源知识"为燃料的 AI 平台。核心假设：从开源项目中提取结构化知识，注入 AI 上下文，能显著提升 AI 对该项目的理解和回答质量。

我们已验证这个假设——实验表明知识注入后 AI 回答质量从 42% 提升到 96%（2.3 倍）。

现在的问题是：**如何将这个能力产品化，让普通用户通过一句话触发，自动完成提取，并得到可用的结果？**

### 当前实现
- 一个 OpenClaw Skill（名为 soul-extractor），由 SKILL.md（markdown 指令）+ prepare-repo.sh（克隆和打包脚本）组成
- 运行在 OpenClaw 平台上，用户通过 Telegram 发消息触发
- 使用平台默认模型（当前为 MiniMax-M2.5，免费模型）
- 目标用户为中国大陆开发者

### Skill 的工作流程（设计）
```
用户发送 "提取灵魂 https://github.com/xxx/yyy"
  → OpenClaw 匹配到 soul-extractor skill
    → Agent 运行 prepare-repo.sh（克隆代码、Repomix 打包）
      → Agent 读取打包后的代码
        → Agent 分步分析，产出知识卡片：
          - 项目本质（解决什么问题、核心承诺）
          - 概念卡片（核心概念的 Is/IsNot 定义）
          - 工作流卡片（关键流程 + Mermaid 图）
          - 规则卡片（使用陷阱、严重度、真实场景）
          - 项目灵魂总结（起步文档）
        → Agent 向用户汇报结果
```

## 我们遇到的全部问题

### 🔴 问题 1：Skill 触发完全失败

用户在 Telegram 中发送任何形式的触发消息，OpenClaw 都没有识别到 soul-extractor skill：

| 尝试的触发方式 | 结果 |
|--------------|------|
| "提取灵魂 https://github.com/..." | Agent 用内置知识回答了 python-dotenv 的介绍 |
| "extract soul https://github.com/..." | 同上 |
| "使用 soul-extractor 技能提取 ..." | Agent 回复"没找到 soul-extractor 这个 skill" |

**我们检查过的**:
- SKILL.md 格式正确（YAML frontmatter + markdown）
- 文件在正确目录（~/.openclaw/skills/soul-extractor/，和其他 skill 一样用 symlink）
- OpenClaw 已重启（新 session）
- 其他 skill（capability-evolver、find-skills）存在但我们也不确定它们是否能被正常触发

**最终绕过方式**: 直接在对话中给 agent 发送完整的步骤指令，不依赖 skill 系统。Agent 这才开始执行。

**这意味着我们的产品核心路径的第一步就断了。**

### 🔴 问题 2：产品形态和价值闭环未定义

这是 CEO 反复追问的根本性问题。

**CEO 的追问链**:
1. "这个输出的结果可以用来做什么了？" — 产出的 .md 文件放在那里，然后呢？
2. "这样一个文档的适合用户是什么？" — 谁会看这些知识卡片？
3. "我们的'抽取是否足够充分'，我们抽取的信息是否可以满足'让 AI 变强的燃料'的诉求" — 核心价值主张是"让 AI 变强"，但产出更像"给人看的文档"

**具体矛盾**:

| 维度 | "AI 燃料"方向 | "人类文档"方向 | 我们目前的状态 |
|------|-------------|-------------|-------------|
| 格式 | 结构化 YAML/JSON，机器可解析 | Markdown，人类可读 | 混合，不三不四 |
| 内容 | 条件化知识（IF/THEN）、精确引用 | 概述、最佳实践 | 有条件化知识但不够深 |
| 消费者 | AI system prompt / RAG | 开发者阅读 | 未定义 |
| 注入方式 | 自动注入 AI 上下文 | 人工查阅 | **不存在** |
| 验证方式 | A/B 测试 AI 回答质量 | 用户满意度 | 手工实验（exp01），未产品化 |

**缺失的关键环节——"注入协议"**:
提取再完美，如果没有定义"这些卡片怎么喂给 AI"，燃料就无处燃烧。可能的注入方式：
- 写入项目的 CLAUDE.md / .cursorrules（让 AI 编码助手自动读取）
- 塞进 system prompt（OpenClaw agent 的上下文）
- 做 RAG 检索（用户提问时动态检索相关卡片）
- 以上都没有实现

### 🟡 问题 3：弱模型指令遵从度不足

SKILL.md v0.4 共 6.7KB，约 20 条独立约束。不同模型执行效果差异巨大：

| 维度 | Claude Opus（exp01 手动） | MiniMax-M2.5（v0.4 测试） |
|------|-------------------------|--------------------------|
| 产出文件数 | 43 | 15 |
| 项目本质（第一性原理） | ✅ 完整回答 5 个根本问题 | ❌ 完全跳过 |
| 概念卡片 | 3 张，Is/IsNot + 边界 + 证据 | 3 张，格式基本正确 |
| 工作流卡片 | 3 张，Mermaid 流程图 | 2 张，ASCII 替代 |
| 代码规则卡 | 5 张，IF/THEN + 证据链 | 2 张，缺严重度和场景 |
| 社区规则卡 | 21 张，置信度 0.89 | 3 张，有 Issue 引用但较浅 |
| 自主完成 | ✅ 全程自动 | ❌ 中途停下问"要继续吗？" |
| project_soul.md | 第一性原理结构 | 技术架构描述 |

**CEO 的评价**: "规则卡片做的有点浅，这不利于体现 allinone 的专业性"

**学术研究数据**: 弱模型指令遵从呈指数衰减 P(all) = P(single)^N。弱模型可靠预算约 5-10 条指令，我们有 20+ 条。

**CEO 的核心担忧**: "用户分不清是 skill 强还是模型强。对于强模型的依赖不是一个好结果。"

### 🟡 问题 4：Agent 行为不可控

即使绕过 skill 系统直接下指令，Agent 行为也偏离了设计：
- **不调用脚本**: prepare-repo.sh 没被执行，Agent 自己 git clone + npx repomix
- **路径不受控**: 文件写到 Agent 自己的工作目录 `~/.openclaw/workspace/`，而非指定的 `~/soul-output/`
- **自行决定结构**: Agent 没严格按 SKILL.md 的目录结构输出
- **中途停顿**: Agent 完成 Eagle Eye 后停下来问"是否继续 Deep Dive？"，需要人工确认

### 🟡 问题 5：中国大陆网络环境

- GitHub 直接访问不稳定
- prepare-repo.sh 已内置 3 个镜像站回退，但如果脚本不被调用（问题 4），镜像逻辑不生效
- Repomix 依赖 npx，需要 Node.js 环境和网络下载

### 🟡 问题 6：输出内容的业务质量

CEO 的 5 条具体反馈：
1. **过程不可见**: 用户不知道进展，干等结果
2. **缺第一性原理**: 没有回答"这个项目到底解决什么问题"
3. **GitHub 镜像**: 中国用户需要
4. **规则太浅**: "不利于体现专业性，更不利于后续提供无误的服务"
5. **输出太技术**: "一点都没有用第一性原理来思考和给用户答案"

### 问题之间的因果关系

```
产品形态未定义（问题2）
  → 不知道产出应该面向人还是 AI
    → SKILL.md 的指令设计方向不明确
      → 指令既像在写文档又像在做数据提取
        → 弱模型更加困惑（问题3）

Skill 触发失败（问题1）
  → 绕过 skill 系统直接下指令
    → Agent 不认这是"预定义的工作流"
      → 自行决定执行方式（问题4）
        → 脚本不被调用、路径不受控

指令遵从度差（问题3）
  → 核心步骤被跳过
    → 产出质量不够（问题6）
      → CEO 质疑产品价值
        → 回到产品形态问题（问题2）

所有问题形成闭环，而非独立存在。
```

## 你需要研究的问题

### Q1: 我们是否在问错误的问题？

从第一性原理出发：
- "让弱模型执行复杂 skill"这个目标本身是否合理？
- 还是应该接受分工：弱模型 = 快速预览，强模型 = 完整提取？
- 如果分工是正确答案，产品体验应该怎么设计？（比如"MiniMax 给你 30 秒概览，想要完整提取请切换到 Claude"）
- 市场上有没有成功做到"同一 skill/plugin 跨模型能力自动适配"的产品？它们怎么做的？

### Q2: 产品形态应该是什么？

三种可能方向的深度分析：

**方向 A — AI 燃料**: 产出是结构化数据，自动注入 AI 上下文，用户无需直接阅读
- 注入协议怎么设计？
- 哪些 AI 工具支持自动注入（CLAUDE.md、.cursorrules、system prompt）？
- 如何验证注入后 AI 真的变强了？

**方向 B — 增强文档**: 产出是人类可读的项目知识库，比 README 更深入
- 目标用户是谁？（新手上手？架构师评估？）
- 和已有方案（README、Architecture Decision Records、awesome-xxx）相比有什么独特价值？
- 如何变现？

**方向 C — 双产出**: 同时产出 AI 可注入版本 + 人类可读版本
- 是否过度复杂？
- 有没有优雅的统一方案？

### Q3: 跨模型指令设计的策略

**渐进式指令（Progressive Enhancement）**:
- 核心指令所有模型执行，增强指令只有强模型执行
- 如何在 markdown 中标记必须 vs 可选？
- 弱模型能否理解"如果你有能力，额外做 X"？
- 强模型看到"可选"会不会跳过？

**多阶段拆分**:
- 拆成 3 个小 skill 还是 1 个 skill 内分阶段？
- 如何编排阶段之间的衔接？

**Few-shot vs 模板描述**:
- 对弱模型，完整示例 vs 抽象模板哪个更有效？
- Gemini Flash / Pro / Ultra 分别对哪种格式遵从度最高？
- Google 官方关于不同 Gemini 模型写指令的最佳实践

### Q4: Gemini 系列的指令遵从特性

- Gemini Flash / Pro / Ultra 在指令遵从上的定量差异
- Gemini 对哪种指令格式遵从度最高？对比：编号列表、XML tags、JSON Schema、Markdown headers、YAML
- Gemini API 的 structured output / function calling 能否被利用来强制输出格式？
- Google Vertex AI Agent Builder 中如何处理多能力模型的差异？

### Q5: 竞品和生态分析

- 有没有其他工具做"开源项目知识提取 + AI 注入"？
- Cursor 的 .cursorrules、Claude Code 的 CLAUDE.md、GitHub Copilot 的 repo context——它们的知识来源是什么？是否有类似"提取→注入"的流程？
- 如果这些工具已经在做类似的事，我们的差异化在哪里？
- OpenClaw 生态中有没有类似定位的 skill？
