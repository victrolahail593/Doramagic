# 研究任务：Soul Extractor Skill 的工程落地问题

## 项目背景

### 我们在做什么
AllInOne 是一个以"开源知识"为燃料的 AI 平台。核心理念：从开源项目中提取结构化知识，注入 AI，让 AI 对该项目的理解从"泛泛而谈"提升到"专家级"。

我们开发了一个名为 soul-extractor 的 OpenClaw Skill，用 markdown 指令文件（SKILL.md）指导 AI agent 从任意 GitHub 开源项目中提取结构化知识卡片。

### 技术栈
- **平台**: OpenClaw — 一个开源 AI agent 平台，支持 Telegram 交互、多模型切换、skill 扩展
- **Skill 格式**: SKILL.md（YAML frontmatter + markdown 步骤指令）+ 可选的 scripts/ 目录
- **测试模型**: MiniMax-M2.5（免费模型，作为弱模型基准）
- **对照模型**: Claude Opus 4.6（强模型，exp01 实验中使用）
- **目标用户**: 中国大陆开发者，通过 Telegram 与 OpenClaw 交互

### 已验证的结论（exp01 实验）
用 Claude Opus 手动执行提取流程（非 skill，直接对话），对 python-dotenv（1105 行 Python）做了 A/B/C 验证：
- 无灵魂知识：AI 回答质量 42%
- 注入代码知识：83%（+96%）
- 注入代码+社区知识：96%（+125%）

**结论：知识提取方向正确，效果显著。问题出在产品化过程中。**

## 我们遇到的全部问题

### 🔴 问题 1：Skill 触发失败（阻断性）

**现象**: 用户在 Telegram 中向 OpenClaw 发送"提取灵魂 https://github.com/theskumar/python-dotenv"，agent（MiniMax-M2.5）完全没有识别到 soul-extractor skill，直接用自己的内置知识回答了一段关于 python-dotenv 的介绍。

**我们尝试了什么**:
1. 将 skill 目录直接放在 `~/.openclaw/skills/soul-extractor/` → 不行
2. 改为 symlink 到 `~/.agents/skills/soul-extractor/`（和其他已安装 skill 的方式一致）→ 不行
3. 重启 OpenClaw（新 session）→ 不行
4. 换触发词："extract soul"、"使用 soul-extractor 技能提取"→ agent 回复"没找到 soul-extractor 这个 skill"
5. 最终绕过 skill 系统，直接在对话中给 agent 下达完整步骤指令 → 这才执行了

**核心疑问**:
- OpenClaw 的 skill 发现和匹配机制是什么？是关键词匹配、语义匹配、还是 agent 自主选择？
- 其他已安装的 skill（capability-evolver、find-skills 等）是怎么被触发的？
- 第三方自定义 skill 如何正确注册让 agent 能发现和使用？
- 是否需要在 openclaw.json 中显式注册 skill？

### 🔴 问题 2：产品形态未定义（根本性）

**现象**: 提取出来的知识卡片"放在那里了"，但没有定义"然后呢"。

**具体问题**:
- **产出定位不清**: exp01 用 Claude Opus 产出了 43 个文件（结构化 YAML 卡片），MiniMax 产出了 15 个文件（更接近人类可读文档）。前者像"AI 可注入的数据"，后者像"更好的 README"。我们要的是哪个？
- **注入协议不存在**: 即使提取完美，这些卡片怎么喂给 AI？塞进 system prompt？写入项目的 CLAUDE.md？做 RAG 检索？没有定义。
- **验证闭环缺失**: exp01 的 A/B/C 验证是手工实验，不是产品化流程。无法自动验证"提取的知识是否真的让 AI 变强了"。
- **用户价值不明确**: 用户触发提取后，拿到一堆 .md 文件——然后呢？用户怎么用这些文件？

**CEO 的原话**: "这个输出的结果可以用来做什么了？"、"我们的'抽取是否足够充分'，我们抽取的信息是否可以满足'让 AI 变强的燃料'的诉求"

### 🟡 问题 3：弱模型指令遵从度不足

**现象**: MiniMax-M2.5 执行 SKILL.md 时只遵从约 60% 的指令。

**对比数据**:

| 维度 | Claude Opus（exp01） | MiniMax（v0.4 测试） |
|------|---------------------|---------------------|
| 产出文件数 | 43 | 15 |
| 项目本质分析 | ✅ 完整 | ❌ 完全跳过 |
| 概念卡片 | 3 张，完整格式 | 3 张，格式基本正确 |
| 工作流卡片 | 3 张，含 Mermaid | 2 张，ASCII 流程图 |
| 规则卡片（代码） | 5 张，有 IF/THEN | 2 张，缺严重度和真实场景 |
| 规则卡片（社区） | 21 张，平均置信度 0.89 | 3 张，有 Issue 引用但较浅 |
| 自主完成 | ✅ 全程自主 | ❌ 中途停下问"要继续吗？" |

**弱模型跳过的具体内容**:
1. Step 2"项目本质"（5 个第一性原理问题）— 完全跳过
2. 规则卡片的严重度（HIGH/MEDIUM/LOW）— 不标注
3. 规则卡片的"真实场景"和"影响范围"— 不写
4. 数量目标（10 张规则卡）— 只产出 5 张
5. project_soul.md 的第一性原理结构（"解决什么问题""如果没有它怎么办"）— 替换为技术架构描述

**已知学术结论**: 弱模型同时遵从 N 条指令的概率为 P(single)^N，呈指数衰减。弱模型可靠指令预算约 5-10 条，我们的 SKILL.md 有 20+ 条。

### 🟡 问题 4：脚本未被调用，路径不受控

**现象**:
- prepare-repo.sh 没被 agent 调用。Agent 自己直接用 git clone + npx repomix 完成了准备工作。
- 文件写到了 agent 自己的工作目录 `~/.openclaw/workspace/soul-extractor-output/` 而非 SKILL.md 指定的 `~/soul-output/python-dotenv/`。

**原因推测**:
- agent 看到了脚本路径但选择自己做（可能 bash 执行权限受限，或 agent 认为自己能做就不调脚本）
- 输出路径是 SKILL.md 中定义的，但 agent 用了自己的默认工作目录

### 🟡 问题 5：中国大陆用户的网络问题

**GitHub 访问不稳定**: 中国大陆用户可能无法直接 git clone GitHub 仓库。我们在 prepare-repo.sh 中加了 3 个镜像站自动回退（ghproxy.net、mirror.ghproxy.com、gh-proxy.com），但：
- 镜像站稳定性未验证
- 如果 agent 自己 clone（绕过脚本），镜像逻辑不会生效
- 用户可能需要手动提供镜像 URL 或本地路径

### 🟡 问题 6：输出质量的业务反馈

CEO 审阅 MiniMax 产出的 PDF 后给出 5 条反馈：
1. **过程不可见**: 用户干等结果，不知道进展 → v0.4 已加进度汇报指令，MiniMax 部分遵从
2. **缺少第一性原理**: 没有回答"这个项目解决什么问题" → v0.4 加了 Step 2，MiniMax 跳过了
3. **GitHub 镜像**: 中国用户需要 → 脚本已加，但脚本不一定被调用
4. **规则卡片太浅**: 不利于体现专业性 → v0.4 加了严重度/场景/影响范围，MiniMax 没遵从
5. **输出太技术**: 没有用第一性原理给用户答案 → v0.4 改了 project_soul.md 模板，MiniMax 部分遵从

### 关键矛盾总结

```
用户说"提取灵魂"
  → Skill 没被触发（问题1）
    → 绕过 skill 系统直接下指令
      → 脚本没被调用（问题4）
        → Agent 自己做，路径不对
          → 指令只遵从 60%（问题3）
            → 产出质量不够（问题6）
              → 产出定位也不清楚（问题2）
                → 用户拿到文件不知道怎么用（问题2）
```

**每一层都是断裂的。不是一个"指令遵从"问题，是整条链路都不通。**
