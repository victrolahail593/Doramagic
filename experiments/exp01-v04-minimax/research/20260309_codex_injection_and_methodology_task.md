# Codex 研究任务：注入协议设计 + 验证深度强化 + 研究方法论评估

## 背景

Soul Extractor 是一个从 GitHub 开源项目自动提取结构化知识的工具，产出 .cursorrules / CLAUDE.md 作为"AI 燃料"。当前版本 v0.7 已实现 5 个阶段的提取管道：

- Stage 0: prepare-repo.sh（git clone + Repomix + 社区信号采集）
- Stage 1: 项目本质提取
- Stage 2: 概念卡 + 工作流卡
- Stage 3: 规则卡（代码规则 DR-001~ + 社区陷阱 DR-100~）
- Stage 3.5: 验证与审查（v0.7 新增：脚本硬校验 + 模型软审查）
- Stage 4: 组装产出（.cursorrules / CLAUDE.md / project_soul.md）

### v0.7 横向对比实验关键发现

我们用 v0.6 和 v0.7 对同一个项目（obra/superpowers）做了提取。结果：

1. **14 张卡片内容完全相同** — Stage 3.5 验证全部通过，模型软审查没有修改任何卡片
2. **v0.7 新增了 validation_report.json + validation_summary.md** — 量化基线到位
3. **验证发现了真实问题** — DR-001 引用了不存在的文件 `SK-20ILL.md`（warning 而非 error）
4. **v0.7 执行更快** — 因为复用了 v0.6 缓存的 artifacts（git clone + repomix + 社区信号）

### 来自 OpenClaw 实际使用的关键发现

**AI 生产了燃料，但自己烧不了。** 这是注入协议缺失的直接体现。

## 研究目标

### 问题 1：注入协议（Injection Protocol）— 最紧迫的产品缺陷

当前状态：Soul Extractor 产出 .cursorrules / CLAUDE.md，但：
- 需要用户手动复制到项目根目录
- 执行提取的 AI 自己无法使用产出
- 没有定义"什么时候注入、注入多少、怎么注入"

**1.1 注入场景分析**

请分析以下几种注入方式的工程可行性：

| 方式 | 描述 | 问题 |
|------|------|------|
| A. 手动复制 | 用户把 .cursorrules 复制到项目根目录 | 最简单，但摩擦大 |
| B. 自动复制 | assemble-output.sh 自动把文件复制到用户指定的项目目录 | 需要知道目标路径 |
| C. 即时注入 | 提取完成后，AI 直接读取产出并"记住" | 受限于上下文窗口 |
| D. RAG 式 | 把卡片存入向量数据库，按需检索 | 过度工程？ |
| E. System prompt | 把 .cursorrules 内容注入 system prompt | 平台依赖 |
| F. Skill 化 | 把产出包装成 OpenClaw skill，其他 agent 可以调用 | 最大复用 |

**1.2 自消费问题**

Soul Extractor 自己执行完提取后，能否在同一会话中"消费"产出？

- 当前 MiniMax 说"做不到"——但其实它可以用 Read 工具读取文件
- 是否应该在 Stage 4 之后增加一个 Stage 5："读取并内化产出"？
- 还是说这个问题应该由 OpenClaw 平台层解决（比如 agent 自动加载 workspace 下的 .cursorrules）？

**1.3 注入量控制**

参考 Superpowers 的原则："上下文是公共资源"。

- 当前 .cursorrules 大约多大？全量注入会占用多少 token？
- 是否需要分层注入：核心规则（必须）+ 概念卡（按需）+ 工作流卡（按需）？
- 如何确保注入不会影响 AI 执行其他任务的能力？

### 问题 2：验证深度强化

v0.7 的 Stage 3.5 验证发现了 DR-001 引用 `SK-20ILL.md`（不存在的文件），但只是 warning。

**2.1 Warning → Error 升级策略**

哪些 warnings 应该升级为 errors（hard fail）？
- 代码引用文件不存在 → error 还是 warning？
- 如果升级为 error，弱模型可能频繁被拦截，怎么平衡？

**2.2 软审查有效性问题**

v0.7 实验中，模型软审查"走过场"——判断全部 pass 但没修改任何卡片。

- 这是因为卡片质量确实好？还是弱模型不擅长自我批评？
- 如何设计更有效的软审查 prompt？
- 是否应该引入"对抗性审查"——故意挑刺？

**2.3 跨模型验证**

如果换一个更弱或更强的模型运行同一个 skill，验证结果会怎样？
- 推荐的跨模型验证实验设计是什么？

### 问题 3：研究方法论评估（元问题）

我们的研究方法是：围绕一个核心问题，同时请 Opus（战略）、Codex（工程）、Gemini（产品）三方做独立研究，然后综合。

**3.1 这种方法有效吗？**

- 优势：多视角、避免单一偏见、并行高效
- 风险：每方只看到部分上下文、可能遗漏跨领域交叉问题
- 大模型的上下文限制是否会导致"顾此失彼"？

**3.2 上下文污染问题**

每一轮研究，我们都把前一轮的结论喂给下一轮。这会导致：
- 确认偏差（后续研究可能只是在重复前序结论）？
- 还是知识积累（后续研究在前序基础上更深入）？
- 如何区分这两种情况？

**3.3 改进建议**

如果你认为当前方法有风险，请提出具体的改进方案。比如：
- 是否需要"红队"研究（故意否定前序结论）？
- 是否需要更严格的"新发现 vs 重复"标注？
- 每轮研究的上下文应该包含多少前序信息？
