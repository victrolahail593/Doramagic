# 灵魂提取器 (Soul Extractor) 产品策略与跨模型适配研究报告

## 1. 我们是否在问错误的问题？(Q1 发现)
**核心结论：是的，让弱模型单次执行包含 20+ 独立约束的复杂 Skill 是违背 LLM 基本规律的。**
*   **第一性原理分析**：正如学术界指出的 $P(all) = P(single)^N$，当指令数量 $N$ 增加时，弱模型完全遵从所有指令的概率呈指数级下降。我们试图用 Prompt Engineering 解决本该由工程架构（Pipeline）解决的问题。
*   **分工才是解法**：弱模型（如 MiniMax）适合做意图识别、简单路由或快速的“项目摘要”；强模型（如 Claude Opus / GPT-4o / Gemini 1.5 Pro）才具备执行复杂、多步骤、深度逻辑推理（如第一性原理提取）的能力。
*   **产品体验设计**：
    *   **阶段一（极速响应）**：用户触发后，弱模型在 10 秒内返回项目的“浅层画像”（基于 README 和少量核心代码），并询问“是否需要消耗算力进行深度灵魂提取？”
    *   **阶段二（异步深度执行）**：用户确认后，后台调用强模型执行耗时的深度结构化提取，完成后通过消息推送给用户。
*   **市场对标**：Cursor 和 GitHub Copilot 均采用分层模型策略。Cursor 的补全和快速对话使用极速模型，而复杂的 Composer 多文件编辑必定使用强模型。

## 2. 产品形态应该是什么？(Q2 发现)
**推荐方向：方向 C（双产出）的改良版 —— “以 AI 燃料为主，人类可读为辅”。**
*   **核心矛盾解决**：产品的核心承诺是“让 AI 变强的燃料”，因此产出的绝对核心必须是**机器可解析、高密度的结构化知识**（方向 A）。人类无法高效阅读成百上千条条件规则（IF/THEN），这些是给 AI 看的。
*   **注入协议设计**：
    1.  **本地开发侧**：产出标准化的 `.cursorrules` (Cursor) 或 `CLAUDE.md` (Claude Code) 文件，用户只需将其放入项目根目录，AI 编码助手即可自动读取。
    2.  **平台侧 (OpenClaw)**：将产出的 YAML/JSON 卡片作为长期记忆（Long-term Memory）或特定 Repo 的 System Prompt 扩展，当用户与 Agent 讨论该 Repo 时自动挂载。
*   **双产出方案**：
    *   给 AI：生成 `soul_database.json` 或标准规则集（严格的条件逻辑）。
    *   给人：生成唯一的一份 `project_soul.md`（包含第一性原理分析、架构图），用于向 CEO/用户证明“我们确实懂这个项目”，建立信任。

## 3. 跨模型指令设计的具体可执行方案 (Q3)
针对不同模型的智力水平，单一大而全的 `SKILL.md` 是不可行的。
*   **多阶段拆分（Pipeline化）**：
    *   将原本的一个大 Prompt 拆分为明确的工程流水线：
        1.  `脚本层`：强制执行 `prepare-repo.sh`（非 AI 决策）。
        2.  `提取层 1`（本质与概念）：输入完整代码，提取第一性原理。
        3.  `提取层 2`（规则挖掘）：基于第一阶段的产出，针对性挖掘 IF/THEN 规则。
*   **Few-shot 与 模板描述**：
    *   **对弱模型**：必须使用**完整的 Few-shot 示例**。抽象模板（如“请输出问题的严重度”）对弱模型无效，必须给出真实的输入输出对。
    *   **对强模型**：使用严谨的 XML 标签或 JSON Schema 约束，强模型能够很好地理解抽象的结构化要求。

## 4. Gemini 特有的技术发现 (Q4)
*   **长上下文优势**：Gemini 1.5 Pro 支持高达 2M 的上下文窗口，这意味着我们可以直接将 `repomix` 的产物（哪怕是百万 token 级别的大型项目）一次性喂给它，这是其最大优势。
*   **指令遵从特性**：
    *   Gemini 对 **XML 标签**（如 `<instruction>`, `<example>`）的结构化 Prompt 敏感度极高，优于单纯的 Markdown 标题。
    *   **Structured Output (JSON Schema)**：这是彻底解决“输出格式不符合预期”的终极武器。直接调用 Gemini API 的 Structured Output 功能，可以 100% 保证输出格式包含（且仅包含）所需字段（如严重度、影响范围），彻底消除模型偷懒漏写字段的问题。

## 5. 竞品分析矩阵 (Q5)
| 竞品 | 知识来源 | 注入方式 | 知识密度 / 结构化程度 | 差异化分析 |
| --- | --- | --- | --- | --- |
| **Cursor** | `.cursorrules`, 本地 RAG | 自动挂载至 System Prompt | 低（多为人类手写散文规则） | Cursor 依赖用户自己写规则。我们的优势是**自动生成**这些规则。 |
| **Claude Code** | `CLAUDE.md`, 内部分析 | 自动注入工作区上下文 | 中（偏向于项目常规约定） | 偏向于通用开发辅助。 |
| **Copilot** | 本地代码库 AST, RAG | 隐式注入 (基于当前文件) | 低（代码片段级别） | Copilot 缺乏对项目“灵魂”（第一性原理、隐性边界）的全局理解。 |
| **AllInOne (我们)** | **全量代码深层提炼** | 目标生成标准协议文件 | **高（条件化 IF/THEN，第一性原理）** | 我们的护城河在于**“高质量的提炼能力”**，把原始代码转化为高浓度的 AI 燃料。 |

## 6. 推荐的端到端方案
**链路设计：**
1.  **用户输入**：Telegram 发送 `/extract https://github.com/xxx/yyy`（不再依赖脆弱的自然语言模糊匹配 Skill 系统，改用明确的斜杠命令触发确定的工作流）。
2.  **工程调度（非 AI）**：系统拦截命令，直接在后台沙盒中运行 `prepare-repo.sh`（解决镜像问题、路径不受控问题）。
3.  **多 Agent 协作**：
    *   将打包好的代码喂给 **Agent A（强模型，如 Gemini 1.5 Pro / Claude 3.5 Sonnet）** 负责提取第一性原理和核心概念。
    *   将代码和 Agent A 的产出喂给 **Agent B（专精规则提取）**，强制使用 JSON Schema 确保产出 IF/THEN 规则。
4.  **组装与推送**：
    *   生成供用户阅读的 `project_soul.md`。
    *   生成供 AI 消费的 `.cursorrules` / 核心 JSON。
    *   Telegram 机器人回复总结报告，并提供文件下载。

## 7. 优先级排序
*   **短期（1-2天）：跑通工程链路与证明价值**
    *   放弃让弱模型一次性跑通全链路。
    *   将触发方式从“自然语言命中 Skill”改为硬编码的系统命令（如 `/extract`），确保脚本能被 100% 稳定调用。
    *   定义好输出给 AI 的格式（例如直接生成一个可用的 `.cursorrules`），形成“提取 -> 放进 Cursor -> AI 变聪明”的最小可用价值闭环。
*   **中期（1-2周）：架构升级**
    *   重构 Skill 系统为 Pipeline 模式（多阶段拆分）。
    *   引入 JSON Schema (Structured Output) 解决格式遵从度问题。
    *   实现双产出逻辑。

## 8. 风险和局限说明
*   **算力成本风险**：深度提取大量代码并生成高质量的规则卡片，必然需要消耗强模型的 Token。成本与质量是正相关的，试图用免费弱模型实现强模型的效果是不现实的。
*   **平台限制风险**：如果 OpenClaw 平台目前不支持多阶段 Agent 编排或强制 API 参数（如 JSON Schema），我们可能需要脱离其原生的单一 Skill 框架，自己编写 orchestrator 脚本。
