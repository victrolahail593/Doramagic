# Soul Extractor v0.6 方案

## 三方输入综合

### Opus — Superpowers 框架启发
1. **反合理化设计**：在 STAGE 文件中列出模型可能跳过步骤的借口，逐一封杀
2. **显式技能链**：每个 STAGE 末尾明确声明下一步
3. **上下文是公共资源**：严控每个文件 token 量
4. **Iron Law / Hard Gate**：关键约束用零例外语言
5. **Description 是触发器不是摘要**：防止模型读摘要就跳过完整指令

### Codex — 工程最佳实践
1. `sort=reactions` 不是 GitHub 官方支持的排序项，必须改为 `sort=comments` + 本地打分
2. 增加 Security Advisories 采集
3. 增加 git log fix: commits 信号（本地、无网络依赖、高 ROI）
4. 过滤 PR（GitHub issues API 会混入 pull requests）
5. community_signals.md 控制在 12-20KB，硬上限 32KB
6. 拆出独立 `collect-community-signals.py`（shell 到达复杂度极限）
7. 认证/缓存/降级设计：有 token 增强，无 token 最小采集，失败不阻塞

### Gemini — 产品价值
1. CHANGELOG breaking changes 是 P0（确定性最高）
2. 社区信号是核心护城河，重度投入
3. 代码证据 + 社区证据应合并为一张卡（代码提供原理，社区提供威力评估）
4. project_soul.md 增加"避坑指南 Top 5"
5. severity 基于 reactions 数分级：50+ CRITICAL, 20+ HIGH, 其余 MEDIUM
