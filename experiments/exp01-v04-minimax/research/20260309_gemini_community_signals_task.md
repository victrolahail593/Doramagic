# Gemini 研究任务：社区信号的产品价值与信息设计

## 背景

我们在构建 Soul Extractor——一个从 GitHub 开源项目中提取结构化知识的 AI 技能。它的核心产出是 .cursorrules / CLAUDE.md 文件，作为 "AI 燃料" 注入给 Cursor 或 Claude Code，让 AI 编程助手瞬间获得某个开源库的专家级认知。

### 产品定位回顾

上一轮研究中，Gemini 你提出了一个关键洞察：**Soul Extractor 的产出是面向 AI 的产品，不是面向人的文档。** 这一点我们高度认同。

但现在我们遇到了一个新问题：产出的知识**太偏向代码静态分析，缺少真实世界的经验智慧。**

### 问题详述

在最新的 v0.5 测试中（对 python-dotenv 项目），Stage 3 产出了 10 张规则卡（DR-001~010），质量不错，但有一个致命缺陷：

**所有 10 张卡片都是从源码分析推导出来的。** 没有一张来自社区真实经验。

举例说明差距：

| 来源 | 示例 | AI 燃料价值 |
|------|------|-------------|
| 代码分析 | "parser.py 的正则把 # 当注释" | ⭐⭐⭐ 有用，但聪明的开发者读源码也能发现 |
| 社区经验 | "Issue #89: 多行字符串在实际项目中不工作，29 人踩过这个坑" | ⭐⭐⭐⭐⭐ 极有价值，只有踩过坑的人才知道 |
| 社区经验 | "从 v0.19 升级到 v1.0，set_key() 默认不跟随符号链接了，3 个生产事故" | ⭐⭐⭐⭐⭐ 升级必知，代码分析看不出历史变迁 |

**代码分析告诉你 "代码是什么"，社区经验告诉你 "别人怎么被坑的"。** 后者对 AI 燃料的价值更高，因为它能直接防止用户重蹈覆辙。

### 我们的初步方案

在 prepare-repo.sh（数据准备脚本）中新增社区信号采集，生成 `community_signals.md` 文件，包含：
1. CHANGELOG 中的 breaking changes
2. README 中的 FAQ / Troubleshooting 章节
3. GitHub Issues 按 reactions 排序的 Top 30

然后在 STAGE-3-rules.md 中要求模型基于这些社区数据产出 DR-100~ 系列的"社区陷阱"卡。

## 研究目标

请你从**产品价值和信息设计**角度，深入研究以下问题，并撰写一份详细报告。

### 问题 1：社区信号对 "AI 燃料" 质量的真实影响

这是最核心的产品问题：**加入社区信号后，.cursorrules 的实际效果会提升多少？**

请思考：
- AI 编程助手（Cursor / Claude Code）读到一条基于社区经验的规则 vs 一条基于代码分析的规则，行为会有什么不同？
- 从 AI 消费者（模型）的视角，什么样的规则描述方式最能影响它的代码生成行为？比如：
  - "不要用未加引号的 # 值" — 抽象规则
  - "Issue #89 报告了 29 人遇到多行字符串不工作，原因是解析器不支持原始换行，必须用 \n 转义" — 带数据的具体规则
  - 哪种更有效？为什么？
- 是否存在"社区噪声"风险？即采集到的 Issues 中大量是低质量、已过期、或不相关的信息，反而降低了 .cursorrules 的质量？

### 问题 3：community_signals.md 的信息设计

这个文件是连接 "采集" 和 "模型消费" 的关键桥梁。请从信息架构角度分析：

**3.1 信息密度 vs 上下文消耗的权衡**

community_signals.md 会被弱模型（MiniMax-M2.5，上下文约 128K）读取。同时模型还要读源码和项目本质。请估算：
- 社区信号文件应控制在多少 tokens 以内？
- 如何在有限 tokens 内最大化信息价值？
- 是否应该对信号做预处理摘要（在脚本层用简单规则过滤），还是把原始数据全给模型让它自己判断？

**3.2 面向弱模型的信息呈现格式**

考虑到消费者是 MiniMax-M2.5 这样的弱模型：
- 应该用什么格式呈现信号？markdown 列表？结构化 YAML？自然语言段落？
- 每条信号应该包含哪些字段？（issue 编号、标题、正文摘要、reactions 数、标签、状态...）
- 是否需要在文件中显式给出指令？比如 "以下 Issues 按社区关注度排序，请从中提取常见陷阱"

**3.3 信号之间的关联和去重**

- 多个 Issues 可能报告的是同一个根因问题（比如 5 个 issue 都在说 # 注释截断），怎么处理？
- CHANGELOG 和 Issues 可能有交叉（CHANGELOG 修复了某个 Issue 提到的 bug），怎么关联？
- 如果不做去重，是否会导致模型过度关注某个问题（因为出现次数多）？这是好事还是坏事？

### 问题 4：DR-100~ 社区陷阱卡的产品形态

社区陷阱卡和代码规则卡（DR-001~）在形态上应该有什么区别？

- **sources 字段**：代码规则引用 file:line，社区陷阱应该引用什么？Issue URL？CHANGELOG 条目？
- **severity 评估**：代码规则的严重度可以从代码逻辑推导，社区陷阱的严重度应该基于什么？受影响人数？issue 的 reactions 数？
- **时效性**：社区陷阱可能已在新版本中修复，是否需要标注 "已修复版本" 和 "影响版本范围"？
- **与代码规则的融合**：如果一个问题既能从代码分析出来（DR-001），又在 Issues 中被大量报告，应该生成一张卡还是两张卡？怎么关联？

### 问题 5：用户感知价值

站在 Soul Extractor 最终用户（开发者）的角度：

- 当用户看到 "DR-101: GitHub Issue #89 报告了 29 人遇到的多行字符串问题"，vs 看到 "DR-003: load_dotenv() 默认不覆盖环境变量"——哪个让用户觉得 Soul Extractor 更有价值？为什么？
- 社区陷阱卡是否应该成为 Soul Extractor 的**差异化卖点**？如果是，在产品描述/营销中应该如何强调？
- 是否应该在 project_soul.md（给人看的总结）中单独列一个 "社区高频陷阱 Top 5" 章节？
