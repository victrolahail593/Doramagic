## Overview
Every code change goes through two sequential reviews: spec compliance first, then code quality.

## Steps

| Stage | Reviewer | Focus |
|-------|----------|-------|
| 1 | Spec Reviewer | Does the code implement what the spec describes? |
| 2 | Code Quality Reviewer | Is the code well-written? (style, patterns, etc.) |

## Flow Diagram

```
Implementer submits code
          ↓
┌─────────────────────────┐
│  SPEC REVIEW            │
│  "Does code match spec?"│
└─────────────────────────┘
          ↓
    ┌─────┴─────┐
    ↓           ↓
  NO            YES
    ↓           ↓
Fix spec    ┌─────────────────────────┐
gaps        │  CODE QUALITY REVIEW     │
            │  "Is code well-written?" │
            └─────────────────────────┘
                      ↓
                ┌─────┴─────┐
                ↓           ↓
              NO            YES
                ↓           ↓
        Fix quality   Task complete
          issues
```

## Failure Modes

| Failure | Symptom | Recovery |
|---------|---------|----------|
| Skip spec review | Code implements wrong thing | Re-run spec review |
| Review in wrong order | Quality issues masked by spec issues | Always spec first |
| Accept "good enough" | Technical debt accumulates | Enforce quality gates |
