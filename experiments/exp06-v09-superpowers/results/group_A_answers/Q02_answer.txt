因为软件开发本身不是线性的。

固定 pipeline 的问题在于：它假设每个任务都走同一条路。但实际工作中：

- 有时你直接修 bug，不需要 brainstorming
- 有时你在 review 别人代码，不需要 TDD
- 有时任务很小，不需要 git worktree 隔离
- 有时你在回答问题，不需要任何实现 skill

**动态激活的核心优势：**

1. **最小必要原则** - 只激活当前情境真正需要的 skill，避免每次都走完整流程的开销

2. **可组合性** - skill 之间没有硬依赖，可以按需组合。`brainstorming` + `subagent-driven-development` 是一种组合，`systematic-debugging` + `verification-before-completion` 是另一种

3. **触发条件即语义** - 每个 skill 的触发条件本身就是它的"使用说明"，描述了它解决什么类型的问题，而不是它在第几步执行

4. **适应不同会话入口** - 用户可能从任意一个点进入：直接给你代码让你 review，或者直接给你 bug 让你修，或者描述一个新功能。没有固定起点

**实际上有隐式顺序：**

`using-superpowers` 中的流程图已经定义了优先级规则——process skills（brainstorming、debugging）先于 implementation skills。这是一种"软顺序"：不强制执行，但在冲突时有明确的优先级。

本质上这是一个**事件驱动**而非**过程驱动**的设计：skill 是响应情境的处理器，不是流水线上的工位。
