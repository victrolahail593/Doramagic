title: "Two-Stage Review -- Spec Compliance Before Code Quality"
severity: HIGH
rule: |
  After each subagent implements a task, conduct two-stage review:
  1. Spec reviewer first -- does code match the design spec?
  2. Code quality reviewer second -- are there patterns, security, performance issues?
  The order matters: fix spec gaps before quality issues.
do:
  - "Dispatch spec reviewer subagent first"
  - "Wait for spec compliance confirmation"
  - "Fix any spec gaps"
  - "Then dispatch code quality reviewer"
  - "Fix any quality issues after spec is correct"
dont:
  - "Review code quality before spec compliance"
  - "Skip either review stage to save time"
  - "Combine both reviews into one step"
confidence: 0.95
sources:
  - "skills/subagent-driven-development/SKILL.md:L104-130"

## 真实场景
你让代码质量 reviewer 直接审查。它发现了一个命名问题要求修复。
你修复后继续。然后发现代码根本没实现需求文档里的核心功能。
你花了时间修复命名问题，但最重要的功能缺失反而被忽略了。

## 影响范围
- 谁会遇到：使用 subagent-driven-development 的用户
- 什么时候：每个任务完成后
- 多严重：HIGH — 错误的审查顺序会导致核心功能缺失
