# Superpowers — AI 知识注入

## 设计哲学
**"AI 代理应该像训练有素的工程师一样思考，而不是像没有经验的码农。"**

这是 Superpowers 背后最核心的信念。所有设计决策都围绕一个原则：让代理在写代码之前先理解问题、设计方案、制定计划、写测试。质量优先于速度——自动化的工作流确保每一步都扎实。如果代理不经过设计阶段就写代码，就像没有蓝图的施工；不写测试就交付，就像没有质检的产品。Superpowers 不是要取代人类的判断，而是确保代理在需要人类确认的每个节点都停下来请求批准。

## 心智模型
把 Superpowers 想象成给 AI 代理配备的一位"虚拟技术导师"——它不会替代理写代码，但会在每个关键决策点追问："你确定要这样做吗？""测试在哪？""代码审查了吗？"。这位导师不取代代理的思考能力，而是确保代理的思考路径符合软件工程的常识。

## 为什么这样设计

**1. 为什么要用 Skills 而非固定工作流？**
因为软件开发不是线性的。每个项目的需求、复杂度、技术栈都不一样。Skills 是可组合的模块——brainstorming 触发在写代码前，TDD 触发在实现时，code review 触发在提交前。代理根据上下文自动选择合适的 skill，而不是被强制套进一个模板。这就是为什么 Superpowers 能适配各种项目和各种代理（Claude Code、Cursor、Codex、OpenCode）。

**2. 为什么要强制 TDD 的 RED-GREEN-REFACTOR 循环？**
因为 AI 代理太容易"作弊"——它们会写出看似正确但边界情况一塌糊涂的代码。强制先写测试再看失败，等于让代理先承认"我不知道正确答案是什么"。这打破了 AI 的幻觉，提高了代码质量。TDD 不是建议，是铁律。

**3. 为什么要双阶段审查（spec + quality）？**
因为审查一次不够。Spec 审查确保代码做了"对的事"——实现是否符合设计意图。Quality 审查确保代码是"对地做事"——代码风格、模式、是否有 bug。跳过任何一个都会导致问题：只有 spec 审查会漏掉代码质量问题，只有 quality 审查会漏掉需求理解偏差。

## 你一定会踩的坑

**1. Skills 找不到？23 个评论告诉你怎么解决**
你兴冲冲安装了 superpowers，输入 `/superpowers:brainstorm`，结果报错 "skill not found"。你反复检查安装步骤，以为自己操作有误。结果发现是 marketplace 没配置——必须先 `/plugin marketplace add obra/superpowers-marketplace`，然后才能 `/plugin install superpowers@superpowers-marketplace`。更坑的是，用 Skill tool 调用的同时不要用 Read tool 直接读文件，那是两个不同的加载机制。
（GitHub Issue #58, 23 comments — 最高赞的安装问题）

**2. disable-model-invocation 错误？Claude Code 的限制**
你用 Skill tool 调用 brainstorming skill，但报错说 skill 配置了 disable-model-invocation。你以为是 skill 本身的问题，疯狂排查自己用法对不对。结果发现这是 Claude Code 的特性，不是你的错。部分 skill 因为这个限制无法直接调用，需要等待维护者修复或使用其他调用方式。
（GitHub Issue #189, 22 comments, 2 👍）

**3. 不知道 Superpowers 是不是真的在工作？**
你按照文档安装了 superpowers，试着让它帮你规划一个功能。你感觉它确实在运行，但和标准 plan mode 相比也说不上有什么区别。没有明显的视觉反馈告诉你" skill 已触发"。你无法验证系统是否真的在工作，心里没底。
（GitHub Issue #446, 20 comments, 2 👍 — 仍在 open 状态）
