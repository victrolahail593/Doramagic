## Identity
TDD is a development discipline requiring a failing test before any production code is written, followed by minimal code to pass the test, then refactoring—all in a strict cycle.

## Is / Is Not

| IS | IS NOT |
|----|--------|
| A quality assurance method | A testing strategy after implementation |
| A design tool (tests define behavior) | Documentation or comments |
| A strict cycle: RED → GREEN → REFACTOR | Write tests whenever you want |
| A constraint on production code | Optional guidelines |

## Key Attributes

| Attribute | Type | Purpose |
|-----------|------|---------|
| RED | phase | Write failing test first |
| GREEN | phase | Write minimal code to pass |
| REFACTOR | phase | Clean up while keeping tests green |
| NO_PRODUCTION_CODE_WITHOUT_FAILING_TEST | rule | Absolute constraint |

## Boundaries
- Starts at: When implementing any feature, bugfix, or refactor
- Enforces: No production code before failing test
- Does NOT handle: Prototypes, generated code, config files (with human approval)
