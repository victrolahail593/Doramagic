# Soul Extractor 三方研究综合决策文档

**日期**: 2026-03-10
**来源报告**: Sonnet（深度研究版）、Gemini（精简整理版）、Codex（结构化证据版）

---

## 1. 三方共识（最可靠结论）

以下结论三份报告完全一致，可直接作为行动依据。

**共识1：多阶段 Pipeline 方向正确，不应回退**
Anthropic（Prompt Chaining）、Google ADK（Sequential Pipeline）、OpenAI（Workflow-level grading）均明确支持分阶段流水线用于确定性知识提取任务。Soul Extractor 的 Stage 0→1→2→3→3.5→4→5 结构与业界最佳实践一致，不应合并为一次性大 prompt。

**共识2：Stage 3.5 必须是真阻断门，而非软审查**
三方均指出当前验证环节失效（证据：v0.6 与 v0.7 注入文件逐字相同）。Anthropic 的 Evaluator-Optimizer 模式要求评估结果能驱动生成器修改；Codex 报告补充须对"规则保留率"和"证据覆盖"做硬校验，而非只审文风。结论：验证失败必须终止流程并强制回写，否则阶段形同虚设。

**共识3：规则卡是唯一事实源，叙事是派生层**
三方均识别到 v0.8 的核心问题：expert_narrative.md 的叙事内容挤占了 CRITICAL 规则，导致可执行性回退。共同结论是：概念卡/工作流卡/规则卡是"源知识层"，叙事只能是派生视图，最终 CLAUDE.md 必须规则优先。

**共识4：纯 AI 自评不可信，必须引入客观指标**
三方均引用了 LLM-as-a-Judge 偏差研究（self-preference bias、verbosity bias），均推荐混合评估（脚本硬校验 + LLM 辅助 + 人工校准）。当前 7 维全靠模型自评是高风险设计。

**共识5：A/B 测试应比较真实下游任务成功率，而非文风**
三方均认为 A/B 应对比"有/无注入文件时真实任务完成情况"，包括规则遵守率、陷阱规避率、测试通过率，而非由模型评判哪份文字更好看。

## 2. 分歧点

**分歧1：多模型分工的优先级**

- Sonnet：列为 P1，属近期改进（模型路由有成本和测试代价）
- Gemini：未明确列优先级，态度中性
- Codex：列为 P0，认为"多角色/多模型拆分"与验证机制修复同级

**我的判断**：列为 P1。理由是多模型路由本身有工程复杂度，且当前最紧迫的问题（验证失效、规则丢失）是流程逻辑问题，与使用几个模型无关。在 P0 问题解决前引入多模型反而会增加调试难度。但方向正确，应在 P0 修复完成后立即推进。

**分歧2：评估三层体系的定位**

- Sonnet：把 A/B 测试放在 P2（长期），强调工程投入高
- Codex：把混合评估体系（含人工 pairwise）列为 P0，认为没有好的评估就无法判断任何改进有效
- Gemini：把评估改进列为 P1

**我的判断**：分层处理。脚本化硬指标（规则数量统计、代码示例比例）属于 P0，成本极低应立即实施；LLM judge 辅助属于 P1；人工 pairwise 校准和完整 A/B benchmark 属于 P2。Codex 把全部评估都定为 P0 略显激进，忽视了工程成本差异。

**分歧3：稳定性评估（pass@k）**

- Codex 独有：建议同一 repo 多次运行统计 pass@3、方差
- Sonnet/Gemini：均未提及

**我的判断**：有价值但属于 P2。当前 pipeline 每次运行成本不低，在核心质量问题未解决前做稳定性评估收益有限。

## 3. 独特发现

**Sonnet 独有**：知识溯源率 + 知识卡片标准化框架
提出将卡片对齐"Facts + Philosophy + Patterns"三要素框架，并建议在 Stage 3 卡片生成时区分"代码信号"与"社区信号"来源，在 CLAUDE.md 中标注可信度。评估价值：高。这是对知识质量管控的精细化设计，v0.9 应吸收。

**Gemini 独有**：精准定位问题三角
提出 Soul Extractor 核心问题是"验证无硬约束、叙事无规则约束、评估过度依赖自评"三点，表述最简洁。评估价值：中等（诊断价值高，但没有超出其他报告的实质内容）。

**Codex 独有**：过程指标体系 + 早停优化
除结果指标外，提出"过程指标"（阶段通过率、回退率、成本、时延、pass@k 稳定性），以及"高分样本早停"（硬指标达阈值时跳过软审查）。评估价值：高。过程指标是工程可观测性的关键，应在 v0.9 或之后纳入。早停优化是 P2 的合理设计。

## 4. 最终决策建议

### 流程改进（针对问题1）

**[P0] 修复 Stage 3.5 为真阻断门**
为规则卡定义机器可检查的必要条件（包含条件语句 + 可运行代码示例），验证失败时终止流程并回写结构化修改意见 JSON 给 Stage 3 重新生成。
为什么是 P0：当前验证形同虚设，整个 pipeline 的质量保障完全失效，是最高优先问题。

**[P0] 强制 Stage 5 组装模板：规则路与叙事路分离**
Stage 5 组装时规则路（来自 Stage 3 卡片）和叙事路（来自 Stage 4）独立合并，CRITICAL RULES 区块强制排在 CLAUDE.md 前 30%，并设最小规则数量阈值（≥5条），低于阈值拒绝输出。
为什么是 P0：v0.8 叙事侵蚀规则是已确认的质量回退，修复成本极低（模板约束），不修复则所有后续改进都无法验证。

**[P1] 多模型分工**
Stage 1（灵魂发现）和 Stage 4（叙事合成）使用强推理模型；Stage 2/3（卡片提取）使用中端模型；Stage 3.5（验证）完全脚本化，不依赖 LLM；Stage 5（组装）脚本化模板填充。
为什么是 P1：方向明确、收益实在，但有工程复杂度，应在 P0 修复完成后推进，避免调试干扰。

### 评估体系改进（针对问题2）

**[P0] 引入脚本化硬指标**
立即新增四个脚本可统计的客观指标：CRITICAL 规则数量、规则含可运行代码示例比例、CLAUDE.md 各区块字数分布、规则总条数。这四个指标实现成本极低，可在当天完成，且直接检测 v0.8 回退问题。
为什么是 P0：成本极低，但能立即替代当前完全依赖 AI 自评的盲区，且是后续所有质量判断的基础。

**[P1] 混合评估：LLM judge 辅助 + Faithfulness 检查**
保留 AI 自评作为辅助信号，新增 RAGAS/DeepEval 的 Faithfulness 检查（注入内容与原始代码库一致性）。LLM judge 需使用不同于生成器的模型，并限定打分维度为"规则可执行性"而非"文字流畅度"。
为什么是 P1：工程量中等，需调试，但能系统性替代 verbosity bias 导致的错误判断。

**[P2] 人工校准 + 完整 A/B 基准任务集**
为目标项目建立 benchmark 题库（设计理解、改 bug、加 feature、避坑题各类），比较无注入/旧版/新版的真实任务成功率，用 Wilcoxon 检验。
为什么是 P2：工程投入高，需人工参与，应在评估体系基础稳固后再推进，否则 A/B 结论本身也不可信。

### 产出结构改进（针对问题3）

**[P0] CLAUDE.md 产出规范化**
强制结构：`CRITICAL RULES`（规则路，≥5条，每条含"如果...则..."结构 + 代码示例）→ `ARCHITECTURE & PHILOSOPHY`（叙事路）→ `COMMON PITFALLS`（踩坑，来自社区信号）。比例约束：规则区块占 60% 以上。
为什么是 P0：产出结构是最终价值载体，格式不对等于工作白做，且修复成本极低。

**[P1] 规则卡绑定源码证据**
每张规则卡生成时必须标注来源（哪个文件、哪段代码、哪个 issue），最终叙事只能引用已绑定证据，降低幻觉风险，同时支持后续知识溯源率统计。
为什么是 P1：显著提升可信度和可维护性，但需要 Stage 3 prompt 改造，工程量中等。

**[P2] 过程可观测性：阶段通过率与回退日志**
为每次运行记录各阶段通过/回退情况、成本、时延，支持后续分析哪个阶段最容易失败。
为什么是 P2：工程可观测性是长期健康运营的基础，但当前问题更根本，待 P0/P1 稳定后再加。

## 5. v0.9 路线图草案（最重要的3条方向）

**方向1：Stage 3.5 真阻断 + 结构化回写（最高优先）**
将验证从"软审查"改为"硬门控"，失败时生成结构化修改意见 JSON 并强制返工，而非继续流程。这一条同时解决了流程失效和评估误导两个核心问题。

**方向2：CLAUDE.md 产出结构固化 + 脚本硬校验**
用脚本强制检查最终输出：CRITICAL 规则数量、代码示例覆盖率、区块比例。低于阈值不允许产出，任何版本迭代都必须通过这组检查。这一条将质量回退变成可检测、可拦截的事件。

**方向3：卡片绑定源码证据，叙事只能引用已绑定证据**
规则卡生成时标注来源，expert_narrative.md 写作时限定只能引用卡片中已有的证据。这一条防止 v0.8 类型的"像专家但不可执行"幻觉，同时为后续 Faithfulness 评估打好数据基础。

---

*综合方法：三方结论通过"三方共识 > 两方共识 > 单方独有"的可信度权重排序；分歧处以工程成本和可测量性作为主要裁决依据。*
