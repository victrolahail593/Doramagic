## 二、7维度评分

### 评分总表

| 维度 | v0.6 | v0.7 | v0.8 | 说明 |
|------|------|------|------|------|
| 设计哲学传递 | 3 | 3 | 7 | v0.8 首次明确提出核心信念 |
| 心智模型清晰度 | 2 | 2 | 6 | v0.8 的"虚拟技术导师"比喻有帮助但不够深刻 |
| 因果链完整性 | 2 | 2 | 7 | v0.8 的"为什么这样设计"三段论是质的飞跃 |
| 踩坑故事质量 | 4 | 4 | 7 | v0.8 从条目式转为叙事体，引用了真实 Issue |
| 可执行性 | 7 | 7 | 5 | v0.6/v0.7 的 DO/DON'T 更直接可执行 |

| 整体知识注入效果 | 4 | 4 | 6 | v0.8 方向正确但有知识丢失 |

### 各维度详细评析

#### 1. 设计哲学传递 (v0.6: 3, v0.7: 3, v0.8: 7)

**v0.6/v0.7**: 完全没有显式的设计哲学陈述。"项目本质"的5个问答虽然解释了项目做什么，但没有回答"为什么选择这种方式"。AI 读完后知道 Superpowers 强制 TDD，但不知道为什么——是因为 AI 容易幻觉？是因为质量优先于速度？这些都没有传达。

**v0.8**: 开篇即以加粗引用给出核心信念——"AI 代理应该像训练有素的工程师一样思考，而不是像没有经验的码农"。紧接着解释了这个信念如何推导出所有设计决策。这是一个真正的设计哲学声明。

**不足**: v0.8 的设计哲学仍然偏通用，缺少 Superpowers 特有的、不可替代的设计取舍（例如：为什么选择 plugin/skill 架构而非 prompt injection？为什么不做运行时验证而是靠 AI 自觉遵守？）。得分7而非9是因为哲学的深度还不够——它说了"质量优先于速度"，但没有解释在什么场景下这个取舍会痛、以及维护者如何平衡。

#### 2. 心智模型清晰度 (v0.6: 2, v0.7: 2, v0.8: 6)

**v0.6/v0.7**: 没有任何心智模型。读者只能从规则列表中自行拼凑理解。

**v0.8**: 提出了"虚拟技术导师"的比喻——在每个关键决策点追问。这个比喻有一定帮助，但还不够"让人顿悟"。一个更强的心智模型应该能让 AI 自行推导出大部分规则。例如：如果心智模型是"Superpowers = 把 code review 的检查点前移到每个开发阶段"，那 AI 就能自行推导出为什么需要 brainstorming（review 前移到需求阶段）、为什么需要 TDD（review 前移到编码阶段）、为什么需要 two-stage review（review 本身也要分层）。

**不足**: "虚拟技术导师"这个比喻太泛——任何 linter、CI、code review 工具都可以这么说。它没有捕捉到 Superpowers 的独特之处：它是通过 prompt 工程来约束 AI agent 行为的。

#### 3. 因果链完整性 (v0.6: 2, v0.7: 2, v0.8: 7)

**v0.6/v0.7**: 规则是孤立的条目。"1% Rule"说必须调用 skill，但没有解释"因为 AI 会合理化跳过 skill → 导致整个工作流崩塌 → 所以即使 1% 可能也要调用"。DO/DON'T 列表给出了行为指令，但缺失了因果推理。

**v0.8**: "为什么这样设计"章节展示了三条清晰的因果链：
- Skills 而非固定工作流 → 因为软件开发不是线性的 → 所以需要可组合模块
- 强制 TDD → 因为 AI 太容易"作弊" → 先写测试让 AI 承认无知 → 打破幻觉
- 双阶段审查 → 因为 spec 和 quality 是正交维度 → 单次审查必然遗漏一个

这是 v0.8 最大的进步之一。得7分是因为因果链虽然清晰，但只覆盖了3个设计决策，还有很多规则（如 1% Rule、Fresh Subagent、Hard Gate）缺少因果解释。

#### 4. 踩坑故事质量 (v0.6: 4, v0.7: 4, v0.8: 7)

**v0.6/v0.7**: 每张卡片有"真实场景"段落和"影响范围"，但都是条目式的简短描述。例如 DR-101 的场景只有两句话，缺少排查过程和根因分析。最终注入文件中完全没有踩坑内容——规则的 DO/DON'T 格式不传递任何踩坑经验。

**v0.8**: "你一定会踩的坑"章节采用了第二人称叙事体，模拟了用户踩坑的完整过程："你兴冲冲安装了 superpowers，输入 `/superpowers:brainstorm`，结果报错..."。每个故事都有：
- 具体的操作步骤
- 错误的排查方向
- 真正的根因
- 对应的 GitHub Issue 编号和评论数

这比条目式的描述强得多。AI 读到这样的故事会形成"如果用户遇到类似问题，先查这几个方向"的直觉。

**不足**: 3个踩坑故事太少（v0.6/v0.7 有更多社区问题的覆盖），且故事之间缺少优先级排序的解释。

#### 5. 可执行性 (v0.6: 7, v0.7: 7, v0.8: 5)

**v0.6/v0.7**: 规则列表非常直接可执行。每条规则都有明确的 DO（做什么）和 DON'T（不做什么），AI 可以直接把这些当成行为约束来遵循。例如"Write failing test (RED phase) → Watch test fail → Write minimal code to pass (GREEN phase)"就是一个可直接执行的步骤。

**v0.8**: 专家叙事风格牺牲了部分可执行性。"为什么要强制 TDD 的 RED-GREEN-REFACTOR 循环？"解释了原因，但没有给出像 v0.6 那样具体的步骤指令。虽然底部有"速查表"，但只有6条规则，且描述比 v0.6 更粗糙。

**关键问题**: v0.8 在追求"让 AI 理解为什么"的过程中，丢失了"告诉 AI 具体怎么做"。一个好的知识注入应该两者兼具。

#### 6. 信息密度 (v0.6: 5, v0.7: 5, v0.8: 7)

**v0.6/v0.7**: 155行中有大量重复。"项目本质"5个问答有信息重叠；8条规则的 DO/DON'T 格式占了大量行数但很多是常识性内容（如"Follow platform-specific installation exactly"）；3个"Core Concepts"段落基本是卡片标题的复述。

**v0.8**: 52行高度浓缩。设计哲学1段（核心信念）、心智模型1段（比喻）、3个"为什么"（因果链）、3个踩坑故事（实战经验）、6条速查规则。每个部分都有独特的信息增量，几乎没有重复。用 1/3 的 token 传递了更多层次的知识。

#### 7. 整体知识注入效果 (v0.6: 4, v0.7: 4, v0.8: 6)

**v0.6/v0.7**: AI 读完后会知道一组规则并机械地遵守，但不理解项目的灵魂。它会按部就班地执行 TDD，但在遇到新场景时（比如规则冲突、边界情况），它无法自主判断该怎么做，因为它没有理解设计背后的意图。

**v0.8**: AI 读完后会理解"为什么 Superpowers 存在"、"它的核心信念是什么"、"常见的坑在哪"。在遇到新场景时，它可以回到设计哲学来推导应该怎么做。但它会缺少一些具体的执行细节（如 Hard Gate 规则在 v0.8 的最终注入文件中消失了、安装路径的平台差异等）。

## 三、产出差异分析

### 3.1 v0.6 vs v0.7：验证环节的实际效果

**结论：在本次实验中，v0.7 的验证环节没有产生任何可观测的改变。**

具体证据：
- 最终注入文件 inject/CLAUDE.md 逐字相同（155行对155行）
- 所有14张知识卡片内容完全相同
- 项目本质文件 00-project-essence.md 完全相同
- project_soul.md 完全相同
- 验证报告都是 14/14 通过，v0.6 有1个 warning (SK-20ILL.md)，v0.7 也有同一个 warning

这说明：
1. v0.6 的产出本身就通过了 v0.7 的验证标准，验证环节无事可做
2. 验证的硬校验（格式合规、可追溯性）在 v0.6 已经满足
3. 验证的软审查没有触发任何修改建议
4. **验证环节的价值可能更多体现在质量较差的提取结果上**——当 v0.6 的产出已经足够好时，验证环节只是确认了质量，没有提升质量

### 3.2 v0.7 vs v0.8：从"规则数据库"到"专家叙事"

**这是一次根本性的范式转变，带来了显著的正面变化和一些需要关注的回退。**

#### 正面变化

1. **新增设计哲学 (Q6)**: 首次给出了项目的核心信念声明，让 AI 理解"为什么"
2. **新增心智模型 (Q7)**: 首次提供了理解项目的比喻框架
3. **因果解释**: "为什么这样设计"章节用三个问答解释了三个关键设计决策的原因
4. **叙事体踩坑故事**: 从干巴巴的条目变成了有情节、有排查过程、有根因的故事
5. **Token 效率**: 从155行压缩到52行，信息密度大幅提升
6. **卡片质量提升**: v0.8 的卡片标题更精准（如 CC-001 从 "Core Trigger Mechanism" 改为 "Composable Workflow Automation"），DR-001 的 sources 从错误的 `SK-20ILL.md` 修正为正确的 `SKILL.md`
7. **规则重新提炼**: v0.8 的 DR-002 从 "Hard Gate" 改为 "Task plans MUST be bite-sized"——说明提取器对源码有了新的理解，识别出了不同的关键规则

#### 知识丢失

以下知识在 v0.6/v0.7 中存在，但在 v0.8 的最终注入文件中丢失或弱化：

| Hard Gate 规则（设计批准前不许写代码） | DR-002, CRITICAL 级别 | 严重——这是项目最重要的规则之一 |

## 四、各版本主要问题

### v0.6 的问题

1. **缺乏"为什么"**: 只告诉 AI 怎么做，不告诉它为什么。这导致 AI 在规则冲突或边界情况时无法自主判断。
2. **信息重复**: 项目本质、规则列表、Core Concepts 之间有大量重叠。
3. **机械式知识注入**: DO/DON'T 列表像一本操作手册，不像一个专家在传授经验。AI 会机械遵守但不会内化。
4. **缺乏社区智慧的深度利用**: 虽然引用了 Issue，但只是条目化的提及，没有提取出真正的排查经验。

### v0.7 的问题

1. **与 v0.6 产出完全相同**: 验证环节在本次实验中没有体现出任何增量价值。
2. **验证标准可能太宽松**: 100% 通过率说明要么验证标准设得太低，要么 v0.6 的质量已经足够好。无论哪种情况，验证环节都需要重新审视其设计目的。
3. **缺乏"软审查"的实际效果**: 软审查应该检查知识质量（如"是否有因果解释"、"是否有心智模型"），但似乎只检查了格式合规性。

### v0.8 的问题

1. **知识丢失**: 从155行压缩到52行的过程中，丢失了多条重要规则（最严重的是 Hard Gate）。
2. **可执行性下降**: 专家叙事告诉了 AI "为什么"，但"怎么做"的指导不足。速查表只有6条且描述过于简略。
3. **心智模型不够锐利**: "虚拟技术导师"的比喻太通用，无法推导出具体规则。需要一个更 Superpowers-specific 的模型。
4. **踩坑故事数量减少**: 从 v0.6/v0.7 的3个社区规则（DR-101/102/103）减少到3个叙事故事，但实际上 DR-102（安装路径问题）的信息在叙事中被吸收到了故事1里，而 DR-103（token 优化）的具体建议则完全丢失了。
5. **速查表的严重度标注不一致**: v0.6 中 1% Rule 是 CRITICAL，v0.8 速查表中降为 HIGH；v0.6 中 TDD 是 CRITICAL，v0.8 保持 CRITICAL。没有解释降级原因。

## 五、v0.8 方向评估

### 方向判断：正确，但执行不完整

**专家叙事方向是正确的**。核心理由：

1. **知识注入的本质是传递理解，不是传递规则**。一个真正理解项目的 AI 只需要少量规则 + 深层理解；一个不理解项目的 AI 即使给100条规则也会在边界情况犯错。v0.8 朝着"传递理解"的方向前进。

2. **Token 效率至关重要**。注入文件会占用 AI 的 context window，155行 vs 52行的差异在长对话中意义重大。v0.8 在更少的 token 中传递了更多层次的知识。

3. **因果链比规则列表更有生成力**。AI 理解了"为什么要 TDD"之后，可以自行推导出"如果已经有了完善的类型系统和 property-based testing，是否可以调整 TDD 的严格程度"。规则列表做不到这一点。

### 执行中的缺陷

1. **丢失关键规则是不可接受的**。专家叙事不应该替代规则——它应该包含规则。"为什么"和"怎么做"必须共存。
2. **速查表太弱**。应该是注入文件中最可执行的部分，但目前只有6条一行描述。
3. **设计哲学和心智模型缺乏深度**。目前停留在"正确但平庸"的水平，还没有达到"让人顿悟"的效果。

## 六、v0.9 建议

### 核心策略：保持叙事方向，补回可执行性，加深哲学深度

### 具体建议

#### 1. 注入文件结构：叙事 + 速查双层设计

```
# 项目名 — AI 知识注入

## 设计哲学（加深版）
- 核心信念 + 不可替代的设计取舍
- 在什么场景下这个取舍会痛？维护者如何平衡？

## 心智模型（锐化版）
- 一个能推导出80%规则的比喻/框架
- 验证：用心智模型推导3条规则，检验是否成立

## 为什么这样设计（保持v0.8质量）
- 每个关键设计决策的因果链

## 你一定会踩的坑（扩充到5个）
- 叙事体故事 + Issue 引用 + 根因

## 速查：关键规则（补回v0.6的具体性）
- 8-10条规则，每条有：规则描述 + 严重度 + 一句话"为什么"
- 比v0.6的DO/DON'T更简洁，但必须足够具体可执行

## 速查：工作流
- 核心工作流的步骤列表（从v0.6的WF卡片中提取）
```

#### 2. 验证环节增强

当前验证只检查格式合规，应增加：
- **知识完整性检查**: v0.6 的 CRITICAL/HIGH 规则是否在最终注入文件中都有对应？
- **因果链检查**: 每条规则是否有对应的"为什么"？
- **心智模型验证**: 用心智模型尝试推导规则，检查覆盖率
- **差异报告**: 如果是增量版本，自动对比前一版本丢失了什么

#### 3. 设计哲学提升

当前的设计哲学声明（"AI 代理应该像训练有素的工程师一样思考"）太通用。v0.9 应该提取 Superpowers 特有的、有争议性的设计取舍：
- "我们选择让 AI 自觉遵守而非运行时强制——因为 prompt engineering 是当前最灵活的约束机制"
- "我们宁愿多花 token 在 skill 上，也不接受 AI 跳过设计阶段——这是一个有意识的成本取舍"

#### 4. 心智模型锐化

用一个更精准的比喻替换"虚拟技术导师"：
- 候选："Superpowers 是给 AI 装的一套 pre-commit hooks，但在思考层面而非代码层面运行"
- 验证方法：用这个模型推导是否能得出 brainstorming（commit 前的 design review hook）、TDD（commit 前的 test hook）、two-stage review（push 前的 CI pipeline）

#### 5. 卡片到叙事的映射保障

Stage 4（专家叙事合成）应该：
- 有一个显式的 checklist，确认每张 CRITICAL/HIGH 卡片都在叙事中有对应
- 如果某条规则因为 token 预算被省略，在元数据中记录"被省略的规则及原因"
- 速查表应该至少覆盖所有 CRITICAL 和 HIGH 级别的规则

## 七、结论

| 版本 | 一句话评价 |
|------|-----------|
| v0.6 | 合格的操作手册——AI 能照着做，但不理解为什么 |
| v0.7 | 有验证但无差异——验证环节在本项目上未体现价值 |
| v0.8 | 正确的方向但执行不完整——传递了"为什么"但丢失了"怎么做" |

**v0.8 的核心突破是引入了设计哲学、心智模型和因果链——这些是知识注入的灵魂，v0.6/v0.7 完全没有。**

**v0.8 的核心缺陷是在"从规则到叙事"的转变中丢失了关键的可执行信息——Hard Gate、Fresh Subagent 等 CRITICAL/HIGH 规则不应该被省略。**

**v0.9 的核心任务是让两者共存：专家叙事提供理解，速查规则提供行动指南，二者互相加强而非互相替代。**
