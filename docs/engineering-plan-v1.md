## 一、项目在哪里开发、运行、存放

### 1.1 代码放在哪里？

**已决定：本地电脑 + Git 版本管理。**

代码现在存放在本地 Mac 电脑的 `~/Documents/vibecoding/Doramagic/` 目录下，用 Git 做版本管理。Git 就像一本无限撤回的笔记本——每次修改都会记录"谁改了什么、什么时候改的、为什么改"，任何时候都能翻回之前的版本。

**建议：尽快同步到 GitHub（一个在线代码仓库，像代码的云端备份）。**

为什么？两个原因：
1. 防灾——本地硬盘坏了不会丢失所有工作
2. 协作——未来如果有其他开发者参与，GitHub 是标准工具

现阶段不需要担心代码被公开，GitHub 可以创建私有仓库（只有你授权的人能看到）。

### 1.2 程序跑在哪里？

**建议：开发和测试在本地 Mac 上运行；用户通过远程 Mac mini 使用。**

当前已有一台远程 Mac mini（192.168.0.251），上面运行着 OpenClaw 平台（Doramagic 面向用户的入口）。这个格局不需要改变：

| 环节 | 运行位置 | 为什么 |
|------|---------|--------|
| 开发和调试 | 本地 Mac | 方便、快速、可以随时断点查看 |
| 用户使用 | 远程 Mac mini | 用户通过 Telegram 触发，OpenClaw 在远程执行 |
| 大规模测试 | 本地 Mac | 测试不应该影响用户可见的线上服务 |

**以后再定：是否需要云服务器。**

现阶段一台 Mac mini 够用。等到真正有持续的用户流量、或者需要 7×24 小时不间断运行时，再考虑云服务。过早买云服务器就像还没开店就租了个大仓库——浪费。

### 1.3 用什么编程语言？

**已决定：Python。**

为什么选 Python 而不是其他语言？三个理由：

1. **AI 生态最好** — Doramagic 的核心是调用大语言模型（LLM，就是 ChatGPT、Claude 这类 AI）做知识提取。Python 是 AI 领域的"普通话"，所有主流 AI 服务都优先提供 Python 接口。
2. **现有代码已经是 Python** — 已经有 `extract_repo_facts.py`（从代码仓库提取事实的工具）、`validate_extraction.py`（检验提取质量的工具）等核心脚本，全部用 Python 写成。推翻重来没有意义。
3. **开发速度快** — Python 的语法简洁，写同样功能的代码量大约是 Java 的三分之一。在产品验证阶段，速度比极致性能更重要。

现有的 Shell 脚本（像 `prepare-repo.sh`、`assemble-output.sh`，Shell 脚本就是一系列命令行指令的合集，类似于"宏"或"批处理"）会逐步迁移到 Python，因为 Python 更容易测试和维护。但不急，它们现在能正常工作。

### 1.4 需要什么工具？

**已决定 / 建议：**

| 工具 | 作用 | 类比 | 状态 |
|------|------|------|------|
| Python 3.12 | 编程语言运行环境 | 像是"发动机" | 已决定 |
| uv | Python 包管理器（管理第三方库的工具） | 像 App Store，负责安装和更新你用到的所有零件 | 建议（比传统 pip 快 10 倍） |
| Git | 版本管理 | 无限撤回的笔记本 | 已决定 |
| Claude Code | AI 辅助编程工具 | 像是一个随时在线的程序员搭档 | 已决定（这就是你现在看到的） |
| ripgrep | 代码搜索工具 | 像是代码世界的"Command+F"，但更快更强 | 已决定（Agent 探索的核心工具） |

不需要额外购买任何软件。以上工具全部免费或已有。

## 二、数据怎么存

### 2.1 用户数据需要数据库吗？

**现阶段不需要。**

为什么？因为 Doramagic 目前不是一个"用户注册登录"型产品。用户通过 Telegram 发消息 → OpenClaw 执行提取 → 返回结果。没有用户个人资料、没有购物车、没有历史记录页面。

类比：现阶段 Doramagic 像一家快餐车——客人走过来点餐、拿到食物、走了。快餐车不需要建会员系统。等到变成连锁餐厅时，才需要。

### 2.2 提取结果存在哪里？

**已决定：存为文件。**

每次提取一个项目的"灵魂"，会产生一组文件：

| 文件 | 内容 | 类比 |
|------|------|------|
| `repo_facts.json` | 项目的硬事实（有哪些命令、多少文件、用了什么技术） | 像是体检报告里的化验单——数字、事实、不带主观判断 |
| `concept_cards/` | 概念卡片（项目的核心概念是什么） | 像一套知识卡片 |
| `workflow_cards/` | 工作流卡片（事情是怎么一步步发生的） | 像流程图 |
| `decision_rule_cards/` | 决策规则卡片（什么情况下该怎么做） | 像"如果……那么……"的决策清单 |
| `CLAUDE.md` | 最终交付物（把所有知识浓缩成 AI 可以直接使用的格式） | 像一份写给 AI 的说明书 |

这些文件存在本地磁盘上，每个项目一个文件夹。

为什么不用数据库？因为这些数据的使用方式是"整个读、整个用"，不需要"查询某一条记录"。文件系统天然适合这种场景。就像你写一篇文章，存成 Word 文件就行，没必要把每段话存进 Excel。

### 2.3 领域图谱用什么存？

**已决定：SQLite。**

领域图谱是 V5 的新能力——当提取了同一个领域（比如 SEO）的多个项目后，Doramagic 可以把它们的知识汇总、对比，形成"领域共识"。这种数据需要频繁查询（"SEO 领域有多少项目提到了这个技术？"），所以需要数据库。

SQLite 就像一个 Excel 文件，但更快更可靠。具体来说：

- 它是一个**单文件数据库**——整个数据库就是一个文件（比如 `domain_map.sqlite`），可以随意复制、备份、传输
- **不需要安装额外的数据库服务器**——不像 MySQL 或 PostgreSQL 需要运行一个后台服务
- **Python 内置支持**——不需要安装任何额外的库
- **性能够用**——在预计的数据量级（几千到几万条知识原子）下，SQLite 的速度绰绰有余

类比：大型数据库（MySQL、PostgreSQL）像是一个需要专人管理的档案室，而 SQLite 像一个你自己桌上的文件柜——对于我们现在的规模，文件柜就够了。

### 2.4 需要考虑"数据搬家"吗？

**以后再定，但留好出路。**

"数据搬家"（数据迁移）指的是将来如果需要换存储方式（比如从 SQLite 换成更大的数据库），现有的数据能不能顺利搬过去。

现阶段的策略：所有数据用标准格式（JSON、SQLite），不用任何冷门格式。这就像用 Word 写文件而不是用某个小众软件——将来无论搬到哪里都能打开。

## 三、代码怎么组织

### 3.1 文件夹结构

**建议的目标结构：**

```
Doramagic/
├── src/                        ← 核心源代码（新建）
│   ├── doramagic/              ← Python 包主目录
│   │   ├── pipeline/           ← 提取管线（Stage 0 → Stage F 的所有阶段）
│   │   ├── agent/              ← Stage 1.5 Agent 探索模块（新增核心功能）
│   │   ├── fingerprint/        ← 项目指纹 + 跨项目对比
│   │   ├── domain/             ← 领域图谱
│   │   ├── health/             ← 知识健康检查
│   │   ├── tools/              ← Agent 使用的 5 个工具
│   │   └── models/             ← 数据结构定义
│   └── tests/                  ← 自动化测试
├── skills/                     ← OpenClaw 技能定义（现有，保持不变）
│   └── soul-extractor/
│       ├── SKILL.md            ← 技能描述文件
│       ├── scripts/            ← 现有脚本（逐步迁移到 src/）
│       └── stages/             ← Stage 指令文件
├── experiments/                ← 实验记录（现有，保持不变）
├── research/                   ← 研究文档（现有，保持不变）
├── docs/                       ← 产品 + 工程文档（现有，保持不变）
├── data/                       ← 运行时数据（新建）
│   ├── domain_maps/            ← 领域图谱 SQLite 文件
│   └── deprecation_events/     ← 废弃事件数据库
└── pyproject.toml              ← Python 项目配置（新建）
```

为什么这样组织？三个原则：

1. **新旧分离** — 现有能工作的代码（`skills/soul-extractor/scripts/`）不动，新代码放 `src/`。就像装修房子时先搭临时厨房，等新厨房建好再拆旧的。
2. **按能力分模块** — 每个子文件夹对应 V5 定义的一个核心能力（管线、Agent、指纹、图谱、体检），找代码时不用猜。
3. **数据和代码分开** — `data/` 存运行时产生的数据，`src/` 存程序代码。就像把菜谱和食材分开放。

### 3.2 现有脚本怎么融入？

现有 7 个核心文件：

| 文件 | 功能 | 融入计划 |
|------|------|---------|
| `prepare-repo.sh` | 克隆代码仓库 + 采样 + 生成代码摘要 | Phase 1 期间保持不变；Phase 2+ 逐步迁移到 Python |
| `extract_repo_facts.py` | 从代码中提取硬事实 | 直接迁入 `src/doramagic/pipeline/` |
| `collect-community-signals.py` | 收集社区信号（变更日志、GitHub 问题等） | 直接迁入 `src/doramagic/pipeline/` |
| `assemble-output.sh` | 把所有卡片组装成最终输出 | Phase 1 保持不变；Phase 2+ 迁移 |
| `validate_extraction.py` | 校验提取质量 | 直接迁入 `src/doramagic/pipeline/` |
| `validate_output.py` | 校验最终输出格式 | 直接迁入 `src/doramagic/pipeline/` |
| `extract.py` | 早期提取脚本 | 评估是否还需要，可能退役 |

策略是"先不拆旧的，新的建好后再替换"。任何时候旧脚本都能继续跑，不会出现"拆了旧的、新的还没好"的窘境。

### 3.3 新代码放在哪里？

| V5 新增能力 | 代码位置 | 首次开发阶段 |
|-------------|---------|-------------|
| Stage 1.5 Agent Loop | `src/doramagic/agent/` | Phase 1 |
| 5 个 Agent 工具 | `src/doramagic/tools/` | Phase 1 |
| 项目指纹 + 对比 | `src/doramagic/fingerprint/` | Phase 2 |
| 领域图谱 | `src/doramagic/domain/` | Phase 3 |
| 知识健康检查 | `src/doramagic/health/` | Phase 4 |

## 四、第一步做什么

基于 V5 的 Phase 0 和 Phase 1，最先开工的内容如下：

### 4.1 Phase 0：打地基（预计 1 天）

这是零成本的准备工作，像是在正式施工前铺好水电管道。

**具体任务：**

| 做什么 | 为什么 | 产出文件 |
|--------|--------|---------|
| 创建 `pyproject.toml` | Python 项目的"身份证"，定义项目名、依赖、版本号 | `pyproject.toml` |
| 创建 `src/doramagic/` 包结构 | 把零散脚本变成一个有组织的项目 | 目录结构 + `__init__.py` 文件 |
| 在知识原子数据结构中预留 `upstream_references` 字段 | 给未来的"知识供应链"功能预留接口——现在不用，但加了不增加任何成本 | `src/doramagic/models/atom.py` |
| 在独立性评分中加入 `normative_force` 调节因子 | 让"描述性知识"和"规范性知识"分开评判（"Python 3.12 已发布"是事实，"你应该用 Python 3.12"是规范——后者传播时更容易走样） | 修改现有评分逻辑 |

**怎么知道成功了？**
- 能在命令行运行 `python -c "import doramagic"` 不报错
- 现有脚本仍然能正常运行（不影响旧功能）

### 4.2 Phase 1：核心升级——Agentic 提取（预计 5-7 天）

这是价值最高的第一个功能迭代。用一句话概括：**让 Doramagic 从"看一遍就下结论"变成"带着问题反复翻阅代码"。**

类比：Phase 0 之前的 Doramagic 像一个学生，拿到课本翻一遍就写读书报告。Phase 1 之后，它变成了一个研究生——先粗读，列出疑问，然后带着疑问回到课本中找证据，最终写出有理有据的报告。

**具体任务（按先后顺序）：**

**第 1 步：Stage 1 新增假说生成（0.5 天）**

修改现有的 Stage 1（灵魂发现阶段）：在原来的 7 个核心问题之后，新增第 8 个问题——"列出 3-5 个你最想深入了解的问题或假说"。

产出：`hypothesis_list.json`——一份待验证的假说清单。

例子：如果分析一个 SEO 工具项目，假说可能是 "项目声称支持 AI 搜索优化，但代码中没有看到相关实现——是真的没做，还是藏在某个不明显的地方？"

**第 2 步：Stage 1.5 Agent Loop 核心开发（3-4 天）**

这是 Phase 1 的主体工程。需要开发：

- **编排脚本**：控制 Agent 一轮一轮探索代码的主程序
- **5 个工具**（Agent 用来"翻阅代码"的手段）：
  - `read_artifact` — 读取之前阶段已经产出的结构化数据
  - `list_tree` — 列出代码目录结构
  - `search_repo` — 在代码中搜索关键词
  - `read_file` — 读取具体文件的内容
  - `append_finding` — 把发现记录下来
- **5 个中间文件**（Agent 的"工作笔记本"——因为 AI 的记忆有限，需要把发现写到文件里）：
  - `hypotheses.jsonl` — 待验证的假说列表
  - `exploration_log.jsonl` — 每次翻阅代码的记录
  - `claim_ledger.jsonl` — "已证实 / 已推翻 / 待定"的知识声明
  - `evidence_index.json` — 证据索引（哪条证据来自哪个文件的第几行）
  - `context_digest.md` — 压缩版的"当前理解"

**第 3 步：Stage 3.5 增强验证（0.5 天）**

升级现有的质量验证关卡：如果 Agent 提出了一条知识声明，它必须指出证据来自哪里。没有证据支持的声明会被标记为"推断"（INFERENCE），不会进入最终输出的"设计哲学"部分。

这就像学术论文的引用要求——你说"这个框架的性能提升了 30%"，必须标注数据来源。

**第 4 步：A/B 验证（1-2 天）**

用 3 个项目做对照实验，证明新版本确实比旧版本好。

### 4.3 怎么知道 Phase 1 成功了？

| 指标 | 目标 | 怎么测量 |
|------|------|---------|
| WHY 提取质量 | 评分提升 ≥ 0.5 分（满分 3 分） | 用现有的 WHY 评分标准，对比新旧版本 |
| 大项目信息保留度 | 明显提升（旧版容易漏掉大项目的重要信息） | 人工检查 wger 项目的提取结果 |
| Agent 走偏率 | < 10%（Agent 探索的内容中，偏离主题的比例） | 检查 exploration_log 中无用工具调用的占比 |
| 降级可靠性 | Agent 失败后能正常退回旧版 | 手动触发降级，检查输出是否正常 |
| 成本增幅 | ≤ 2 倍 | 对比 API 调用费用 |

## 五、怎么测试和验证

### 5.1 用什么项目做测试？

**已决定：三个项目，分别代表小、中、大规模。**

| 项目 | 规模 | 为什么选它 | 已有基线数据？ |
|------|------|-----------|--------------|
| python-dotenv | 小（~2000 行代码） | 从 exp01 到 exp08 一直在用，数据最丰富 | 有（多个版本的提取结果 + 评分） |
| wger | 大（~12000 文件） | exp08 验证过跨领域适应性 | 有（MiniMax 提取结果 + 质量分析） |
| 1 个 SEO 项目 | 中 | 4 个 SEO 项目已提取，为 Phase 2-3 铺路 | 有（4 份 CLAUDE.md） |

### 5.2 怎么知道新版本比旧版本好？

用 A/B 对照测试（也叫"控制实验"——和医学临床试验的逻辑一样）：

- **A 组（对照组）**：用当前版本（v0.9 + P0 补丁）提取
- **B 组（实验组）**：用新版本（加了 Stage 1.5 Agent Loop）提取
- **评分方式**：用已有的 Benchmark 基础设施——Claude Sonnet 回答问题，Gemini 判分
- **关键维度**：WHY 因果链质量、UNSAID 陷阱捕获率、整体信息保留度

为什么不直接问"新版输出看起来好不好"？因为人的感觉会骗人——看到更长的输出会觉得更好，但信息密度可能更低。量化评分消除了这种偏差。

### 5.3 不同阶段的验证方式

| 开发阶段 | 验证方式 | 成功标准 |
|---------|---------|---------|
| Phase 0 | 冒烟测试（最基本的"能跑不能跑"） | 项目结构正确，现有功能不受影响 |
| Phase 1 | A/B 对照（3 项目） | WHY +0.5，走偏率 <10%，成本 ≤2x |
| Phase 2 | Gold Set 验证（用 4 个 SEO 项目的已知结论做对照） | 项目指纹能正确识别相似项目 |
| Phase 3 | A/B 对照（有/无图谱注入） | 有图谱的提取结果质量更高 |
| Phase 4 | 人工审查 + 指标核对 | 7 类信号的标注准确率 |

## 六、怎么部署给用户

### 6.1 当前用户使用流程

```
用户在 Telegram 发消息 → OpenClaw 平台收到消息 → 调用 soul-extractor 技能 → AI 执行提取 → 结果返回给用户
```

### 6.2 这个流程需要改吗？

**短期（Phase 0-3）：不需要改。**

为什么？因为 Phase 1 的 Agentic 提取是在"技能内部"的升级——对用户来说，发的消息一样，收到的结果更好。就像餐厅升级了后厨设备，但顾客还是在同一个窗口点餐、同一个窗口取餐。

唯一的变化是**提取时间会变长**——旧版可能 3 分钟出结果，新版可能 10-20 分钟（因为 Agent 要做多轮探索）。需要给用户一个"进度提示"，比如 Telegram 中定时发送"正在探索第 3 轮，已发现 2 条设计决策..."

### 6.3 API 什么时候需要？

**以后再定，但 V5 已明确"API 是一等公民"。**

API（应用程序接口，就是"让别的程序调用 Doramagic"的标准通道）在以下情况下才需要：

1. 有第三方想把 Doramagic 嵌入自己的产品
2. 需要批量处理（一次提取 100 个项目）
3. 需要比 Telegram 更丰富的交互界面（比如网页版）

预计 Phase 4（知识健康检查）之后才有建 API 的必要性。在那之前，Telegram + OpenClaw 完全够用。

**关键设计原则**：从第一天起，所有核心功能都写成"可被调用的函数"，不要写成"只能从命令行运行的脚本"。这样未来加 API 时，只需要包一层接口，不需要重写逻辑。就像做饭——你可以在家用自己的碗吃，也可以装进外卖盒——关键是菜做好了，装进什么容器都行。

## 七、风险和应急方案

### 7.1 最可能出问题的地方

按风险从高到低排列：

**风险 1：Agent 探索质量不稳定（高风险）**

问题：AI Agent 每次探索的路径不同，可能时好时坏。就像同一个研究生，有时候研究思路对了出好论文，有时候跑偏了浪费时间。

应急方案：
- **降级路径已内置** — Agent Loop 失败时自动退回旧版提取，结果不会比现在差
- **弱模型不进 Agent Loop** — 能力不够的模型（如 MiniMax）直接跳过 Agent 阶段，用增强版的旧方法
- **三层预算控制** — 设置工具调用次数上限、费用上限、和"连续几轮没新发现就停止"的刹车机制

**风险 2：假说质量是瓶颈（高风险）**

问题：Agent 的探索方向由 Stage 1 产出的假说驱动。假说不好，Agent 再努力也找不到有价值的东西。就像侦探手里的线索质量决定了破案效率。

应急方案：
- Phase 1 的 A/B 验证会专门评估假说质量
- 假说生成的 prompt（给 AI 的指令）设计为可快速迭代
- 如果假说质量持续不行，可以考虑用更强的模型专门做假说生成（类似于请资深侦探给线索，再派助手去跑腿）

**风险 3：模型 API 费用超预期（中风险）**

问题：Agentic 提取需要多轮调用 AI，费用是旧版的约 2 倍。如果控制不好，可能更高。

应急方案：
- 三层预算控制（硬上限 + 软上限 + 信息增益停止）
- 小项目默认 5 轮探索，大项目 10 轮——不无限制探索
- 预计单次提取费用：小项目 ~$0.11，大项目 ~$0.31——短期内完全可控

**风险 4：领域图谱冷启动（中风险，Phase 3 才出现）**

问题：领域图谱需要同一领域足够多的项目才能产生有价值的"共识"。一开始项目太少，图谱不够丰富。

应急方案：
- 已选定 SEO/GEO 作为第一个领域，已有 4 个项目的提取数据
- Phase 3 会再补充 3-4 个 SEO 项目，达到 7-8 个的"最小可用规模"
- 图谱功能设计为"有则增强、无则不影响"——没有图谱时，提取功能正常运行，只是少了"领域对比"视角

**风险 5：OpenClaw 平台限制（低风险）**

问题：Doramagic 跑在 OpenClaw 上，如果 OpenClaw 有 bug 或不支持某个功能，可能卡住。

应急方案：
- 核心提取逻辑与 OpenClaw 完全分离——提取引擎是"平台无关"的 Python 代码
- OpenClaw 只是用户触发入口，换一个触发方式（比如命令行、网页）不影响核心逻辑
- 已发现的 OpenClaw 已知问题（废弃配置字段阻断命令行、nohup 不稳定等）都有已知的解决方法

### 7.2 哪些决策现在不需要做？

| 决策 | 为什么可以以后再定 | 建议什么时候定 |
|------|-------------------|--------------|
| 是否需要云服务器 | 一台 Mac mini 够用，等有真实用户流量再说 | 有持续日活用户时 |
| API 接口设计 | 当前只需 Telegram 触发，API 需求未出现 | Phase 4 之后 |
| 选择哪个数据库替代 SQLite | SQLite 在当前规模下完全够用 | 数据量超过 100 万条时 |
| 是否支持多语言界面 | 当前用户群体单一 | 有海外用户需求时 |
| 是否做网页前端 | Telegram 是足够的交互方式 | 需要复杂可视化（如热力图）时 |
| 定价模型 | 产品还在验证阶段，定价为时过早 | 产品价值得到用户确认后 |
| 第二个领域选什么 | 第一个领域（SEO）还没做好 | SEO 图谱达到 Beta 后 |

## 九、总结：三个关键认识

1. **Phase 1 是价值最高的起点** — Agentic 提取直接提升核心产品质量（从"翻一遍"到"反复翻阅"），5-7 天就能看到效果。其他所有后续功能都建立在这个基础上。

2. **架构设计的核心原则是"能力叠加而非替代"** — 每个新 Phase 都是在现有基础上加能力，不是推翻重来。Phase 1 加了 Agent 但旧管线还在；Phase 3 加了图谱但没图谱也能用。这意味着任何一步走不通，都不会影响已有的成果。

3. **最大的风险不是技术，是"做对了没有"** — 工程实现有成熟方案，真正的风险在于：Agent 的假说好不好？领域图谱有没有人用？健康检查有没有价值？所以每个 Phase 都以 A/B 验证结尾——用数据说话，不靠感觉。
