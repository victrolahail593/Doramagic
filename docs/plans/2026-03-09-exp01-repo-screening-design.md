# AllInOne exp01 候选验证仓库筛选设计

**日期**：2026-03-09
**文档类型**：实验前置设计
**目标**：为 `exp01` 建立一套可复用的候选验证仓库筛选方法，并给出首批推荐样本池。

---

## 1. 设计目标

`exp01` 的首个验证仓库不应依靠主观直觉选择，而应通过一套轻量但明确的筛选方法得出。该方法必须同时服务两个目标：

1. 让首个实验尽快暴露 AllInOne 的核心价值是否成立；
2. 避免因样本过大、过乱或过冷门，导致实验失败原因无法归因。

## 2. 设计原则

- 小而真：优先真实小仓库，不选 toy repo
- 单一职责：避免多子系统、多角色混杂
- 文档友好：必须有可读 README / docs
- 问题足够丰富：最好有少量 Issue，便于后续扩展社区半魂
- 可人工评估：问题域不能陌生到无法判断回答质量
- 先低风险：第一个实验优先选择实现边界清晰、概念稳定的库

## 3. 筛选方法

### 3.1 硬过滤（必须满足）

- 仓库职责单一
- 有 README 或基础文档
- 源码结构清晰，不是明显 monorepo
- 有测试或示例，便于推断真实使用方式
- 没有明显重型基础设施依赖

### 3.2 软评分（用于排序）

- 范围清晰度
- 代码表面积
- 文档质量
- Issue 信号密度
- 规则密度
- 工作流密度
- 人工可评估性
- 提取友好度
- 隐藏复杂度风险

### 3.3 当前阶段的关键判断

对于 `exp01`，**可验证性优先于代表性**。首个样本不需要代表所有目标仓库，只需要足够清楚地回答：“灵魂提取之后，模型回答是否更好？”

## 4. 当前推荐结构

建议输出三层候选池：

- **Primary**：默认首选，最适合做第一次验证
- **Secondary**：备选样本，可在首选样本不合适时替换
- **Stretch**：社区信号更强，但规模或复杂度更高，不建议作为第一样本

## 5. 当前推荐结果

基于本地项目标准与 GitHub 官方仓库页面的快速校验，当前建议：

- **Primary**：`python-dotenv/python-dotenv`
- **Secondary**：`jpadilla/pyjwt`、`pallets/itsdangerous`
- **Stretch**：`psf/requests`

## 6. 设计备注

- 候选仓库的“规模”判断当前属于**基于 GitHub 仓库表面积的工程推断**，不是已测 LOC；正式定稿前应做一次本地快速拉取或 Repomix 打包校验。
- `Requests` 虽然社区信号极强，但更适合作为第二阶段或对照样本，而不是第一样本。
- `itsdangerous` 极适合小样本验证，但社区信号偏弱，可能削弱后续社区半魂扩展价值。
