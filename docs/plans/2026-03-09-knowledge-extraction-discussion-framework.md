# 开源知识获取：技术解决方案讨论框架

**日期**：2026-03-09
**定位**：AllInOne 最核心子项目 — 没有这个，一切为零
**版本**：v2（基于全部 48 份文档 + 三方复盘 + OpenClaw 用户调研的综合重构）

---

## 第一性原理分解

### 根问题

AllInOne 要成为"以开源知识为燃料、以 AI 为引擎的个人数字化操作系统"。这个愿景拆到底，依赖一条因果链：

```
用户说需求 → 系统找到相关开源项目 → 从项目中提取知识 → 用知识驱动AI交付解决方案
         ↑                              ↑
      （匹配问题）                    （本子项目）
```

**本子项目只解决一个问题**：如何从一个开源项目中，提取出能让 AI 变强的结构化知识？

### 为什么这是最重要的

用第一性原理拆解为什么其他事都可以等，唯独这个不行：

1. **如果提取出的知识没有用** → AllInOne 的核心假设崩塌，整个项目方向需要重新审视
2. **如果提取出的知识有用** → 后续的匹配、编排、交付才有了燃料，才值得投入
3. **当前状态**：47 份文档、13,600+ 行文本资产、12 套 Prompt 模板、0 行业务代码、**0 次真实验证**

因此：**在证明"知识提取有用"之前，不应该做任何其他事。**

## 一、我们到底要提取什么？

### 1.1 六类知识（已确认共识）

三方复盘一致确认的知识分类体系：

| 类型 | 含义 | 来源 | 示例 |
|------|------|------|------|
| **WHAT** | 做什么 — 概念、对象、边界 | 代码 + 文档 | "dotenv 从 .env 文件加载环境变量" |
| **HOW** | 怎么做 — 流程、调用链、状态变化 | 代码 + 测试 | ".env 文件的发现→解析→注入流程" |
| **IF** | 什么条件 — 规则、校验、分支逻辑 | 代码 + 测试 | "变量覆盖规则：系统环境变量优先" |
| **STRUCTURE** | 怎么连 — 模块关系、数据流、依赖 | 代码 | "CLI 模块调用核心解析模块" |
| **WHY** | 为什么 — 设计决策、trade-off、演进意图 | Issue/PR/Discussion | "为什么选择覆盖而非追加" |
| **UNSAID** | 没说的 — 未文档化的惯例、坑、常识 | 社区讨论/长尾Issue | "Docker 中 .env 路径问题" |

**关键洞察（来自 Codex 技术决策报告）**：前四类（WHAT/HOW/IF/STRUCTURE）主要来自代码，后两类（WHY/UNSAID）主要来自社区。这就是"代码半魂"和"社区半魂"的划分依据。

### 1.2 知识的载体：卡片

提取的知识以卡片（Card）形式存储。**exp01 聚焦三种核心卡片**（Codex 技术决策报告建议：不要一开始追求所有卡片类型）：

| 卡片类型 | 承载的知识 | 对应阶段 |
|----------|-----------|----------|
| **concept_card** | WHAT — 核心概念、边界、定义 | L1 深潜 |
| **workflow_card** | HOW — 流程、步骤、状态变化 | L2 深潜 |
| **decision_rule_card** | IF + WHY — 规则、条件、设计理由 | L3 深潜 |

其他两种（contract_card、architecture_card）留给后续阶段。

### 1.3 来自 OpenClaw 调研的启示

OpenClaw 生态揭示了一个关键事实：**Agent 的"灵魂"目前依赖用户手写**（SOUL.md/MEMORY.md），缺乏自动提取能力。

- ClawHub 排名第一的是"Agent 自进化"类 skill（50,000+ 下载）—— 用户最渴望的是 Agent 自主获取知识的能力
- 用户最大痛点之一是"记忆膨胀失真"—— 扁平 Markdown 记忆越用越差
- Agent 无法准确操作专业软件 —— 因为缺乏领域知识注入

**这验证了 AllInOne 的核心假设**：结构化知识提取是有真实需求的。但这也意味着提取出的知识必须是高质量的、结构化的、可被 AI 真正消费的 —— 不是"更多的 Markdown 文本"，而是精炼的知识资产。

## 二、提取方法论：鹰眼 + 深潜

### 2.1 核心思想（六条工程原则）

这套方法论经三方独立验证，核心原则是：

1. **先画像，后提取** — 先知道仓库是什么，再决定怎么提取
2. **先鹰眼，后深潜** — 先找灵魂候选区，再精读
3. **先证据，后结论** — 卡片必须可追溯到源码/Issue
4. **先卡片，后汇总** — 先产出独立可用的知识对象，再做整体收敛
5. **先保守，再补全** — 不确定就标记，不强行脑补
6. **代码半魂优先保证真实性** — 社区半魂补充使用经验

### 2.2 完整流水线

```
Repo Input
  → Repo Profile（画像：规模/语言/结构/适合度）
  → Repomix --compress（代码打包为 LLM 可消费的上下文）
  → Eagle Eye 1: 项目身份 + 核心概念（5-10个）
  → Eagle Eye 2: 核心模块 + 模块关系（4-8个）
  → Eagle Eye 3: Soul Locus 评分排序 → Deep Dive Queue
  → L1 Deep Dive: concept_card（做什么）
  → L2 Deep Dive: workflow_card（怎么流）
  → L3 Deep Dive: decision_rule_card（为什么这样）
  → [阶段二] Community Pipeline: WHY/UNSAID 知识
  → Merge & Validate → Judge → Gap Queue
  → Final Soul Cards
```

### 2.3 鹰眼阶段详解

**输入**：Repomix 压缩打包 + repo_profile
**目标**：回答三个问题——这个项目是什么？它由哪几块组成？灵魂藏在哪里？

**Soul Locus Score**（8 维加权评分，决定深潜优先级）：

| 维度 | 权重 | 含义 |
|------|------|------|
| Domain Centrality | 20% | 是否承载核心业务概念 |
| Execution Influence | 15% | 是否影响主执行路径 |
| Orchestration Power | 15% | 是否编排多个模块 |
| Data/State Gravity | 15% | 是否拥有关键数据结构 |
| Rule Density | 10% | 条件判断/校验密度 |
| Boundary Importance | 10% | 外部接口/API 重要性 |
| Cross-Module Bridge | 10% | 跨模块连接作用 |
| Concept Density | 5% | 核心术语/抽象密度 |

85-100 = 必须深潜，70-84 = 应纳入首轮，55-69 = 相关时再看，<55 = 仅做上下文

**合格标准**（六条硬规则）：
1. 能一句话解释项目用途
2. 每个核心概念绑定具体文件路径
3. 模块按职责划分（不是目录复述）
4. Soul Locus 分数有区分度（不能全挤一个区间）
5. 概念/模块/loci 能互相对应
6. 存在不确定性声明

**对 python-dotenv 的适配考量**：这是一个 <5K 行的小项目，三步鹰眼可能过重。但第一次实验应完整走一遍流程来验证方法论本身，后续再根据项目规模决定是否精简。

### 2.4 深潜阶段详解

**严格分三层，不混在一个 prompt 里**（混在一起最容易出现概念和流程互相污染）：

| 层级 | 目标 | 输入 | 产出 |
|------|------|------|------|
| **L1** | 做什么 | locus 完整代码 + 鹰眼产物 | concept_card |
| **L2** | 怎么流 | locus 代码 + L1 卡片 | workflow_card |
| **L3** | 为什么 | locus 代码 + L1/L2 卡片 | decision_rule_card |

**输入要求**：深潜阶段使用**完整代码**（不压缩），因为 WHY 知识藏在函数体内部的注释和异常处理逻辑中，压缩版会丢失这些信息（来自 Gemini Repomix 调研的关键结论）。

### 2.5 Repomix 的角色与边界

Repomix 是提取管线的入口，但需要理解它的能力边界（来自 Gemini Repomix 调研）：

| 模式 | 做什么 | 丢什么 | 适用阶段 |
|------|--------|--------|----------|
| `--compress` | 保留函数签名、类结构、接口、导入关系 | 删除函数体、内部逻辑、中间变量 | **鹰眼**（定位灵魂在哪） |
| 完整版 | 保留全部代码 | 无 | **深潜**（提取具体知识） |
| `--split-output` | 按 token 限制切分 | 无 | 大项目降级 |

**python-dotenv 预期规模**：<5K 行核心代码，压缩后约 2-5K tokens，完整版约 15-30K tokens —— 完全在 128K 上下文窗口内，不需要分片。

## 三、卡片格式与质量标准

### 3.1 统一格式

```yaml
---
card_type: concept_card  # concept_card | workflow_card | decision_rule_card
card_id: CC-001
repo: python-dotenv
commit: <sha>
confidence: high  # high | medium | low
evidence_level: E1  # E1=源码/测试 E2=maintainer PR E3=社区共识 E4=弱证据
inference_strength: explicit  # explicit | strong_inference | weak_hypothesis
status: draft  # draft | active | deprecated | rejected
schema_version: 1
---
# 卡片标题

## 正文（Markdown）

## Evidence
- [E1] src/dotenv/main.py:42-58
- [E1] tests/test_main.py:120-135
```

### 3.2 质量门控

卡片的质量直接决定 A/B 验证的成败。低质量卡片反而会让 AI 回答更差（技术决策报告警告的"上下文装配失控"风险）。

**每张卡片必须满足**：
- 有至少一条 E1/E2 级证据
- 置信度标注诚实（不确定就标 low）
- 不包含从代码中无法推断的内容
- 正文简洁可用，不是代码摘要

## 四、验证体系：A/B 实验

### 4.1 核心假设

| 假设 | 内容 | 如果成立 | 如果不成立 |
|------|------|----------|------------|
| **H1** | 加载灵魂后 LLM 回答更好 | 方向正确，继续 | 根本性反思 |
| **H2** | 纯代码半魂就有效果 | 可以先不做社区半魂 | 必须引入社区半魂 |
| **H3** | Repomix+鹰眼+深潜链路可行 | 方法论验证 | 需要替代方案 |

### 4.2 问题集设计

10-15 题，覆盖 5 类知识，**关键设计原则**：

| 类别 | 考察点 | 为什么重要 |
|------|--------|------------|
| 领域概念 | 概念定义/边界 | 验证 concept_card 价值 |
| 工作流 | 流程顺序/依赖 | 验证 workflow_card 价值 |
| 规则/条件 | 条件/限制/异常 | 验证 decision_rule_card 价值 |
| 风险/踩坑 | 常见误区 | 验证 WHY/UNSAID 知识的增量 |
| 方案建议 | 迁移/排查 | 验证知识的实际可用性 |

**检查清单**：
- 至少 2 个容易诱发幻觉的问题（验证灵魂是否能减少胡说）
- 至少 2 个需要结构化理解的问题（验证卡片比原始代码更好用）
- 能区分"泛化回答"和"项目定制回答"（这是核心价值所在）

### 4.3 评估维度

| 维度 | 考察什么 | 为什么重要 |
|------|----------|------------|
| **Groundedness** | 回答是否基于项目证据 | 区分真知识 vs 编造 |
| **Specificity** | 是否具体到模块/规则 | 区分有灵魂 vs 泛泛而谈 |
| **Usefulness** | 是否可直接行动 | AllInOne 要交付的是能力 |
| **Risk** | 是否存在误导 | "Almost Right"是最大威胁 |
| **Uncertainty Handling** | 是否诚实处理不确定 | 知道不知道 > 假装知道 |

**评估原则**：
- 长答案不天然更好
- 自信但错误应重罚
- 合理承认不确定性是正向行为
- 优先评价"是否基于项目知识"，而不是"是否写得流畅"

## 六、风险与回退策略

### 6.1 已识别的核心风险

| 风险 | 级别 | 来源 | 应对 |
|------|------|------|------|
| **灵魂提取没有显著价值** | 最高 | 核心假设未验证 | exp01 就是为了验证这个 |
| **"Almost Right"调试陷阱** | 高 | 三方都识别，零解方案 | 实验中观察，暂不解决 |
| **Prompt 模板实测效果差** | 中 | 12 套模板零实测 | 实验中迭代调整 |
| **上下文装配失控** | 中 | 卡片过多导致 prompt 膨胀 | 从最少卡片开始 |
| **评估偏差** | 中 | 人工评估的主观性 | Judge Rubric + 多轮复核 |
| **python-dotenv 过于简单** | 低 | 可能测不出差距 | 备选 pyjwt/itsdangerous |

### 6.2 回退路径（来自技术决策报告）

如果执行不顺，按以下顺序逐级回退：

1. **换更小的验证范围** — 只对 python-dotenv 的核心解析模块做提取
2. **减少卡片类型** — 只做 concept_card，不做 workflow 和 rule
3. **简化鹰眼** — 三步合一，单次 prompt 完成定位
4. **减少评估轮数** — 5 题而非 15 题
5. **最小回退底线** — `repo_profile + concept_cards + 3题A/B` 三件事

## 八、执行路径

### 推荐方案：方案 B（极简验证路线）

技术决策报告比较了三条路线，结论明确：

| 路线 | 验证速度 | 复杂度 | 当前适配度 |
|------|----------|--------|------------|
| A: 平台先行（全栈重型） | 低 | 极高 | 低 |
| **B: 极简验证（推荐）** | **高** | **低** | **最高** |
| C: 静态分析优先 | 中 | 高 | 中 |

### 执行步骤

```
Phase 0: 环境准备（0.5天）
├── 确认 Node.js / Repomix 可用
├── 克隆 python-dotenv
├── 跑一次 repomix --compress，确认 token 规模
└── 快速审读 12 套 Prompt 模板的 System Prompt 部分

Phase 1: 鹰眼提取（0.5-1天）
├── 生成 repo_profile（手工 + 自动）
├── Eagle Eye Step 1: 项目身份 + 核心概念
├── Eagle Eye Step 2: 核心模块 + 关系
├── Eagle Eye Step 3: Soul Locus 评分 → Deep Dive Queue
└── 检查：是否通过六条硬规则

Phase 2: 深潜提取（1-2天）
├── 对 top 2-3 loci 跑 L1（concept_card）
├── 对相同 loci 跑 L2（workflow_card）
├── 对相同 loci 跑 L3（decision_rule_card）
├── 整理为 soul/ 目录
└── 记录：每步的 token 消耗、耗时、人工修正

Phase 3: A/B 验证（1天）
├── 设计问题集（10-15题，覆盖 5 类知识）
├── A组：无灵魂基线（同模型直接回答）
├── B组：system prompt 加载灵魂卡片
├── 用 Judge Rubric 逐题评分
└── 人工复核 + 写结论

Phase 4: 复盘与决策（0.5天）
├── 填写 run log 模板
├── 写 exp01 实验报告（填入主文档预留区）
├── 回答核心问题：灵魂是否有用？
└── 决定下一步：成功→exp02（社区半魂） / 失败→回退分析
```

**总预估：3-5 天**

## 九、与 AllInOne 全局的关系

本子项目的结果直接决定 AllInOne 的后续路径：

```
exp01 结果
├── 成功（灵魂有用）
│   ├── exp02: 社区半魂验证（GitHub Issues → decision_rule_card）
│   ├── exp03: 多项目验证（pyjwt / itsdangerous）
│   ├── 架构收敛：存储/检索/装配的技术选型
│   └── 最终：集成到 AllInOne 的知识获取管线
│
├── 部分成功（某些卡片有用，某些没用）
│   ├── 分析哪类知识有效、哪类无效
│   ├── 精简卡片体系
│   └── 在第二个仓库上重复验证
│
└── 失败（灵魂没用）
    ├── 分析失败原因（提取质量差？问题设计差？仓库选错？）

    └── 如果根本性失败 → 重新审视 AllInOne 方向
```
