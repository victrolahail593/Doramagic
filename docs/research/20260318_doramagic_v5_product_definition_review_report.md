## 0. 执行摘要

这份文档的优点很明确：

- 产品灵魂清晰，核心主张稳定：`代码说事实，AI 说故事`、`不教用户做事，给他工具`
- 对单项目提取、多项目综合、平台适配、质量门控形成了较完整的体系化思考
- 已经明显超过“灵感文档”阶段，开始具备工程落地意识

但当前版本的主要问题不是缺想法，而是：

> **产品定义已经同时覆盖了多个方向，研究视角强于产品收敛，导致文档看起来“开发就绪”，但严格来说仍未完成产品边界冻结。**

最关键的结论有四个：

1. `v1` 目标用户没有真正冻结。文档名义上说核心用户是开发者，但正文已同时扩展到项目作者、选型者、创业者、投资人、PM/CTO。
2. `v1` 交付物没有真正冻结。文档名义上以 `Skill` 为核心，但又并行承诺热力图、领域真相、体检、对比、缝合等多个产品输出。
3. 研究型能力与首发能力没有完成分层。领域图谱、知识通胀、错误共识、热力图都很有价值，但当前更适合作为 `v1.5+` 增强项，而不是 `v1` 的硬承诺。
4. “开发就绪”的判断偏乐观。真实世界端到端验证、失败兜底、指标定义、License 风险策略、成本与时延目标仍未写实。

因此，本报告建议：

> **将 v5.2 收敛为“v5.3 开发冻结版”时，只保留一个主用户、一个主交付、一个关键路径。**

建议的一句话定义是：

> **Doramagic v1 是一个面向开发者的知识提取与组装系统，输入用户的模糊需求，输出带溯源和限制说明的可部署 OpenClaw Skill。**

## 1. 总体判断

### 1.1 这份文档已经做对了什么

- 产品哲学稳定，没有在多轮研究后失魂
- 单项目管线和多项目管线之间的关系定义得较清楚
- 已意识到平台适配、静态验证、模板防腐、证据绑定等工程现实
- 质量门控比多数同类“AI 产品方案”更严谨

### 1.2 这份文档当前最大的问题是什么

- 目标用户扩张过快
- 输出形态扩张过快
- 研究能力写进了首发承诺
- 多个模块定义到了“机制层”，但还没收敛到“产品验收层”

### 1.3 本次评审的主判断

这不是一份“方向错了”的文档。  
这是一份“方向对，但范围比首发产品大了 2 到 3 倍”的文档。

## 2. 关键风险清单

### 2.1 还没有考虑清楚的问题

1. `v1` 到底是谁的产品  
当前文本同时服务开发者、项目作者、技术选型者、创业者、投资人、PM/CTO。不同角色的任务触发、容忍时延、可接受输出密度完全不同，不可能在首发版本内同时满足。

2. `v1` 的主交付到底是什么  
文档同时承诺 `SKILL.md`、热力图、`DOMAIN_TRUTH.md`、`HEALTH_CHECK.md`、`SOUL_DIFF.md`、`STITCH_MAP`。这会让产品体验、文档结构、质量门控和路标都失焦。

3. “给工具不给建议”与“替用户做知识选择”之间的边界  
如果系统最终会自动做候选筛选、冲突解决、Skill 组装，那么它已经不是纯观察工具，而是半自动决策系统。这里需要明确定义允许替用户做什么选择，不允许做什么选择。

4. 成功指标缺失  
缺少对交付时延、单次成本、通过率、采纳率、用户修改次数、失败兜底率的定义。没有这些指标，后续会陷入“看起来不错，但不知道有没有变好”。

5. 冷启动策略没有定死  
文档明确说首个领域待定，但领域图谱、健康检查、通胀、错误共识都建立在领域积累之上。如果首个领域不冻结，后续所有“跨项目智能”都难进入真实迭代。

### 2.2 技术难点

1. 三层知识表示难以稳定  
`Code Fingerprint`、`Knowledge Atom`、`Soul Graph` 要同时满足可比较、可解释、可扩展，难度很高。

2. 多信号阈值校准难  
`ALIGNED / MISSING / ORIGINAL / DRIFTED / DIVERGENT / CONTESTED` 这类信号在跨领域时容易失真，且阈值需要大量标注样本。

3. 错误共识检测容易“看起来聪明”  
风险信号体系有价值，但精度仍偏研究阶段，若过早进入主链路，容易制造高置信错报。

4. 社区信号与代码证据融合难  
代码层是显式证据，社区层是高噪声证据。二者融合后如何维持可追溯和可复现，是长期技术难题。

### 2.3 工程难点

1. 多代理编排复杂度高  
并行提取、文件式记忆、回退路径、证据绑定、失败重试都要求很强的流程控制和幂等设计。

2. 外部资源稳定性不可控  
GitHub API、ClawHub、社区教程、第三方 Skill 文档的格式和可用性都不稳定。

3. 配置与模板系统会演化成第二套产品  
`SKILL.md.tmpl`、`skill_gen.py`、`skill_check.py`、`platform_rules.json` 本身也需要治理，否则会反向拖慢研发。

4. 评测体系资产仍然偏薄  
目前更像研究验证而不是产品回归体系。特别是跨领域、多场景、多许可证风险下的真实样本还不够。

### 2.4 用户视角的挑战

1. 用户并不天然想要“信息完整”，而是想要“下一步最省事”  
如果系统严格坚持不给任何优先级表达，很多用户会觉得信息丰富但决策负担太重。

2. 输出过多会提升认知负担  
一个用户请求如果同时产出 Skill、体检、热力图、领域真相、对比报告，用户很难判断先看什么。

3. “开袋即食”承诺很强  
一旦最终还需要用户花大量时间人工修改，预期落差会非常明显。

4. 交付时延与成本会直接影响可用性  
当前文档只对局部阶段有成本估算，没有定义全链路时延目标和预算上限。

## 3. 逐章节审稿清单

以下按“该删 / 该补 / 该降级”给出。

### 第一部分：产品灵魂

#### 该删

- 删掉部分过度自证型内容，尤其行业趋势背书和类比性论述，避免主文过重
- 删除或收缩会让人误以为“长期护城河已被证明”的表达

#### 该补

- 补一个明确的 `v1 一句话定义`
- 补一个 `非目标清单`
- 补“什么叫工具，什么叫建议”的产品边界

#### 该降级

- 将中长期护城河改写为“战略假设”，而非当前事实
- 将 Agent Harness、gstack 的借鉴内容降为设计依据，不要占主文过高权重

### 第二部分：用户视角

#### 该删

- 从主文删除非核心角色矩阵
- 从主文删除大量“扩展消费者”叙事

#### 该补

- 只保留 1 个主用户画像，最多保留 1 个次级用户画像
- 补核心用户的触发场景、操作路径、成功标准、失败原因
- 补“使用工具”旅程的具体流程，不要只停留在概念词

#### 该降级

- 将领域探索彻底降级为衍生能力
- 将热力图降为解释层或增强层，不要作为首发核心输出

### 第三部分：核心机制

#### 该删

- 删除或收缩研究型概念展开过多的段落，特别是通胀和错误共识中不直接影响首发实现的部分

#### 该补

- 为每个核心机制补一行“为什么用户会为此买单”
- 增加 `v1 必做能力 / v1.5+ 增强能力` 对照表
- 增加各机制的输入、输出、验收方式

#### 该降级

- 将 `领域图谱`、`知识供应链`、`错误共识`、`热力图` 统一降级为 `v1.5+`
- 保留 `STALE` 作为可提前上线的最小检查子集

### 第三部分 3.11：多项目提取到组装管线

#### 该删

- 删掉“知识选择建议”这种带处方色彩的措辞

#### 该补

- 补各 Phase 的 schema ownership
- 补失败兜底：搜不到候选、提取不稳定、门控不通过时怎么办
- 补人工确认点：哪些环节允许自动，哪些环节必须让用户确认

#### 该降级

- 将“直接输出可部署 Skill”降级为“输出候选 Skill + 自动校验 + 限制说明”
- 将社区知识采集先限定为少量高价值来源，不要一开始做开放式抓取

### 第四部分：技术架构

#### 该删

- 删掉过早定死的轮次、预算、收益分布等参数性判断

#### 该补

- 补 `事实 / 解释 / 推断` 的责任边界
- 补各 Stage 的阻断条件和回退条件
- 补平台适配规则的维护机制

#### 该降级

- `Stage 3.7` 继续保持可选增强
- OpenClaw 适配检查先只覆盖最关键的结构项，避免首发阶段规则爆炸

### 第五部分：质量保证体系

#### 该删

- 删除没有充分历史样本支撑的绝对化表述，例如“95% 的问题免费发现”

#### 该补

- 增加阻断型与提示型检查分类
- 增加人工抽检协议
- 增加 License 风险处理策略

#### 该降级

- `LLM-as-judge` 降级为辅助评测，不作为交付主裁判
- `Dark Trap Scan` 初期作为提示项，不宜过度影响主链路

### 第六部分：开发路线与实施计划

#### 该删

- 删除当前过细的乐观工期，避免造成“方案等于排期”的错觉

#### 该补

- 补 `必须做 / 可延后 / 研究继续` 三层路线图
- 补关键路径说明
- 补真实验收标准，而不是只给模块清单

#### 该降级

- 将 `领域图谱 Alpha`、`供应链 P1`、`错误共识 MVP` 后移
- 将首发目标收敛为：需求理解、作业发现、并行提取、Skill 组装、质量门控

### 第七部分：实验成果

#### 该删

- 删除模拟验证与真实验证混叙造成的成熟度错觉

#### 该补

- 增加真实用户任务样本池
- 增加统一实验模板
- 增加失败案例复盘

#### 该降级

- 将“跨领域通用性确认”改成“流程初步可迁移”
- 将模拟通过的结论降级为“结构验证”，不要直接等同产品可用

### 第八部分：研究索引

#### 该删

- 从主文删掉大段研究索引，挪到附录或独立索引文件

#### 该补

- 为每份研究增加状态标签：已采纳 / 待验证 / 已放弃

#### 该降级

- 将 chat log 类非正式材料从主证据链中降级

### 附录

#### 该删

- 删除过多影响产品表达的术语限制

#### 该补

- 补 `非目标清单`
- 补 schema ownership
- 补数据结构版本策略

#### 该降级

- 将大部分数据结构速查迁移到架构文档，而不是继续放在产品定义主文

## 4. 对 v1 范围的收敛建议

### 4.1 建议冻结的 v1 范围

**目标用户**

- 主用户：开发者

**主任务**

- 用户描述一个想做的能力
- 系统寻找可借鉴作业
- 系统提取并综合知识
- 系统输出候选 OpenClaw Skill

**主交付**

- `SKILL.md`
- `PROVENANCE.md`
- `LIMITATIONS.md`

**主能力**

- Phase A 需求理解
- Phase B 作业发现
- Phase C 并行灵魂提取
- Phase F Skill 组装
- Phase G 质量门控

### 4.2 建议后置到 v1.5+

- 领域图谱
- 知识健康检查完整体系
- 热力图
- 知识供应链 P1
- 错误共识检测
- 面向选型者、投资人、PM/CTO 的独立输出

## 5. 建议新增的文档章节

建议在下一版产品定义文档中新增以下章节：

### 5.1 v1 成功指标

至少包含：

- 首次交付时延目标
- 单次平均成本上限
- 质量门控一次通过率
- 用户采纳率
- 用户平均手改时长
- 高严重度错误率

### 5.2 失败兜底策略

至少回答：

- 找不到合格候选项目怎么办
- 候选间冲突太大怎么办
- 质量门控连续失败怎么办
- License 不兼容怎么办
- 平台适配失败怎么办

### 5.3 非目标清单

至少明确：

- 不做通用问答产品
- 不做通用代码生成助手
- 不做项目托管平台
- 不做面向所有角色的统一 BI 产品

## 6. 修订优先级建议

### P0：本周必须改

- 冻结主用户
- 冻结主交付
- 冻结 v1 范围
- 删除“开发就绪”的绝对判断，改为“核心路径待冻结后可启动开发”

### P1：启动开发前必须补

- 成功指标
- 失败兜底
- License 风险策略
- 自动与人工边界
- 关键路径与后置能力分层

### P2：开发中持续补

- 真实样本评测
- 抽检协议
- schema ownership
- 平台规则版本化

## 7. 最终结论

这份 v5 文档最强的部分是灵魂、机制和研究广度。  
这份 v5 文档最弱的部分是首发产品收敛、指标定义和风险边界。

如果继续按当前版本直接开工，最大的风险不是“做不出来”，而是：

> **会做出一个技术上很丰富、研究上很先进，但产品上不够聚焦的首发版本。**

因此，下一步最值得做的不是再加新研究，而是：

1. 冻结 `v1` 的用户与交付  
2. 砍掉不阻塞首发的研究型承诺  
3. 把“产品定义”从“知识全书”改写成“开发冻结文档”

当这三步完成后，Doramagic 才算真正进入可控的产品化阶段。
