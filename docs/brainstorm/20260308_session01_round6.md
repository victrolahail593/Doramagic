## Tangsir 的三个关键问题

### Q1: DeepWiki 为什么止步于"理解"？
**答案**：不是技术做不到，是产品定位和商业逻辑。
- Cognition 的核心产品是 Devin（AI 程序员），DeepWiki 做到"用起来"会自我蚕食
- "理解"是定义清晰的问题，"用起来"是定义模糊的问题
- 技术上 AI Agent 已经能做事，缺的是"知识→服务"的翻译层
- **这正是 AllInOne 的机会**

### Q2 & Q3: 服务器采集 + 本地AI判断的分工架构
**Tangsir 提出**：利用伦敦/新加坡服务器采集社区素材，本地 AI 做判断
**结论**：完全可行，且和 WhisperX 架构一脉相承

## 架构设计

### 分工原则
| 角色 | 职责 | 原因 |
|------|------|------|
| 伦敦/新加坡服务器 | 采集 + 粗筛（眼睛和耳朵） | 稳定网络、IP池、24/7运行、便宜 |
| 本地 AI | 深度判断 + 知识整合 + 服务交付（大脑） | 需要用户上下文、需要交叉关联 |

### 三个难题的解决方案

**质量过滤**：
- 服务器端：投票数、作者信誉、是否有具体经验、时效性 → 打分
- 本地 AI：结合用户需求做精筛，判断知识互补/矛盾

**知识过期**：
- 服务器端：订阅 GitHub Release/Commits，定期搜索新内容，增量更新
- 本地 AI：灵魂文档带版本号，新旧知识并存，优先最新

**交叉关联**：
- 在本地 AI 完成，因为需要同时访问代码知识和社区知识
- 例：SO 说"别用 calculateTax() 默认参数" + 代码显示只有 US 逻辑 → 标注地区限制

### 架构本质
采集在远端，智能在本地 = WhisperX 已验证的模式
AllInOne 是在同一基础设施上做新事情
