## 关键决策

### 1. 知识积木从"过滤器"升级为"产品"
CEO 提出："积木是否够缝合？" 这个问题改变了 Doramagic 的战略定位。
- 之前：积木只在提取时当过滤器（"这些是已知的，去找新的"），提取完就扔掉
- 之后：积木本身就是可交付的知识资产，能直接缝合成 skill
- 战略路线：工具 → 知识引擎 → 知识平台

### 2. 积木选样标准来自市场，不来自技术
CEO 纠正了我"扩 20 个流行框架"的方案。正确的逻辑是：
- Doramagic 是 skill 生产 skill 的工具
- ClawHub 13729 个 skill 的下载数据反映真实需求
- 积木类别应该对齐用户高频场景，不是对齐框架排行榜
- CEO 原话："应该根据 openclaw 的官方 skill 市场作为选择的基础"

### 3. 赛马结果：Codex 在集成任务上强于 Claude
- #1 直缝集成：Codex 改了 8 个文件完整集成 FlowController，Claude 只改了路由层。Codex 胜。
- 原因：集成任务需要理解多个模块的交互，Codex 的一次性全量分析在这里更有效。
- 教训：Claude 应该一开始就做完整集成，而不是"最小化改动"。

## 我犯的错

### 错误 1: 跳过赛马流程
Epic 5（因果推理）和 BrickStitcher（直缝引擎）都是核心功能开发，按规范必须赛马。我直接独做了，被 CEO 发现后才补做交叉审查。Codex 审查发现了 7 个真实问题（3 HIGH + 4 MEDIUM），证明自己审自己确实有盲点。

**教训**: 赛马不是形式主义，交叉审查能发现自己看不到的问题。Codex 有独立的代码审查额度，不消耗常规开发额度，没有理由跳过。

### 错误 2: 提出"验证 3 个项目"的方案
CEO 指出："验证始终是有限范围，不能代表用户的多样性。" 我提出的"先跑 3 个测试"方案解决的是正确性，不是普适性。正确的思路是扩大知识覆盖面（积木库），而不是在少数样本上反复验证。

**教训**: 区分"正确性"和"普适性"。少量样本证明"能跑通"，广覆盖才能证明"多数场景都能用"。

### 错误 3: 按"流行框架"而非"市场需求"选样
我建议"扩 20 个流行框架（Flutter、Svelte、Elixir…）"。CEO 三连问纠正了我：为什么是 20？选样标准是什么？如何向 AI skill 靠近？正确答案是从 ClawHub 市场数据反推，不是从技术排行榜正推。

**教训**: 产品决策从用户需求出发，不从技术能力出发。

### 错误 4: 发布只推代码不创建 Release
打了 v12.3.0 tag 推到 GitHub 后就宣布完成。CEO 问"GitHub 上版本号不对"时才发现：只推了 tag 没创建 Release，版本号不同步（SKILL.md/README/marketplace 还是 12.1.5），README 没更新新功能用法，开发日志没写，踩坑没记录。

**教训**: 发布是一个完整流程，不只是 `git push`。版本同步 + Release notes + 文档更新 + 日志 + 踩坑全部是流程的一部分。

### 错误 5: 设计第三步"清理 untracked 文件"
在制定修复计划时，我加了一个"清理 untracked 文件"的步骤。CEO 问"第三步和第四步的意义是什么"，我自己分析后承认第三步是伪工作——不影响任何目标，纯粹是"整理抽屉"。

**教训**: 计划的每一步都要问"删掉这步，目标会受影响吗？"如果不会，就不该存在。

## 踩过的坑

### 坑 1: Sonnet 子代理并行修改后切换分支丢改动
10 个 Sonnet 代理在 main 分支上并行修改文件。我在 Sonnet 完成前切换到 claude/ 分支做赛马，切回 main 时 Sonnet 的改动丢了一部分。
- 根因：未提交的工作区改动在 `git checkout` 时被覆盖
- 修复：补做了丢失的改动
- 规则：Sonnet 代理产出必须先 commit 或 stash 再切换分支

### 坑 2: sys.path.insert 清理留下空 if 块
Sonnet 代理删除 `sys.path.insert(0, str(X))` 时，只删了 insert 行，没删整个 if 守卫块，导致：
```python
if str(pkg_dir) not in sys.path:
    # 这行被删了

from xxx import yyy  # SyntaxError: expected an indented block
```
- 修复：批量脚本删除空 if 块 + 补回被误删的 `from pathlib import Path`
- 规则：清理 sys.path 必须删除整个代码块（if 守卫 + 路径变量 + import sys），不能只删一行

### 坑 3: Codex 速率限制截断赛马任务
第一轮启动 10 个 Sonnet 积木生成代理，全部在写完第一个文件后被速率限制截断（"You've hit your limit"）。每个代理只完成了一半任务。
- 修复：等限额重置后启动第二轮补完
- 规则：大批量并行任务需要考虑速率限制，分批启动或预估消耗

### 坑 4: publish_preflight 与 publish_to_github 逻辑不一致
预检报 `docs/ is tracked by git (should be gitignored)` 阻断发布。但 `publish_to_github.sh` 已有排除逻辑会在发布时清理内部目录。两个脚本对 docs/ 的处理标准不同。
- 修复：在发布脚本中补全 docs/designs、docs/logs、docs/pitfalls.md 的排除
- 规则：预检脚本和发布脚本的排除列表必须同步

### 坑 5: skills/doramagic/packages 副本未同步
发布时 `publish_to_github.sh` 报工作区 dirty，因为 skills/doramagic/packages/ 是手动维护的副本，没有随主包更新。
- 修复：rsync 同步所有 11 个包
- 规则：每次发布前必须同步 skill 副本（或改成 symlink）

### 坑 6: pre-commit ruff 版本落后于 .venv ruff
pre-commit 用 ruff v0.8.0，.venv 用 ruff v0.15.8。v0.8.0 不认识 RUF059 规则导致提交失败。
- 修复：升级 .pre-commit-config.yaml 到 v0.15.8
- 规则：pre-commit 和 .venv 的工具版本必须一致

### 坑 7: uv 创建的 venv 用 pip 装包找不到
mypy 用 `pip install` 安装后 `.venv/bin/python -m mypy` 报 ModuleNotFoundError。原因是 venv 由 uv 创建，pip 装的包和 uv 的包索引不一致。
- 修复：改用 `uv pip install mypy`
- 规则：uv 创建的 venv 必须用 uv pip 装包，不能混用系统 pip
