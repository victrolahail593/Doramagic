今天完成了 Doramagic 从"知识提取工具"到"个性化编译器"的产品转型设计，并建设了积木 v2 体系的基础设施。

## 时间线

### 中午：产品方向重定义

- `/office-hours` YC Startup 模式，6 个逼问
- 核心洞察："不要教用户做事，给用户工具"
- 三个前提确认：
  1. 核心价值 = 生成可用的个性化工具，不是提取设计智慧
  2. 废弃实时 GitHub 搜索，用预构建积木库替代
  3. 交付物 = live outcome，不是 SKILL.md 知识包
- Codex 第二意见："Doramagic 最强形态是个性化编译器"
- 用户修正：先重新定义知识积木体系，不急于建原型

**DeerFlow 源码深度分析**（Opus）：
- 5 个模块逐一拆解（Skills、编排、Sandbox、记忆、Telegram）
- 3 个灵魂：Markdown 即知识、渐进加载+延迟绑定、中间件链

**三个关键分歧解决**：
1. 积木 = 结构化信封 + 自然语言内容（两者都要，分层）

3. 架构从第一天支持 10000+，存储用数据库 + API

**体系建设**：
- BrickV2 Pydantic Schema
- BrickStore（SQLite + FTS5 + YAML 离线回退 + 自动导入）
- 3 个手写示例积木
- 21 个迁移积木（460 条约束 + 148 条失败模式）
- 全量集成测试通过

## 关键决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 产品定位 | 个性化编译器 | 用户需要的是工具，不是知识 |
| 实时 GitHub 搜索 | 废弃 | 不可控、延迟高、结果差 |
| 积木格式 | 结构化信封 + 自然语言内容 | 信封给系统检索，内容给 LLM 生成代码 |
| 存储方案 | SQLite + FTS5（Phase 1）→ API（Phase 2） | 离开 API 依然工作 |
| 积木规模目标 | 10000+，API 可达 100万 | CEO 决策 |
| 旧积木迁移 | 提取 failure 数据，不全量重建 | 21 个 HIGH 文件的失败模式最有价值 |
