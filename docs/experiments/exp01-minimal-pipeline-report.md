# exp01 最小灵魂提取闭环实验报告

**实验编号**：exp01  
**状态**：设计完成，待执行  
**目标阶段**：验证优先  
**关联报告**：`docs/research/2026-03-09-allinone-technical-decision-report.md`

## 1. 实验目标

验证 AllInOne 的最小主张是否成立：针对一个小型真实仓库，能否提取出足以提升 LLM 回答质量的“灵魂资产”。

## 2. 核心假设

### 2.1 假设 H1
在不改变模型本体的前提下，给模型加载结构化灵魂资产后，其回答会比无灵魂基线更具体、更可靠、更有用。

### 2.2 假设 H2
即使只做代码半魂，不引入社区半魂，也能在一组标准问题上观察到显著的回答质量提升。

### 2.3 假设 H3
使用 Repomix + 鹰眼/深潜 + Markdown 卡片化表示的最小链路，足以支撑首次验证，不需要重型 RAG / 图数据库 / 多模态扩展。

## 3. 成功标准与失败标准

### 3.1 成功标准

- 能选出一个合适的小型真实仓库作为验证对象
- 能产出至少一组基础灵魂资产（概念 / 工作流 / 规则）
- 能完成至少一轮 A/B 对比问答
- 实验记录完整，包含 token、耗时、问题集、评分、异常与人工修正
- 最终能回答“灵魂是否有用”这一问题

### 3.2 失败标准

- 无法产出任何可用灵魂卡片
- A/B 对比无法显示任何正向增益
- 灵魂上下文导致回答更混乱或更高风险
- 实验记录不完整，无法复现或复盘

## 4. 范围与非目标

### 4.1 范围

- 小型真实仓库
- 代码半魂优先
- 最小上下文打包与最小卡片产出
- 单模型或单 provider 即可
- 明确记录成本、耗时、失败点

### 4.2 非目标

- 证明 AllInOne 全部愿景
- 处理超大仓库或 monorepo
- 接入社区半魂全量采集
- 引入 MCP、DeepWiki、向量库、图数据库
- 构建正式产品界面或完整服务系统

## 5. 验证仓库选择标准

### 5.1 仓库标准

- 小于约 5K 行核心代码
- 单一职责
- 有 README 或基本文档
- 最好有少量 Issue / Discussion 可供后续扩展
- 你对其问题域有基本理解，便于人工评估输出质量

### 5.2 当前选择状态

当前已完成首轮候选筛选，详见：`docs/experiments/exp01-repo-selection-guide.md`

- **Primary**：`python-dotenv/python-dotenv`
- **Secondary**：`jpadilla/pyjwt`、`pallets/itsdangerous`
- **Stretch**：`psf/requests`
- **排除项**：WhisperX 全仓等过大样本不作为首个验证对象

### 5.3 当前默认建议

`exp01` 当前默认首个验证仓库为 **`python-dotenv/python-dotenv`**。其职责单一、规则与工作流足够丰富、文档清晰、社区信号适中，最适合做第一次价值验证。

## 6. 实验输入与输出

### 6.1 输入

- 目标仓库路径 / URL
- Repomix 打包结果
- 鹰眼阶段 prompt / 规则
- 深潜阶段 prompt / 规则
- 问题集模板
- Judge Rubric 模板
- 仓库评分卡模板（如需复核候选）

### 6.2 输出

- `repo_profile`
- 最小灵魂卡片集
- A/B 问答结果
- 评估记录
- 运行日志
- 实验结论

## 7. 最小实验流程

### Step 1：仓库画像

目标：确认样本是否适合做首个验证实验。

输出：仓库定位、规模、结构质量、是否适合最小实验。

### Step 2：代码上下文打包

目标：生成适合鹰眼阶段消费的结构化上下文。

优先方式：Repomix。

### Step 3：鹰眼阶段

目标：识别项目身份、核心概念、关键模块、深潜优先区域。

输出：候选概念和候选深潜区。

### Step 4：深潜阶段

目标：围绕少量高价值区域提取卡片。

建议优先卡片类型：
- `concept_card`
- `workflow_card`
- `decision_rule_card`

### Step 5：灵魂资产整理

目标：把提取结果整理成稳定、可复用、可装配的 Markdown 资产。

### Step 6：上下文装配

目标：按问题类型将最小灵魂资产装入 prompt。

### Step 7：A/B 问答对比

对照组：无灵魂基线  
实验组：加载灵魂资产

### Step 8：记录与复盘

记录 token、耗时、成本、人工修正、失败原因和结论。

## 8. 问题集设计

问题集应覆盖至少五类：

1. 领域概念理解
2. 工作流理解
3. 规则 / 边界条件
4. 风险 / 坑点识别
5. 方案建议 / 迁移建议

详细模板见：`docs/experiments/templates/exp01-question-set-template.md`

## 9. A/B 评估方法

### 9.1 对照方式

- **A 组**：不加载灵魂资产
- **B 组**：加载灵魂资产

### 9.2 评估维度

- Groundedness
- Specificity
- Usefulness
- Risk
- Uncertainty Handling

详细标准见：`docs/experiments/templates/exp01-judge-rubric.md`

## 10. 记录字段

每次实验至少记录：

- 仓库信息
- 模型信息
- 上下文模式
- 生成的卡片数量与类型
- token / latency / estimated cost
- 问题集版本
- Judge 结果
- 人工复核结论
- 异常与人工修正点

详细模板见：`docs/experiments/templates/exp01-run-log-template.md`

## 11. 风险与偏差控制

### 11.1 样本选择偏差

过熟悉或过简单的仓库可能让实验结果失真。

### 11.2 Judge 偏差

评估时需避免“因为回答更长所以误判为更好”。

### 11.3 Prompt 偏差

如果实验组 prompt 远比对照组复杂，则结果不可比较。

### 11.4 人工修正污染

必须区分模型原始产出和人工后修正结果。
