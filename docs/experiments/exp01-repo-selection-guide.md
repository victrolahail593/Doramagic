# exp01 候选验证仓库筛选指南

**用途**：为 AllInOne 的首个最小闭环实验选择合适的验证仓库  
**适用实验**：`exp01` 及其后续类似的早期验证实验

---

## 1. 为什么要先做仓库筛选

`exp01` 的目标不是证明 AllInOne 的全部愿景，而是验证一个更窄、更关键的命题：

> 对一个小型真实仓库进行灵魂提取后，模型回答是否比无灵魂基线更好。

如果首个样本选错，实验失败原因会变得不可归因：

- 可能是 AllInOne 方法没价值；
- 也可能只是仓库太大、太乱、太冷门、太偏门或太不适合人工评估。

因此，首个验证仓库必须优先满足“**可验证**”，而不是“**最有代表性**”。

## 2. 硬过滤标准（必须满足）

候选仓库必须同时满足以下条件：

1. **单一职责**：仓库目标清晰，不是多产品拼装或 monorepo
2. **小型或偏小型**：优先小于约 5K 行核心代码；若未量化 LOC，则至少从 GitHub 仓库表面积看应明显小于大型框架/平台
3. **有 README / docs**：可帮助形成项目身份、目标用户和使用方式
4. **结构清晰**：源码目录、测试目录、文档目录边界明显
5. **有测试或示例**：便于理解真实用法和规则边界
6. **没有重型基础设施依赖**：不要求数据库、消息队列、微服务部署才能理解核心逻辑

## 3. 软评分标准（用于排序）

每个候选仓库按以下维度评分，建议 1–5 分：

| 维度 | 说明 |
|------|------|
| 范围清晰度 | 项目边界是否一眼看清 |
| 代码表面积 | 是否适合作为首个最小样本 |
| 文档质量 | README / docs 是否足够支撑项目身份判断 |
| Issue 信号密度 | 是否有少量但有价值的问题讨论可供后续扩展 |
| 规则密度 | 是否包含明确规则、边界条件、验证逻辑 |
| 工作流密度 | 是否存在清晰的流程或调用顺序 |
| 人工可评估性 | 人类是否容易判断回答好坏 |
| 提取友好度 | 是否适合 Repomix + 鹰眼/深潜链路 |
| 隐藏复杂度风险 | 是否可能表面简单、实际暗坑很多 |

## 4. 不推荐作为 exp01 首样本的仓库类型

- monorepo / 多子系统仓库
- 明显超过目标规模的大型框架
- 缺少 README 或结构极差的仓库
- 依赖大量运行时环境才能理解的系统
- 过于 toy 的 demo 仓库
- 领域过度陌生、人工很难判断回答质量的仓库

## 5. 当前候选池（2026-03-09）

> 说明：以下候选基于 GitHub 官方仓库页面快速校验得出。关于“规模偏小/偏大”的判断属于**工程推断**，不是已测 LOC；正式选定前应做一次本地快速拉取或 Repomix 打包校验。

### 5.1 Primary：`python-dotenv/python-dotenv`（推荐）

**推荐理由：**

- 单一职责非常清晰：从 `.env` 文件加载环境变量
- README 和 docs 完整，项目身份容易判断
- 具备明显的规则密度：加载顺序、变量覆盖、变量展开、文件发现逻辑等
- 有源码、测试、CLI、文档，适合同时提取概念卡、工作流卡和规则卡
- GitHub 官方页面显示其维护成熟，且有足够数量的 Issues，适合作为阶段二扩展到社区半魂

**适合作为首样本的原因：**

它既不至于太玩具，也不会像大型框架那样在第一次实验里掺入过多噪音。

**官方来源：**
- 仓库：https://github.com/theskumar/python-dotenv
- 文档：README / `docs/` / `src/dotenv` / `tests`

### 5.2 Secondary：`jpadilla/pyjwt`

**推荐理由：**

- 单一职责明确：JWT 编解码与校验
- 规则密度高：过期、签名、算法、声明字段等都天然适合作为规则卡
- 文档和测试相对完备
- 有一定社区信号，便于后续观察 WHY 类知识

**不作为首选的原因：**

安全与加密语义会引入更高的领域复杂度，首次实验可能把“规则密度高”与“人工评估门槛高”绑定在一起。

**官方来源：**
- 仓库：https://github.com/jpadilla/pyjwt
- 文档：README / `docs/` / `jwt/` / `tests/`

### 5.3 Secondary：`pallets/itsdangerous`

**推荐理由：**

- 体量小、结构干净
- 单一职责明确：签名与序列化安全工具
- 很适合验证“从小而清晰的代码中提取规则与概念”

**不作为首选的原因：**

GitHub 页面显示其当前公开 Issue 信号较弱，这会削弱后续从代码半魂平滑扩展到社区半魂的价值。

**官方来源：**
- 仓库：https://github.com/pallets/itsdangerous
- 文档：README / `docs/` / `src/itsdangerous` / `tests/`

### 5.4 Stretch：`psf/requests`

**推荐理由：**

- 社区信号极强，文档与 Issue 非常丰富
- 概念、工作流和坑点都很典型
- 如果要验证“社区半魂”的增益，它会是很好的后续样本

**不作为首样本的原因：**

从仓库表面积和生态复杂度看，它更像一个“第二阶段样本”，不适合首个最小闭环实验。否则实验一开始就会把范围、规模和社区复杂度同时拉高。

**官方来源：**
- 仓库：https://github.com/psf/requests

## 6. 当前推荐结论

### 6.1 默认首选

- **首选仓库**：`python-dotenv/python-dotenv`

### 6.2 备选仓库

- `jpadilla/pyjwt`
- `pallets/itsdangerous`

### 6.3 延后样本

- `psf/requests`

## 7. 正式定稿前的最后一步

在真正开始 `exp01` 之前，建议再做一次轻量校验：

1. 本地拉取候选仓库
2. 用 Repomix 跑一次压缩上下文打包
3. 记录 token 规模与源码结构
4. 如果主候选明显超出预期，再切换到备选仓库

## 8. 对 exp01 的直接影响

基于本指南，`exp01` 的当前默认路线应当是：

- **验证仓库默认选 `python-dotenv`**
- 先只做代码半魂
- 等首次 A/B 成功后，再考虑在同一仓库或下一仓库上引入社区半魂
