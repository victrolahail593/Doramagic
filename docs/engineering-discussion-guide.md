# Doramagic 工程实施方案 — 讨论清单与节奏安排

> **目的**: 在正式开发前，确保所有关键决策经过你的确认
> **原则**: 不追求一次讨论完所有问题。按"现在必须定 → 开工后再定 → 以后再定"分层
> **形式**: 每次讨论聚焦 1-2 个主题，你提问/我回答/你做决策

---

## 讨论全景图

一共 **4 轮讨论**，每轮 1 个主题，预计每轮 15-30 分钟。

```
第 1 轮：开工前必须确认的事（方向 + 第一步）
第 2 轮：Phase 1 的具体做法（Agentic 提取怎么做）
第 3 轮：测试和验证策略（怎么知道做对了）
第 4 轮：后续 Phase 的节奏（Phase 2-7 怎么安排）
```

前 2 轮讨论完就可以开始写代码。第 3-4 轮可以边开发边讨论。

## 第 1 轮：方向确认（开工前）

**一句话主题**: 确认"做什么"和"从哪里开始"。

### 需要你决策的问题

**Q1：是否同意从 Phase 1（Agentic 提取）开始？**

工程方案建议 Phase 1 是价值最高的起点。但你可能有不同想法：
- 也许你觉得跨项目对比（Phase 2）对用户更有吸引力？
- 也许你觉得应该先修复现有问题，而不是加新功能？
- 也许你有完全不同的优先级？

这个问题决定了接下来 1-2 周做什么。

**Q2：代码要不要推到 GitHub？**

好处：防灾备份 + 未来协作基础。
坏处：需要花 10 分钟设置。
风险：零（私有仓库，只有你能看到）。

你觉得现在做还是以后做？

**Q3：开发节奏怎么安排？**

方案 A：全力冲刺——每天 8 小时以上，6 周做完
方案 B：稳定节奏——每天 4-6 小时，9 周做完
方案 C：灵活安排——没有固定时间，按心情推进

你的时间和精力安排是怎样的？这决定了我在每次讨论中给你多大的信息量。

**Q4：你希望在开发过程中参与到什么程度？**

方案 A："我审核最终结果就行"——你看产出，不看过程
方案 B："关键决策找我，日常不用"——遇到选择题才来找你
方案 C："我想全程参与"——每天同步进度，我做什么你都知道

### 不需要你决策的（我来定）

- Python 项目的具体配置文件怎么写
- 文件夹内部的细节结构
- 用哪个 Python 测试框架

## 第 2 轮：Phase 1 具体做法

**一句话主题**: 确认 Agentic 提取的具体实现方式。

### 需要你决策的问题

**Q5：Agent 探索时间的上限是多少？**

现在一次提取大约 3-5 分钟。加了 Agent 后，预计变成 10-20 分钟。

问题是：你能接受用户等多久？
- 如果上限是 10 分钟，Agent 只能做 3-5 轮浅探索
- 如果上限是 30 分钟，Agent 可以做 10+ 轮深探索，质量更高

这是质量和体验之间的权衡。你怎么看？

**Q6：弱模型（比如 MiniMax）要不要支持 Agent 探索？**

方案建议：弱模型跳过 Agent 探索，用增强版的旧方法。因为 exp07 的教训是——弱模型的"帮助"可能比"不帮"更糟。

但你可能觉得：所有模型都应该有同样的功能，否则用户体验不一致。

你的立场是什么？

**Q7：提取过程要不要给用户看进度？**

方案 A：只显示"提取中，请稍候"
方案 B：显示简化进度（"正在探索第 3 轮..."）
方案 C：显示详细过程（"发现假说：项目可能有自研路由引擎"）

Gemini 研究提出了"剧场效应"——过程透明度本身就是产品价值。但这需要额外开发工作。

这个是 Phase 1 就做，还是以后做？

**Q8：Phase 1 做完后，API 调用费用的预算是多少？**

预估每次提取费用：
- 小项目：约 $0.11（约 ¥0.8）
- 大项目：约 $0.31（约 ¥2.2）

你的月度预算上限是多少？这影响测试的规模。

### 不需要你决策的

- Agent 的 5 个工具具体怎么实现
- 中间文件的 JSON 格式怎么定义
- 假说生成的 prompt 怎么写（我来迭代优化）

## 第 3 轮：测试和验证（可以边开发边讨论）

**一句话主题**: 确认"怎么知道做对了"。

### 需要你决策的问题

**Q9：你会亲自审核提取结果吗？**

A/B 对照测试由 AI 自动评分，但最终判断"好不好用"还是得人来看。
- 你是否愿意花时间读几份提取报告，给出你的主观判断？
- 还是完全依赖 AI 评分？

**Q10：如果 Phase 1 的质量提升没达到预期（比如只提升了 0.2 而不是 0.5），怎么办？**

方案 A：继续优化 Phase 1，直到达标再进入 Phase 2
方案 B：接受现有提升，先推进 Phase 2，回头再优化
方案 C：根据具体原因决定（假说质量问题 vs 工具设计问题 vs 模型能力限制）

**Q11：wger（大项目测试）的结果由谁来判断"好不好"？**

python-dotenv 有历史基线数据可以对比。但 wger 只有 exp08 的一次提取结果。
- 需要你找一个了解 wger 项目的人来审核吗？
- 还是我们自己对比判断"提取结果是否合理"？

### 不需要你决策的

- 具体用什么评分维度（已有 4 维 Rubric）
- 测试脚本怎么跑
- A/B 对照的统计方法

## 第 4 轮：后续节奏（Phase 1 完成后讨论）

**一句话主题**: Phase 2-7 怎么安排。

### 需要你决策的问题

**Q12：Phase 2（跨项目对比）和 Phase 3（领域图谱）的优先级**

Phase 2 是基础设施（指纹+对比），Phase 3 是第一个用户可感知的新功能（领域真相）。
- 先打基础还是先出功能？
- 你觉得用户更期待哪个？

**Q13：领域图谱的第一个领域是 SEO 还是其他？**

你之前说过"不一定是 SEO/GEO，需要多领域横向比较"。
- Phase 3 开始前需要确认这个决策
- 是否需要先对 2-3 个候选领域做小规模试点？

**Q14：什么时候开始考虑商业化？**

工程方案有意不涉及定价和商业模型。但在某个时间点你需要决定：
- 免费用到什么时候？
- 哪些功能是付费的？
- 是否需要建用户系统？

这不急，但想听听你的初步想法。

**Q15：什么时候做 V3 产品定义的 5 条反馈修订？**

Session 27 的 5 条反馈 + Session 29 的 3 条反馈（领域探索降级/输出多样化/社区信息），V5 手册部分采纳了但还有一些未合并。什么时候做一次统一修订？

### 不需要你决策的

- Phase 2-7 的具体技术实现
- 各 Phase 的内部任务排序
- 数据结构的具体字段

## 讨论节奏建议

| 轮次 | 主题 | 什么时候讨论 | 预计时长 |
|------|------|------------|---------|
| **第 1 轮** | 方向 + 第一步 | **现在就可以开始** | 15-20 分钟 |
| **第 2 轮** | Phase 1 具体做法 | 第 1 轮结束后 | 20-30 分钟 |
| **第 3 轮** | 测试验证策略 | Phase 1 开发到一半时 | 15 分钟 |
| **第 4 轮** | 后续 Phase 节奏 | Phase 1 完成后 | 20-30 分钟 |

**第 1 轮和第 2 轮讨论完就可以正式开工。**
