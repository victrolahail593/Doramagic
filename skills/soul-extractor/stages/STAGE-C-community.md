## 完整示例（python-dotenv）

```markdown
# 社区智慧：python-dotenv

## 社区集体认知
python-dotenv 的社区在多年使用中形成了一个共识：这个库的所有"怪异行为"都源于同一个未明说的原则——它模拟的是 shell 的 `export` 命令行为，而不是配置文件库的行为。明白了这一点，一切都变得可预测。

但这个原则从来没有在文档首页写出来。新用户第一次遇到"为什么 .env 里的值没有覆盖已有的环境变量"时，往往以为是 bug。花几小时排查，最终在某个 Issue 的第 15 条评论里找到了答案。这个过程在社区里重复了数百次。

还有一个集体认知：这个库不适合复杂场景。它的甜点是"本地开发的配置注入"，不是"生产环境的配置管理"。社区里有大量"我能用它做 X 吗"的问题，答案几乎都是"可以，但你需要另一个工具"。

## 反复出现的痛点

### 痛点一：CI/CD 里的幽灵覆盖
**现象**：开发者在 CI/CD 里设了环境变量（如 `DATABASE_URL=production_db`），
然后在测试代码里调用 `load_dotenv()`。本地测试没问题，CI 里的行为和本地不一致。
这个问题在 Stack Overflow 上有 200+ 个相似提问。
**根因**：`override=False` 是默认值，让已有的环境变量优先。
这是"shell export 哲学"的直接结果——shell 也不覆盖已有变量。
但 CI/CD 用户的直觉是"代码里调用了 load_dotenv，所以 .env 里的值会被加载"，
忽略了"除非环境里没有这个变量"这个前提。
**社区摸索出的解法**：在需要 .env 优先的场合明确传 `override=True`；
在 CI/CD 中检查是否有多余的环境变量与 .env 冲突。
**证据**：Issue #448（29 👍，是 python-dotenv 社区反应最多的 issue）

### 痛点二：多行值的静默截断
**现象**：用户把证书、私钥、JSON 粘贴进 .env，加载后值不完整，程序报神秘错误。
这个问题在 Issues 里至少有 8 个独立的报告。
**根因**：不加引号的多行值，parser 只读第一行。这是 shell 变量赋值的标准行为——
`export KEY=some value with space` 在 shell 里也只取第一个词。
但用户预期 .env 是个"配置文件"，应该能存多行。
这是"模拟 shell"和"配置文件"两种心智模型之间的冲突。
**社区摸索出的解法**：多行值用双引号包裹，内部换行写 `\n`；
或者把证书等大块内容存在单独文件，.env 里只存文件路径。
**证据**：Issue #82（21 条评论）、SIG-003（CHANGELOG v0.19.0 改进了多行解析）

### 痛点三：pip install dotenv 装错包
**现象**：用户运行 `pip install dotenv`，安装成功，但 `from dotenv import load_dotenv` 报 ImportError。
Stack Overflow 上这个问题有 600+ 赞的答案。
**根因**：PyPI 上 `dotenv` 是另一个废弃的第三方包。正确的包名是 `python-dotenv`。
这是命名空间问题，不是代码问题。
**社区摸索出的解法**：`pip install python-dotenv`（记住有"python-"前缀）
**证据**：Issue #6（34 条评论，9 👍，是项目最早的 issue 之一）、SIG-001

## 社区验证的最佳实践
- **项目根目录放 .env.example**：提交到 git，包含所有需要的 key（值留空或填示例值），帮助新成员快速上手。这是社区公认的最佳实践，官方文档很晚才提到。
- **CI/CD 里不用 load_dotenv()**：在 CI/CD 里直接用环境变量，不依赖 .env 文件。社区认为 .env 文件是本地开发工具，不应该进入 CI/CD 流程。
- **find_dotenv() + load_dotenv() 组合使用**：在子目录里运行脚本时，用 `load_dotenv(find_dotenv())` 确保找到项目根目录的 .env，而不是当前目录。

## 演化故事
python-dotenv 在早期版本中有很多解析行为不一致的地方（引号处理、多行值、转义字符），
这导致了大量社区 bug 报告。v0.19.0 是一个重要的分水岭——这个版本重写了 parser，
统一了解析规则。这是社区压力的直接结果：数年间积累的 bug 报告终于推动了根本性的修复。

同时，`find_dotenv()` 函数和 `.env.example` 的推荐模式都是响应社区真实需求加入的，
而不是最初设计就有的。这说明 python-dotenv 是一个真正被社区驱动演化的项目——
它的功能边界是由用户的实际痛点画出来的。
```
