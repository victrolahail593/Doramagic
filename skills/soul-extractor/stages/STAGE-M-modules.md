## 完整示例（python-dotenv）

```markdown
# 模块地图：python-dotenv

*python-dotenv 按职责严格分层：Parser 只解析文本，Main 只协调流程，Variables 只做替换。
每个模块知道自己的边界，不越界处理别人的事——这正是"模拟 shell export"哲学的结构体现：简单、可预测、无副作用。*

## 模块列表

### M-001：Parser（解析器）
**职责**：把 .env 文件文本解析成 (key, value) 键值对序列
**关键文件**：
- `src/dotenv/parser.py`：正则解析引擎，处理引号、注释、转义、多行值
**依赖**：无依赖（基础模块）
**被依赖**：Main（DotEnv 类调用 parse()）
**对外接口**：`parse(stream)` — 接受文件流，yield `Binding(key, value, error)` 元组

### M-002：Main（协调器）
**职责**：串联 Parser + Variables，向用户暴露 load_dotenv/dotenv_values/find_dotenv 等 API
**关键文件**：
- `src/dotenv/main.py`：DotEnv 类（核心协调逻辑）+ 所有公开函数
**依赖**：Parser、Variables
**被依赖**：CLI、用户代码
**对外接口**：`load_dotenv()`, `dotenv_values()`, `find_dotenv()`, `set_key()`, `get_key()`

### M-003：Variables（变量替换）
**职责**：处理 `${VAR}` 引用，递归展开变量，防止循环引用
**关键文件**：
- `src/dotenv/variables.py`：解析变量占位符，递归查找已知变量字典
**依赖**：无依赖（基础模块）
**被依赖**：Main（协调时调用 resolve()）
**对外接口**：`resolve(value, known_vars)` — 返回替换后的字符串

### M-004：CLI
**职责**：把 Main 层的 API 包装为命令行接口 `dotenv get/set/run/...`
**关键文件**：
- `src/dotenv/cli.py`：Click 框架定义的命令组和子命令
**依赖**：Main
**被依赖**：用户直接从终端调用
**对外接口**：`dotenv` CLI 命令

## 架构模式
严格的单向分层依赖：Parser/Variables（无依赖）→ Main（协调）→ CLI（对外）。
这种设计保证了每层可以独立测试，也让"只想用 API 不想要 CLI"的用户不需要带入多余依赖。
这与"模拟 shell export"的哲学一致——简单、无副作用、可组合。

## 横切关注点
- **错误处理**：统一在 Main 层捕获 Parser 的解析错误，CLI 层再包一层用户友好的提示
- **文件查找**（find_dotenv）：实现在 Main 层，属于协调职责而非 Parser 职责
- **类型标注**：全项目使用 typing，定义在各自模块内，无集中的 types.py
```

## 约束
- 模块数量：3-7 个。太少说明分析粒度不够，太多说明过度细分
