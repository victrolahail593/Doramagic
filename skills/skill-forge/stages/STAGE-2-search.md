# Stage 2: 开源发现

## 你的角色

你是一位开源生态专家。根据 Stage 1 的需求档案，找出最相关的开源项目——不一定是单个项目，可以是**项目组合**（比如 LangChain + ChromaDB + FastAPI 的组合）。

## 输入

读取 `<output>/discovery/needs-profile.md`，重点关注：
- 核心问题
- 关键词提炼
- 复杂度偏好

## 搜索策略

**第一步：从训练知识出发**

根据需求关键词，从你的知识库中召回最相关的开源项目。考虑：
- 直接解决该问题的项目
- 提供核心基础设施的项目
- 在该领域有最佳设计的项目（即使不是最流行的）
- 相关项目的**组合**（有时组合比单个项目更合适）

**第二步：网络搜索验证（如果有搜索能力）**

用以下查询验证和补充：
- `"<关键词> open source <年份>" site:github.com`
- `"<关键词> best practices" site:reddit.com OR site:hackernews.com`
- `"<关键词> alternatives"`

**第三步：评估与筛选**

对每个候选项目评估：

| 评估维度 | 说明 |
|---------|------|
| 相关性 | 解决的问题与用户需求的重叠程度 |
| 设计质量 | 架构是否清晰，有没有值得提取的设计哲学 |
| 社区活跃度 | Issues/PR/Discussions 是否丰富（意味着社区智慧丰富） |
| 可提取性 | 代码结构是否适合灵魂提取 |

## 产出

将发现结果写入 `<output>/discovery/projects.md`：

```markdown
# 开源发现报告

## 需求关键词
[从 needs-profile.md 提取]

## 推荐方案

### 方案一：[项目名或组合名]
**核心项目**：
- [项目名] — [GitHub URL] — [一句话说明为什么选它]
- [项目名] — [GitHub URL] — [一句话说明]

**为什么推荐**：[2-3 句话说明这个方案如何契合用户需求]

**可提取的核心知识**：
- 设计哲学：[这个项目解决问题的核心思路]
- 关键模式：[它用了什么技术/设计模式]
- 社区踩坑：[已知的常见问题]

**提取优先级**：🔴 高 / 🟡 中 / 🟢 低

---

### 方案二：[备选方案]
[同上格式]

---

## 提取计划

**主要提取目标**（Stage 3 将重点提取）：
1. [项目名] — 原因：[为什么最值得深度提取]
2. [项目名] — 原因：[...]

**轻量合成目标**（LLM 知识合成，不需要下载代码）：
- [项目名] — [已知的核心知识要点]
```

## 选择原则

- 优先选**设计清晰、有明确哲学**的项目（比盲目选最流行的好）
- 如果用户需求是"组合"问题，推荐**2-3 个项目的最佳组合**，而不是单个万能项目
- 数量控制：**3-5 个项目**，精不在多
- 至少有 1 个"主力项目"（最值得深度提取灵魂）

## ⛔ Hard Gate

- 必须写入 `<output>/discovery/projects.md`
- 每个推荐项目必须有 GitHub URL（如果确实不知道，注明"URL 待验证"）
- 必须有"提取计划"部分，明确区分"深度提取"和"轻量合成"

## 完成后

告诉用户："找到了 N 个相关开源项目，推荐提取 [主力项目名]。接下来提取它们的设计灵魂..."

**下一步：执行 Stage 3 — 灵魂提取。**
