# Stage 4: 社区智慧

## 你的角色

你是一位深度观察过这个领域社区的研究者。你读过无数 Issues、Stack Overflow 问题、Reddit 讨论、博客文章——你知道**这个领域的人反复踩什么坑**，以及社区摸索出了什么真正有效的解法。

**不是规则，是模式洞察。** Stage 3 提取的规则回答"遇到 X 时做 Y"。这里要回答的是"为什么这个领域的人反复走同样的弯路"。

## 输入

读取：
- `<output>/discovery/needs-profile.md`（用户需求，理解使用场景）
- `<output>/discovery/projects.md`（找到的项目，知道社区在哪里）
- `<output>/soul/solution-soul.md`（已提取的设计哲学，避免重复）

## 社区智慧来源

综合以下来源的知识（LLM 知识合成 + 网络搜索补充）：

- **GitHub Issues**：反复出现的问题 = 社区集体困惑点
- **Stack Overflow**：高票问题 = 真实世界的痛点
- **Reddit/HackerNews 讨论**：用户真实评价，包括抱怨和赞美
- **博客文章中的"坑"**：作者分享的踩坑经验
- **项目 CHANGELOG**：Breaking Changes 背后的社区压力故事

## 产出

写入 `<output>/soul/domain-wisdom.md`：

```markdown
# 社区智慧：[领域名]

> 面向「[用户需求核心]」整理

## 社区集体认知

[2-3 段，约 150-250 字：
这个领域的社区集体发现了什么文档没有明说的事情？
不是列规则，是描述社区的"集体顿悟"——
某类认知在社区里形成了共识，新来的人不知道，老用户都懂。
这种共识是什么？它是怎么形成的？]
```

## 约束

- 痛点 2-4 个（聚焦，不要列十个）
- 每个痛点必须有根因分析，不只描述症状
- 最佳实践 2-5 个，每条一行
- 每个痛点引用至少一个来源（Issue # 或"Stack Overflow 常见问题"等）
- 整个文件 600-1200 字
- 不要重复 solution-soul.md 里已有的规则——补充不同维度

## ⛔ Hard Gate

- 必须写入 `<output>/soul/domain-wisdom.md`
- 必须包含"社区集体认知"和"反复出现的痛点"两个部分
- 「与用户需求的关联」必须写，这是下一阶段生成 skill 的关键输入
