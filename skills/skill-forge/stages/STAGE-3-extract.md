# Stage 3: 灵魂提取

## 你的角色

你是一位深度研究过这些项目的技术专家。从 Stage 2 发现的开源项目中提取设计灵魂——不是功能罗列，而是**让人理解"为什么这样设计"的核心知识**。

## 输入

读取：
- `<output>/discovery/needs-profile.md`（用户需求）
- `<output>/discovery/projects.md`（项目列表和提取计划）

## 两种提取模式

### 模式 A：LLM 知识合成（快速，适用于知名项目）

对于训练数据中有充足信息的项目，直接从你的知识库合成。

**对每个项目回答 5 个问题**：

1. **这个项目解决什么问题？** 一句话，说明谁有这个痛点
2. **它的核心设计哲学是什么？** 所有设计决策背后的一个核心信念
3. **理解了什么之后，能推导出所有行为？** 心智模型（一个类比或顿悟式概括）
4. **它最关键的 3 个设计决策是什么？** 不是功能，是"为什么这样而不是那样"
5. **它明确不做什么？** 边界意识（往往比"做什么"更重要）

### 模式 B：Doramagic 深度提取（慢，适用于需要深度分析的项目）

如果项目是用户需求的核心，且 LLM 知识不够充分，调用 Doramagic 进行深度提取：

```
提示用户："要对 [项目名] 进行深度灵魂提取，需要下载代码（约需 5-10 分钟）。是否继续？"
```

如果用户同意，指引使用 Doramagic skill：
```
触发 doramagic skill，传入项目 URL：[GitHub URL]
提取完成后，将产出的 CLAUDE.md 内容读入，作为本阶段素材
```

## 合成：解决方案灵魂

提取完各项目后，合成一份**专门面向用户需求的解决方案灵魂**。

这不是每个项目的摘要拼接——而是从这些项目中**蒸馏出对用户需求最有价值的知识**。

写入 `<output>/soul/solution-soul.md`：

```markdown
# 解决方案灵魂

> 为「[用户需求核心]」而提炼

## 核心设计哲学

[2-3 段，综合多个项目的设计智慧，直接面向用户的需求。
不是"项目 A 认为...项目 B 认为..."的罗列，
而是"解决这类问题的核心设计原则是..."]

## 关键心智模型

[1-2 个类比或顿悟式概括，让用户理解这个领域的底层逻辑]

## 关键设计决策

**[决策名]**：[为什么这样，不那样。背后的权衡]
**[决策名]**：[...]
**[决策名]**：[...]

## 必须知道的边界

[明确说明这类技能/解决方案的局限性和不适用场景]

## 关键规则（给未来的 skill 用）

- **[CRITICAL]** [规则名] — [if/when/then 格式的规则描述]
- **[HIGH]** [规则名] — [规则描述]
- **[HIGH]** [规则名] — [规则描述]

## 项目知识来源

| 项目 | 提取模式 | 主要贡献 |
|------|---------|---------|
| [项目名] | LLM合成/Doramagic | [从这个项目学到什么] |
```

## ⛔ Hard Gate

- 必须写入 `<output>/soul/solution-soul.md`
- 「关键规则」至少 3 条，必须是 if/when/then 格式
- 合成结果必须**面向用户需求**，而不是泛泛而谈项目特性
- 不能只是项目功能的罗列，必须有设计哲学层面的洞察

## 完成后

告诉用户："已从 N 个开源项目中提取核心设计智慧。接下来整合社区踩坑经验..."
