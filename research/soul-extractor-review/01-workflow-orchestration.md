# 流程编排评审报告

研究日期: 2026-03-10 | 模型: glm-5

## 现状分析

### 整体架构

Soul Extractor 采用 **6 阶段顺序流水线设计**：

```
Stage 0 (准备) → Stage 1 (灵魂发现) → Stage 2 (概念+工作流) → Stage 3 (规则卡) → Stage 3.5 (验证) → Stage 4 (合成) → Stage 5 (组装)
```

### 各阶段职责

| 阶段 | 输入 | 输出 | 执行方式 |
|------|------|------|----------|
| Stage 0 | repo_url | packed_compressed.xml, community_signals.md | 脚本 (`prepare-repo.sh`) |
| Stage 1 | packed_compressed.xml | 00-soul.md (7问答) | LLM 执行指令 |
| Stage 2 | 代码 + 灵魂 | CC-*.md (概念卡), WF-*.md (工作流卡) | LLM 执行指令 |
| Stage 3 | 代码 + 灵魂 + 社区信号 | DR-*.md (规则卡 + 社区陷阱卡) | LLM 执行指令 |
| Stage 3.5 | 所有卡片文件 | validation_report.json, validation_summary.md | Python 脚本硬校验 + LLM 软审查 |
| Stage 4 | 所有中间产物 | expert_narrative.md | LLM 执行指令 |
| Stage 5 | expert_narrative.md | .cursorrules, CLAUDE.md, project_soul.md | 脚本 (`assemble-output.sh`) |

### 硬门控机制

每个阶段都设置了 **Hard Gate**：
- Stage 1: 必须完成 7 个问答
- Stage 2: 必须产出 3 张概念卡 + 3 张工作流卡
- Stage 3: 必须产出至少 5 张代码规则 + 3 张社区陷阱卡
- Stage 3.5: 硬校验必须 PASS（Python 脚本检查格式合规性）
- Stage 4: 必须产出专家叙事，内容需符合结构约束

## 问题发现

### 1. 阶段依赖关系与并行化机会

**当前设计问题**：
- Stage 2 和 Stage 3 存在 **假顺序依赖**
- 两者都依赖：`代码` + `00-soul.md`
- 两者产出互不依赖
- 这两个阶段可以 **并行执行**

**并行化收益估算**：
- 假设 Stage 2 耗时 T2，Stage 3 耗时 T3
- 当前总耗时：T2 + T3
- 并行后：max(T2, T3)
- 理论节省：min(T2, T3)

### 2. 中断续跑机制缺失

**严重问题**：当前设计 **没有 checkpoint 机制**

当 LLM 执行 Stage 1-4 过程中发生：
- Token 额度耗尽
- 网络中断
- 模型服务不稳定
- 用户主动终止

**后果**：
- 所有已完成的阶段产出丢失
- 需要从 Stage 0 重新开始
- 浪费 API 调用和时间

**缺失的关键功能**：
- 无进度持久化（如 `.progress.json`）
- 无阶段完成标记
- 无断点恢复命令

### 3. 失败恢复策略不完善

**硬门控设计优点**：
- 强制每个阶段完成质量检查
- Stage 3.5 有 Python 脚本验证

**问题点**：

| 场景 | 当前处理 | 问题 |
|------|----------|------|
| Stage 3.5 硬校验失败 | 脚本返回 exit 1 | LLM 需要手动解读错误、修复卡片、重新运行 |
| 软审查发现 fail | 要求修复后继续 | 无自动修复循环 |
| 脚本执行失败 | 无明确错误处理 | 用户需要手动排查 |
| LLM 输出不合规 | Hard Gate 拦截 | 但无引导式修复建议 |

**具体问题**：
- `prepare-repo.sh` 失败时无清理回滚
- `validate_extraction.py` 报告错误后，LLM 需要手动解析 JSON 找问题
- 无"修复-重验"自动化循环

### 4. 验证阶段（Stage 3.5）的位置和逻辑问题

**位置合理性**：
- ✅ 放在 Stage 3 之后、Stage 4 之前是正确的
- ✅ 确保进入合成阶段的素材质量

**逻辑问题**：

**问题 4.1: 硬校验与软审查混合**
```
硬校验（脚本）→ PASS → 软审查（LLM）→ 修复 → 重跑硬校验
```
- 软审查没有对应的自动化支持
- LLM 审查标准主观性强
- 无"软审查 Pass"的明确标记

**问题 4.2: 社区陷阱卡验证薄弱**
- 代码规则可检查 file:line 引用是否存在
- 社区陷阱卡只能检查 Issue 编号是否出现在 community_signals.md
- **无法验证 Issue 编号是否真的相关**（如 DR-101 引用 Issue #89，但内容可能无关）

**问题 4.3: 验证报告的位置**
- `validation_summary.md` 写入 `soul/` 目录
- 但这是临时验证产物，不应混入最终交付物

### 5. 用户反馈机制分析

**当前设计**：
每个阶段完成后 LLM 向用户汇报：
- Stage 0: "代码已下载（X bytes），开始分析..."
- Stage 1: 告知"设计哲学"和"心智模型"
- Stage 2: "已提取概念 N/3"、"已分析工作流 N/3"
- Stage 3: "已提取规则 N: [标题] (严重度: X)"
- Stage 3.5: "验证完成，X/Y 张卡片通过硬校验，Z 张卡片经过内容改进"
- Stage 4: "专家叙事合成完成"
- Stage 5: 展示 project_soul.md 内容

**优点**：
- 每阶段有汇报
- 进度可见

**问题**：

| 问题 | 描述 |
|------|------|
| 无定量进度 | 没有百分比、无 ETA |
| 无错误可视化 | 用户看不到具体哪个卡片失败、为什么失败 |
| 无交互反馈 | 用户无法中途干预（如"跳过这个规则"、"补充这个方向"） |
| 无取消机制 | 开始后无法优雅中止 |

## 改进建议

### P0: 中断续跑机制

**方案：引入 `.progress.json` 进度文件**

```json
{
  "repo": "python-dotenv",
  "started_at": "2026-03-10T12:00:00Z",
  "stages": {
    "0": {"status": "completed", "output": "~/soul-output/python-dotenv/artifacts/"},
    "1": {"status": "completed", "output": "~/soul-output/python-dotenv/soul/00-soul.md"},
    "2": {"status": "in_progress", "completed": ["CC-001", "CC-002"], "pending": ["CC-003", "WF-001", "WF-002", "WF-003"]},
    "3": {"status": "pending"},
    "3.5": {"status": "pending"},
    "4": {"status": "pending"},
    "5": {"status": "pending"}
  },
  "last_updated": "2026-03-10T12:15:00Z"
}
```

**恢复命令**：
```
extract soul <repo_url> --resume
```

**实现要点**：
- Stage 0 完成后写入 `.progress.json`
- 每个子任务（如每张卡片）完成后更新进度
- 中断后检查 `.progress.json`，跳过已完成阶段
- Stage 5 完成后可选删除进度文件

### P0: 失败自动修复循环

**方案：Stage 3.5 引入自动修复循环**

```
硬校验 → FAIL → 自动解析错误 → LLM 修复建议 → 应用修复 → 重跑硬校验
        ↑___________________________________________|
                    最多 3 轮
```

**伪代码**：
```python
max_retries = 3
for attempt in range(max_retries):
    result = run_validation()
    if result.pass:
        break
    errors = parse_errors(result)
    fixes = llm_suggest_fixes(errors)
    apply_fixes(fixes)
else:
    raise ValidationError("Max retries exceeded")
```

### P1: Stage 2/3 并行化

**方案：引入并行阶段标记**

修改 SKILL.md：
```markdown
## Stage 2-3: 并行提取（概念/工作流 + 规则）

读取并并行执行以下指令：
- `{baseDir}/stages/STAGE-2-concepts.md`
- `{baseDir}/stages/STAGE-3-rules.md`

两者都依赖代码和灵魂文件，但产出互不依赖。
```

**实现方式**：
- 如果单 LLM 会话：在 prompt 中明确"先输出 Stage 2 全部，再输出 Stage 3 全部"
- 如果支持多 Agent：可 spawn 两个子任务并行执行

### P1: 验证阶段增强

**方案 4.1: 分离硬校验与软审查**

```
Stage 3.5a: 硬校验（脚本，自动循环修复）
Stage 3.5b: 软审查（LLM，输出审查报告）
```

硬校验失败 → 自动循环修复（P0 方案）
软审查失败 → 输出警告，但不阻塞流程

**方案 4.2: 社区陷阱卡增强验证**

新增检查：
- Issue 编号必须在 community_signals.md 中存在
- Issue 编号对应的 Tier 必须是 1 或 1.5（高价值信号）
- 可选：Issue 标题与卡片标题相关性检查（基于 embedding）

### P2: 用户反馈增强

**方案：引入结构化进度报告**

```markdown
## 进度报告

### 总体进度
- [x] Stage 0: 准备 (完成)
- [x] Stage 1: 灵魂发现 (完成)
- [ ] Stage 2: 概念提取 (进行中 2/6)
  - [x] CC-001
  - [x] CC-002
  - [ ] CC-003
  - [ ] WF-001
  - [ ] WF-002
  - [ ] WF-003
- [ ] Stage 3: 规则提取 (待开始)
...

### 当前状态
- 已用 Token: 45,000 / 200,000
- 预计剩余时间: ~8 分钟
- 最后更新: 2026-03-10 12:15:00
```

**交互式反馈**：
- 允许用户输入 `skip` 跳过当前子任务
- 允许用户输入 `pause` 暂停并保存进度
- 允许用户输入 `more details on X` 深入某个卡片

## 优先级排序

### P0（阻塞问题，必须修复）

| 问题 | 影响 | 建议方案 |
|------|------|----------|
| 中断续跑机制缺失 | 中断后需完全重来，浪费资源 | 引入 `.progress.json` + `--resume` 命令 |
| 失败无自动修复 | LLM 需手动解读错误、修复 | Stage 3.5 引入自动修复循环 |

### P1（重要改进，显著提升体验）

| 问题 | 影响 | 建议方案 |
|------|------|----------|
| Stage 2/3 假顺序依赖 | 浪费时间，无法并行 | 并行化改造 |
| 验证阶段混合逻辑 | 硬校验失败阻塞软审查 | 分离 Stage 3.5a/3.5b |
| 社区陷阱卡验证弱 | 低质量卡片可能通过 | 增强 Issue 相关性检查 |

### P2（体验优化，可迭代改进）

| 问题 | 影响 | 建议方案 |
|------|------|----------|
| 进度反馈不够量化 | 用户不知道还要多久 | 结构化进度报告 |
| 无交互式反馈 | 用户无法干预 | 支持 skip/pause 命令 |
| 验证报告位置混乱 | 临时文件混入交付物 | 移至 `artifacts/validation/` |
