# Doramagic v2 技术先进性差距分析

> 日期: 2026-03-11
> 问题: 当前 v2 设计方案是否代表 2026 年最先进的技术和工程实践？
> 结论: **产品方向正确，技术架构落后 1-2 代**

---

## 1. 六个关键差距

### 差距 1: Resource Card 本质上是在手工重造 MCP

**现状**: v2 方案设计了 Resource Card（资源卡片），手动提取 API 的调用方式、认证要求、速率限制等。

**2026 现实**: MCP (Model Context Protocol) 已经是 AI 工具集成的行业标准。一个 MCP Server 就是一个"活的 Resource Card"——它不仅描述自己能做什么，还能直接被 AI 调用。

**差距本质**: 我们在用静态文档描述资源，而 2026 的标准是用可执行接口暴露资源。

**应该做的**:
- Phase 3 的"资源提取"不应该生成 Resource Card 文档
- 而应该**发现现有 MCP Server** 或 **自动生成 MCP Server wrapper**
- 生成的 Skill 通过 MCP 协议连接资源，而不是内嵌 API 调用代码

```
v2 原方案（2024 思维）:
  提取资源信息 → 写成 Resource Card → 拼装时内嵌调用代码

v2 升级方案（2026 思维）:
  发现/生成 MCP Server → Skill 通过 MCP 协议调用 → 资源可热插拔
```

**好处**:
- 资源可以独立更新（API 变了，换 MCP Server 就行，Skill 不用改）
- 资源可以跨 Skill 复用（akshare MCP Server 给所有股票 Skill 用）
- 符合 OpenClaw 生态的原生集成模式

---

### 差距 2: 批处理流水线是 2024 的 Agent 设计

**现状**: 5 阶段串行流水线（需求→发现→提取→拼装→交付），每个阶段完成后才进入下一个。

**2026 现实**: 最先进的 Agent 架构是**自适应循环**（Adaptive Loop），不是线性流水线。Agent 可以：
- 在拼装阶段发现知识不够，自动回到提取阶段补充
- 在交付阶段发现冒烟测试失败，自动回到拼装阶段修复
- 根据中间结果动态调整策略（而不是预设好5个阶段硬跑）

**差距本质**: 线性流水线假设"每个阶段的输出是确定的"，但现实中提取和拼装是高度不确定的——需要反馈循环。

**应该做的**:
```
v2 原方案（线性流水线）:
  P1 → P2 → P3 → P4 → P5 → 完成

v2 升级方案（自适应循环 + 子代理团队）:
  ┌─────────────────────────────────────────┐
  │           Orchestrator Agent             │
  │  （总控代理：理解目标，动态分配任务）      │
  └─────────┬───────────────────────────────┘
            │ 动态调度
    ┌───────┼───────┬───────────┐
    ▼       ▼       ▼           ▼
  需求      发现     提取        拼装
  Agent    Agent   Agent(s)    Agent
    │       │       │           │
    └───────┴───────┴───────────┘
            │ 共享工作记忆
            ▼
      Delivery Agent
```

核心变化：
- **Orchestrator** 不是按顺序执行阶段，而是根据当前状态决定下一步
- 子代理之间通过**共享工作记忆**（而非文件传递）通信
- 任何子代理发现问题，Orchestrator 可以动态调整计划
- 使用 Claude Agent SDK 的 handoff 模式实现代理间切换

### 差距 3: Repomix 文本压缩是 2025 的代码理解方式

**现状**: 用 Repomix 把整个仓库压缩成一个 XML 文件，喂给 LLM 理解。

**2026 现实**:
- **Tree-sitter AST 分析**: 结构化理解代码，而不是把代码当文本
- **代码嵌入 (Code Embeddings)**: 将函数/类转为向量，做语义搜索
- **按需加载**: 不是一次性喂入全部代码，而是 Agent 按需读取相关部分
- **Repomix 自己也在进化**: compress mode + Tree-sitter 已经内置

**差距本质**: 把代码当"一坨文本"处理 vs 把代码当"结构化知识图"处理。

**应该做的**:
```
v2 原方案:
  git clone → repomix compress → 一个大 XML → 喂 LLM

v2 升级方案:
  git clone → AST 分析（函数/类/模块结构）
            → 代码嵌入（语义索引）
            → Agent 按需查询（"找到所有 API 端点定义"）
            → 只把相关代码片段喂 LLM
```

**好处**:
- 大仓库不再受 token 限制（按需加载，不是全量压缩）
- 提取更精准（AST 级别理解 vs 文本匹配）
- 资源提取准确率大幅提升（直接从 AST 找到函数签名、路由定义、SDK 调用）

---

### 差距 4: 没有持续学习机制——生成的 Skill 是"死"的

**现状**: Skill 生成后就是静态文件。用户使用过程中的反馈（好用/不好用/错误）不会回流改进 Skill。

**2026 现实**: 最好的 AI 产品都有反馈闭环：
- 用户说"这个选股结果不准"→ Skill 自动调整策略权重
- 用户反复修改某个参数 → Skill 记住偏好
- 数据源 API 变更 → Skill 自动检测并提醒升级

**差距本质**: 一次性交付 vs 持续进化。

**应该做的**:
```
v2 原方案:
  生成 Skill → 交付 → 结束

v2 升级方案:
  生成 Skill → 交付 → 用户使用
                         ↓
                    反馈收集（隐式 + 显式）
                         ↓
                    Skill 自我优化
                    ├─ 参数微调（用户偏好）
                    ├─ 规则补充（新踩坑经验）
                    └─ 资源更新（API 变更检测）
```

**实现方式**: 利用 OpenClaw 的 MEMORY.md 机制
- Skill 运行时将用户反馈写入记忆
- 下次运行时读取记忆，动态调整行为
- 定期触发"Skill 体检"——检查资源是否仍然可用

---

### 差距 5: 源泉发现靠关键词搜索——没有语义理解

**现状**: Phase 2 搜索靠关键词（"A股数据"、"量化选股"）匹配 GitHub 项目。

**2026 现实**: 语义搜索 + 知识图谱是标准做法：
- 用户说"帮我选股"→ 语义理解为"需要：金融数据获取 + 筛选算法 + 呈现能力"
- 在预建的开源项目知识图谱中，沿着"金融数据获取"节点找到 akshare、tushare、wind
- 沿着"筛选算法"节点找到 vnpy、qlib、backtrader
- 根据用户约束（免费、简单、A股）排序

**差距本质**: 关键词匹配找到"名字像的"，语义搜索找到"功能匹配的"。

**应该做的**:
- 建立开源项目的**能力向量索引**（不是项目名索引）
- 用户需求 → 能力需求分解 → 语义匹配最相关的项目/Skill/MCP Server
- 冷启动：可以先用 LLM 知识 + 实时搜索；长期建立持久索引

---

### 差距 6: 没有利用 OpenClaw 的原生能力

**现状**: 方案把 Doramagic 当作一个独立系统设计，只是最终"部署到 OpenClaw"。

**2026 现实**: OpenClaw 提供的原生能力应该被充分利用：

| OpenClaw 能力 | 当前方案是否利用 | 应该怎么用 |
|-------------|-------------|---------|
| **MEMORY.md** | 未利用 | Skill 运行记忆 + 用户偏好持久化 + 反馈闭环 |
| **AGENTS.md** | 未利用 | 定义子代理角色（提取代理、拼装代理等） |
| **内置工具** (web search, file ops) | 未利用 | Phase 2 源泉发现直接调用 |
| **MCP Server 集成** | 未利用 | 资源层通过 MCP 接入，而非内嵌代码 |
| **Skill 间调用** | 未利用 | 生成的 Skill 可以调用其他已安装 Skill |
| **多模型调度** | 部分利用 | 复杂任务用 Opus，简单任务用 Haiku/Flash |

## 3. 分层实施建议

不需要一次性全做。按影响力排序：

### 第一优先级：MCP-first 资源层 (影响力最大)

**原因**: 这是"抄作业引擎"的核心——不是抄知识，是抄能力。MCP 让抄来的能力可以直接运行。

**具体做法**:
1. 生成的 Skill 通过 MCP 协议调用外部资源（而非内嵌代码）
2. Phase 3 的资源提取变为"发现或生成 MCP Server"
3. 维护一个 MCP Server 注册表（已知的、可用的）
4. 对于没有现成 MCP Server 的资源，自动生成 wrapper

**用股民老王的例子**:
```
原方案: Skill 内嵌 `import akshare; ak.stock_zh_a_spot_em()`
升级方案: Skill 调用 `mcp://stock-data/realtime?market=A`
         → 背后对接 akshare MCP Server（可独立更新/替换）
```

### 第二优先级：Orchestrator 自适应循环

**原因**: 线性流水线在提取失败或拼装不满意时无法自愈。

**具体做法**:
1. 用 Claude Agent SDK 构建 Orchestrator
2. 子代理通过 handoff 协议切换
3. 共享工作记忆存储中间结果
4. Orchestrator 根据状态决定下一步（而非硬编码阶段顺序）

### 第三优先级：反馈闭环 + Skill 自我优化

**原因**: 这是从"工具"变成"产品"的关键——用户留存取决于 Skill 是否越用越好。

**具体做法**:
1. Skill 运行时将用户反馈写入 MEMORY.md
2. 下次运行读取记忆，动态调整
3. 定期触发"Skill 体检"
4. 可选：用户授权后，匿名反馈汇聚改进全局模板

### 第四优先级：AST 级代码理解

**原因**: 提升提取质量上限，解决大仓库 token 限制问题。

### 第五优先级：语义发现 + 能力图谱

**原因**: 长期竞争壁垒，但冷启动可以先用 LLM 知识 + 实时搜索替代。

## 4. 哪些原方案的设计是正确的（不需要改）

1. **"需求理解先于知识提取"** — 这个产品直觉完全正确
2. **苏格拉底对话** — 3-5 轮对话发现真需求，方法论成熟
3. **"开袋即食 + 可选升级"** — 出色的产品设计，降低使用门槛
4. **进度透明** — 实时展示而非黑盒等待，UX 正确
5. **知识卡片体系 (CC/WC/DR)** — 结构化知识仍然有价值，作为内部中间表示
6. **质量验证门控** — 硬门控防止垃圾输出，必须保留
7. **多源提取并行** — 子代理并行提取，效率最优

这些都是对的，不需要推翻。需要升级的是**底层技术实现**，不是产品逻辑。

---

## 5. 一句话总结

**v2 原方案是用 2024 的技术栈做了 2026 的产品设计。**

产品方向（超级 Copy Cat + 开袋即食 + OpenClaw 生态）完全正确。
技术架构需要升级到 MCP-first + 自适应 Agent + AST 理解 + 反馈闭环。

好消息是：产品方向对了，技术升级是工程问题；
坏消息是：如果技术不升级，竞争对手用 MCP + Agent SDK 做同样的事会快得多。
