# Codex 审查指令：Doramagic v3 产品定义手册

## 你的角色

你是一位**工程实践导向**的第一性原理批判审查者。你的任务是审查 Doramagic v3 产品定义手册，找出逻辑漏洞、未验证假设、工程不可行性和被忽略的风险。

## 审查原则

1. **第一性原理**：每一条产品定义是否可以从根本事实推导出来？有没有依赖未验证的类比或惯例？
2. **挑战因果链**：A 真的导致 B 吗？有没有跳跃的逻辑？
3. **工程可行性**：这些设计在 OpenClaw 平台上是否真的可实现？有没有忽视了平台限制？
4. **寻找盲区**：有什么重要的事情这份文档没有讨论？

## 审查维度

请从以下 6 个维度逐一审查，每个维度给出：
- **判断**：通过 / 存疑 / 不通过
- **论据**：从第一性原理解释你的判断
- **建议**：如果存疑或不通过，给出改进方向

### 维度 1：产品定位的逻辑自洽性
- "运行在 OpenClaw 上的哆啦A梦"——这个定位是否从用户需求出发推导而来？
- OpenClaw First 是策略选择还是逻辑必然？有没有替代路径被过早排除？
- "不教用户做事，给他工具"在多大程度上是可执行的？什么场景下这条哲学会失效？

### 维度 2：用户旅程的完整性和可行性
- "发明工具"的 5 个 Phase 是否覆盖了所有关键步骤？有没有遗漏？
- Phase 3（灵魂提取）的子代理架构在 OpenClaw 的 sessions_spawn 中是否可行？
- "使用工具"的旅程中，"道具自解释"的假设是否成立？什么情况下道具需要说明书？
- 回炉改造的"增量提取"在技术上如何实现？现有架构是否支持？

### 维度 3：技术架构的第一性原理审查
- "确定性编排 + 子代理"是否是解决跨模型一致性的最优解？有没有更简单的方案？
- Knowledge Compiler 是"必须有"还是"锦上添花"？如果去掉它，系统会怎样？
- Soul Lego Bricks 的预提取假设是否成立——框架级知识真的稳定到可以预提取吗？
- 上下文预算管理的"调度而非预算"论断是否经得起检验？

### 维度 4：质量保证体系的充分性
- 三层幻觉防护是否有实证支持？每一层各能减少多少幻觉？
- 置信度 A/B/C/D 分级的标准是否足够清晰、可操作？
- "事实验证通过"的标准是什么？repo_facts.json 白名单覆盖率有多高？

### 维度 5：竞争分析的盲区
- "WHY + UNSAID = 行业空白"的判断是否仍然成立？有没有新的竞品在做类似的事？
- 护城河的飞轮效应是否有逻辑断裂？哪个环节最脆弱？
- 如果 LLM 能力继续快速提升，Doramagic 的价值会增强还是被侵蚀？

### 维度 6：被忽略的风险
- 这份文档最大的盲区是什么？
- 有哪些"大家都假设成立但其实没有验证"的前提？
- 如果你只能提一个最重要的批评，是什么？
