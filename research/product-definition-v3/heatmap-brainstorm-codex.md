# 依赖热力图头脑风暴 - Codex

## 先换一个看法：热力图不是一张图，而是一组"抄作业雷达"

一个项目里，"值得抄"至少有 4 种完全不同的价值：

1. **共识价值**：这是开源世界反复验证过的标准做法，适合做积木。
2. **杠杆价值**：这段代码很小，但牵动全局，抄到就抓住主脉络。
3. **差异价值**：这里偏离框架默认，体现项目真正的独门手法。
4. **踩坑价值**：这里充满异常处理、补丁、兼容层，蕴含大量隐性经验。

所以一个单一分数的 heatmap 可能过早压平信息。更好的方向也许不是 `score=87`，而是：

```json
{
  "node": "auth/session_resolver.py",
  "consensus_heat": 0.62,
  "leverage_heat": 0.91,
  "novelty_heat": 0.83,
  "trap_heat": 0.77
}
```

这样 Doramagic 后续每个组件都能按自己的目标消费不同热度，而不是被迫吃一个混合分数。

新颖度：有趣  
关联度：直接

## 1. 维度发现：除了引用计数，还能看什么

### 1.1 爆炸半径

如果一个点改动后，影响很多测试、很多调用链、很多下游模块，它就不是普通代码，而是"脉络开关"。

- 可计算方式：静态 fan-in/fan-out + 测试覆盖映射 + 入口到该节点的最短路径数量
- 抄作业价值：抓到这种点，等于抓到项目真正的受力结构

新颖度：常规  
关联度：直接

### 1.2 协议密度

有些代码不是复杂，而是承载了大量"隐形协议"：字段命名、状态码、事件格式、序列化约定、错误语义。

- 典型位置：serializer、schema、DTO、adapter、parser、migration
- 可计算方式：字段数、schema 版本数、接口实现数、格式转换层数量
- 抄作业价值：这种地方最像"答案模板"，适合 Doramagic 把经验压成可复用知识

新颖度：有趣  
关联度：直接

### 1.3 状态转移密度

如果一个模块管理很多状态切换，它往往背后有真正的业务判断。

- 典型信号：state enum、workflow step、retry state、pending/confirmed/failed/cancelled
- 可计算方式：状态节点数、转移边数、守卫条件数
- 抄作业价值：状态机往往比 CRUD 更接近项目灵魂

新颖度：有趣  
关联度：直接

### 1.4 不可逆操作权重

不是所有代码都一样重要。写数据库、发消息、扣款、认证、删数据、执行外部副作用的代码，天然更值得提取。

- 可计算方式：识别写路径、事务边界、消息发布、文件删除、第三方 API side effect
- 抄作业价值：不可逆动作附近的防御逻辑，往往就是作者最宝贵的踩坑经验

新颖度：有趣  
关联度：直接

### 1.5 人类注意力密度

代码里最值得学的地方，不一定最被代码引用，但可能最被人类解释。

- 信号来源：README 链接、架构图提及、ADR、注释密度、长 commit message、PR discussion
- 抄作业价值：作者愿意解释的地方，通常是默认直觉不够用的地方

新颖度：有趣  
关联度：直接

### 1.6 入口收敛度

很多请求、很多命令、很多任务最后都汇聚到同一组少数节点，这些节点就是系统里的"收费站"。

- 可计算方式：从入口函数、HTTP route、CLI command、queue consumer 向下游做调用图收敛分析
- 抄作业价值：收费站后面的设计，通常代表项目的主组织方式

新颖度：常规  
关联度：直接

### 1.7 偏离默认值的幅度

项目最值得学的，常常不是"用了 Django"，而是"为什么没按 Django 默认方式来"。

- 可计算方式：用积木里的标准模式作为 baseline，检测项目代码的结构偏差
- 例子：自定义权限系统、自定义 ORM 包装、自定义 cache invalidation
- 抄作业价值：偏离即增量，增量即精华

新颖度：有趣  
关联度：直接

### 1.8 低样板率

真正有价值的模块通常不是重复模板，而是低重复、强组织性、强语义压缩的代码。

- 可计算方式：token 去重率、AST 模式重复率、生成代码识别、CRUD 模板识别
- 抄作业价值：先把样板降权，Doramagic 才能把预算留给精华

新颖度：常规  
关联度：直接

### 1.9 故障吸收能力

一个点是不是承担了大量 fallback、retry、circuit breaker、degrade、兼容逻辑？

- 这类模块像减震器，不显眼，但含有大量现实世界经验
- 失败世界的信息密度，常常高于成功世界

新颖度：有趣  
关联度：直接

### 1.10 术语锚点

如果某个文件/类/函数反复出现项目特有术语，它很可能是领域知识的核心载体。

- 可计算方式：领域词频、命名聚类、术语与调用图重叠
- 抄作业价值：帮助 Doramagic 分辨"业务灵魂"和"技术基础设施"

新颖度：有趣  
关联度：直接

### 1.11 新手踩坑浓度

某些模块会在 git 历史里反复被修修补补，尤其是小修小补、边界 case 修复、回滚修复。

- 可计算方式：bugfix commit 聚类、hotspot analysis、短周期反复修改
- 抄作业价值：这些地方可能不是最优雅，但最接近真实经验

新颖度：有趣  
关联度：直接

### 1.12 安静但稳定的抽象

有些模块几乎不改、也不大、甚至调用数不高，但一旦存在就多年不动。这可能是项目真正磨出来的成熟抽象。

- 与"频繁修改 = 重要"相反
- 稳定本身也可能是知识成熟度的信号

新颖度：疯狂  
关联度：待验证

## 2. 跨项目信号：如果能看开源世界，会发生什么

### 2.1 共识图 vs 差异图

Doramagic 不应该只知道"大家都用什么"，还要知道"这个项目跟大家不一样的地方在哪"。

- 共识图：跨项目高频模式
- 差异图：当前项目相对共识图的偏移量
- 真正值得抄的，常常不是高频本身，而是"在高频背景上的偏移"

新颖度：有趣  
关联度：直接

### 2.2 用法级依赖图

不是只看谁依赖某个库，而是看大家**怎么**依赖它。

- 同一个库，不同函数、不同 hook、不同扩展点，价值完全不同
- 可形成"库内热点图"
- 比如不是"大家用了 Django"，而是"大家把自定义 manager、signal、middleware 用在哪些组合里"

新颖度：有趣  
关联度：直接

### 2.3 技术栈共现图

不是单个库热，而是组合热。

- React + Zustand + TanStack Query
- Django + Celery + Redis
- FastAPI + Pydantic + SQLAlchemy

抄作业不是抄一个零件，而是抄一套已验证的搭配关系。

新颖度：常规  
关联度：直接

### 2.4 迁移轨迹图

如果很多项目从 A 迁到 B，这条迁移路径本身就是高价值知识。

- 迁移意味着旧方案的痛点和新方案的收益已被大规模验证
- Doramagic 可以把"迁移经验"做成特殊积木

新颖度：有趣  
关联度：间接

### 2.5 维护强度图

下载量高不代表知识质量高。还要看维护者响应速度、issue 关闭结构、breaking change 频率。

- 高使用 + 高维护 = 共识且健康
- 高使用 + 低维护 = 共识但脆弱
- 低使用 + 高维护 = 小众但高质量

新颖度：有趣  
关联度：间接

### 2.6 "复制谱系"追踪

如果多个项目里出现高度相似的代码形态，说明开源世界已经形成了隐形范式。

- 类似论文里的 citation lineage，但对代码做 clone lineage
- 不只知道某模式存在，还能知道它如何演化

新颖度：疯狂  
关联度：待验证

### 2.7 问题密度信号

Stack Overflow、GitHub issue、博客问答里反复出现的问题，反而说明这里值得做积木。

- 不是因为它优雅，而是因为它是现实世界的高频痛点
- 高问题频率 = 高辅导价值

新颖度：常规  
关联度：直接

## 3. 备件库设计：积木仓库本身也应该有热力图

### 3.1 积木不是列表，是分层图

建议至少有 4 类边：

- `contains`：django-core -> django-orm
- `composes-with`：django -> celery
- `substitutes`：celery -> rq
- `conflicts-with`：某些积木组合后会互相稀释或冲突

这样积木仓库本身就成为一张"知识供应链图"。

新颖度：有趣  
关联度：直接

### 3.2 积木要分"标准积木"与"偏离积木"

- 标准积木：框架默认哲学、常规最佳实践
- 偏离积木：哪些偏离是常见而有效的，哪些偏离是危险反模式

这样 Doramagic 不只会告诉模型"一般怎么做"，也会告诉它"什么情况下应该违背一般做法"。

新颖度：有趣  
关联度：直接

### 3.3 积木的优先级不只看流行度

可以用一个三轴排序：

1. 需求频率
2. 迁移价值
3. 失败成本

例如认证、支付、缓存一致性，即便不是最流行，也应该优先做，因为失败代价高。

新颖度：有趣  
关联度：直接

### 3.4 积木要有"保鲜期"和"冻结期"

- 保鲜期：多久需要复查一次
- 冻结期：这个知识足够稳定，可以长期信任

例如 React Router 的实践可能短保鲜，Unix 管道哲学可能长冻结。

新颖度：常规  
关联度：直接

### 3.5 积木需要"证据索引"

每块积木不只是结论，还应该带：

- 来自哪些高质量项目
- 在哪些版本范围内成立
- 哪些反例出现过
- 适用条件是什么

这让积木从"提示词碎片"升级为"可验证知识模块"。

新颖度：有趣  
关联度：直接

### 3.6 可以有"备胎积木"

当主积木置信度不足、版本不匹配、或项目明显偏离时，自动切换到备胎积木。

- 类似数据库读写分离里的 failover
- 积木系统不再是静态加载，而是动态路由

新颖度：疯狂  
关联度：待验证

## 4. 热力图与知识密度：能否纯确定性估算

我怀疑可以做一个"知识密度代理模型"，不求完美，但求排序有用。

### 4.1 决策点密度

- 条件分支数
- feature flag 数
- config 读取点数
- 错误分支数

这些都代表作者被迫做判断的地方。

新颖度：常规  
关联度：直接

### 4.2 协议表面积

- schema 字段数
- 接口方法数
- 序列化映射数
- 向后兼容分支数

协议越多，隐藏知识越多。

新颖度：有趣  
关联度：直接

### 4.3 状态机复杂度

- 状态节点
- 转移边
- 守卫条件
- 回退路径

状态越多，设计决策越浓。

新颖度：常规  
关联度：直接

### 4.4 跨工件耦合度

如果一段逻辑同时出现在代码、测试、文档、配置、迁移脚本中，它很可能不只是代码技巧，而是系统级知识。

- 这是"多处被显式表述"的强信号

新颖度：有趣  
关联度：直接

### 4.5 例外密度

正常路径很容易被 LLM 猜对，例外路径才体现项目经验。

- retry
- fallback
- 兼容旧版本
- patch third-party bug
- 灰度和回滚

例外越多，知识密度往往越高。

新颖度：有趣  
关联度：直接

### 4.6 历史沉淀比

一个模块经历过大量修改后趋于稳定，可能意味着它吸收了长期经验。

- 不是只看"改得多"
- 而是看"改得多之后稳下来了"

新颖度：疯狂  
关联度：待验证

### 4.7 反 boilerplate 分数

先确定性识别：

- generated code
- ORM model field declarations
- REST CRUD controller 模板
- 构建工具配置模板

然后把这些区域降权。知识密度估算的关键不只是"发现高价值"，还要"去掉假热度"。

新颖度：常规  
关联度：直接

## 5. 类比：别把热力图只想成代码图

### 5.1 城市交通图

主干道流量大，但真正决定城市品质的，常常是换乘节点、桥梁、隧道、收费站。

启发：

- 不要只看高频节点
- 要看跨区域连接点
- 跨上下文桥接层，往往高于普通高流量模块

新颖度：常规  
关联度：间接

### 5.2 机场枢纽图

大型机场不是因为航班最多才重要，而是因为它连接了很多原本不相通的线路。

启发：

- auth adapter
- data normalization layer
- event bus translator

这些"连接异质系统"的地方，可能是项目真正的抄作业黄金点。

新颖度：有趣  
关联度：直接

### 5.3 生物神经系统

肌肉很多，神经很少，但神经决定全身行为。

启发：

- 代码量大的地方不一定重要
- 控制信号、协调信号、反馈信号密集处更重要

新颖度：有趣  
关联度：直接

### 5.4 法律判例网络

不是每条法律都常用，但某些判例一旦出现，会改变后续一大片案件的解释方式。

启发：

- 某些 commit 或 ADR 可能是解释枢纽
- 热力图对象不应只限于代码节点，还应包括文档节点、迁移节点、设计节点

新颖度：有趣  
关联度：直接

### 5.5 供应链关键零件

一个便宜但不可替代的小零件，价值远高于一个昂贵但可替代的大零件。

启发：

- 识别"缺了就全线停工"的模块
- 这类模块适合重点提取 WHY 和失败条件

新颖度：常规  
关联度：直接

## 6. 反直觉想法

### 6.1 最冷的地方可能最值钱

高热节点很多时候只是大家都知道的公共套路。真正难得的，可能是一个冷门文件里藏着项目的独家判断。

- 例如计费规则、配额策略、兼容旧客户的奇怪逻辑
- 少人调用，不代表不重要；可能只是被包裹在一个窄接口里

新颖度：疯狂  
关联度：直接

### 6.2 最值得抄的不是代码，而是版本演化

某个方案为什么从 v1 改成 v2，往往比 v2 本身更有价值。

- commit message
- migration note
- deprecated path
- compatibility shim

这些地方直接暴露"作者为什么改主意"。

新颖度：疯狂  
关联度：直接

### 6.3 不应该只有一张热力图

建议至少拆成三张：

- **共识图**：哪些地方适合复用积木
- **偏离图**：哪些地方体现项目独门手法
- **风险图**：哪些地方隐藏大量失败经验

一个区域可以在三张图上的表现完全不同。

新颖度：有趣  
关联度：直接

### 6.4 Stage 0 只做粗图，真正精图在后面长出来

Stage 0 擅长静态结构，不擅长理解"为什么"。所以它更像 CT，不是完整诊断。

- Stage 1 读到 WHY 后修正热区
- Stage 2-3 提炼概念时再修正
- 用户使用道具后的真实反馈继续修正

热力图应该是演化对象，不是一次性产物。

新颖度：有趣  
关联度：直接

### 6.5 热力图对象不应该只是代码

还应该把这些纳入节点：

- ADR / design doc
- migration script
- test case
- CI rule
- example / cookbook
- incident postmortem

因为抄作业抄的是智慧，不是只抄 `.py` 和 `.ts`。

新颖度：有趣  
关联度：直接

### 6.6 可以故意寻找"LLM 记忆盲区"

如果一个项目很新、很垂直、star 不高，但设计质量好，它反而更适合作为 Doramagic 的提取对象。

- 因为这时真正的增量知识更多
- 热力图可以加入"训练集污染概率"的反向权重

新颖度：疯狂  
关联度：待验证

## 7. 与 Doramagic 其他组件怎么联动

### 7.1 热力图 x 积木系统 = 偏离定位器

积木给出"世界通常怎么做"，热力图给出"这个项目真正把精力花在哪"。

两者一相减，就得到：

- 哪些地方只是标准框架套路
- 哪些地方是项目自己的增量智慧

这可能是 Doramagic 最核心的协同关系。

新颖度：有趣  
关联度：直接

### 7.2 热力图 x 置信度系统 = 验证预算分配器

高热且低置信度的点，应优先做交叉验证。

低热且高置信度的点，可以快速压缩。

这等于把置信度系统从"判断对不对"升级为"决定验证预算投到哪"。

新颖度：有趣  
关联度：直接

### 7.3 热力图 x Knowledge Compiler = 上下文配额控制器

Knowledge Compiler 不该平均压缩所有内容，而应按热度和热度类型做配额。

- 高杠杆节点保留更多 WHY
- 高风险节点保留更多失败条件
- 高共识节点更多依赖积木摘要

新颖度：有趣  
关联度：直接

### 7.4 热力图 x 子代理架构 = 调度中枢

不同热区交给不同类型的子代理：

- 共识高：偏积木消费型子代理
- 偏离高：偏对比分析型子代理
- 风险高：偏证据验证型子代理
- 协议高：偏结构抽取型子代理

这样子代理不是按阶段分工，还能按知识类型分工。

新颖度：疯狂  
关联度：直接

### 7.5 热力图 x 道具使用反馈 = 真实价值闭环

如果用户反复使用某个能力，说明支撑该能力的知识区域在市场上真的值钱。

热力图不再只是代码世界的图，而是"代码价值 x 用户需求"的联合图。

新颖度：有趣  
关联度：直接

## 8. 动态性：热力图应该如何演化

### 8.1 三阶段演化

建议把热力图拆成三个版本：

1. **静态热力图**：Stage 0 基于结构和历史快速生成
2. **语义热力图**：Stage 1-3 结合 WHY/概念/规则修正
3. **市场热力图**：道具使用反馈、失败反馈、重锻请求反哺

新颖度：有趣  
关联度：直接

### 8.2 时间衰减

热度不应该永久有效。

- 老旧模块如果长期无变化，可能是稳定共识
- 也可能是无人维护的废墟

所以热度需要与更新时间、积木版本、依赖版本共同解释。

新颖度：常规  
关联度：直接

### 8.3 用户群体差异

不同用户想抄的不是同一份作业。

- 新手用户更需要共识热区
- 进阶用户更需要偏离热区
- 企业用户更需要风险热区

也许热力图最终应该被用户画像重排。

新颖度：有趣  
关联度：间接

### 8.4 从"项目热度"进化到"任务热度"

同一个项目，做登录系统、做爬虫、做后台管理、做支付接入，关注区域完全不同。

所以热力图可能不是 project-level 唯一真相，而是 task-conditioned view。

新颖度：疯狂  
关联度：直接

---

## 9. 一个可能很有用的产物：热力图卡片类型

热力图最终未必要只输出分数。可以输出更适合编排层消费的卡片类型：

- `consensus_anchor`：适合交给积木吸收
- `novel_pattern`：适合重点提取 WHY
- `risk_sink`：适合优先做证据验证
- `protocol_hub`：适合抽取接口与约定
- `glue_code`：适合理解技术栈组合方式
- `artifact_node`：非代码但高价值的文档/测试/迁移节点

这样下游系统消费的是"可行动标签"，不是抽象分数。

新颖度：有趣  
关联度：直接

## 11. 一句话总结

如果只把热力图理解成"哪里引用多"，Doramagic 只能抄到代码骨架。  
如果把热力图理解成"哪里汇聚了共识、偏离、杠杆、风险与用户真实价值"，Doramagic 才有机会抄到开源项目真正的灵魂。
