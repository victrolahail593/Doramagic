# Codex 第一性原理审查报告：Doramagic v3 产品定义

## 总体评价
v3 比 v2 明显更工程化，尤其把“代码说事实，AI 说故事”落到了编排层、验证层和消费格式层；但当前版本仍存在核心因果链未闭合的问题：若把“可实现性”和“可验证性”作为第一性约束，v3 里最关键的三条主张（OpenClaw First 必然性、子代理可行性、三层防幻觉有效性）都还缺少可复现实证与失败边界定义，因此整体判断是“方向正确，但论证强度不足以支撑大规模投入”。

## 维度 1: 产品定位
### 判断: 存疑

### 论据
- “AI 道具需要运行时”可以推出“需要某个运行时”，但不能直接推出“必须是 OpenClaw 且非可选”（v3:101-106, 28）。这是从“存在性”跳到了“唯一性”。
- 文档把“自建运行时=重复造轮子”作为核心论证（v3:103-105），但没有比较“多运行时适配层”或“OpenClaw-first + 可迁移内核”的总成本曲线。
- “不教用户做事，给他工具”作为哲学是清晰的（v3:53-57），但在高风险场景会失效：当工具能力边界、数据权限、错误后果不可忽略时，用户仍需要最小说明和确认链路；否则“自解释”会和“可控性”冲突。

### 建议
- 把“OpenClaw First”从“逻辑必然”改写为“阶段性最优策略”，并补一页替代路径对比（单平台/多平台/可迁移内核）。
- 明确“自解释的失效条件”：高风险决策、付费 API、外部执行操作三类场景必须触发说明与确认。
- 给出平台解耦最小设计：提取内核（source discovery + extraction + compiler）与 OpenClaw adapter 分离。

## 维度 2: 用户旅程
### 判断: 不通过

### 论据
- 5 Phase 主流程完整，但缺少“冲突消解、超时降级、失败回滚、版本回退”这四个必需步骤（v3:166-199, 368-387, 426-450）。
- v3 依赖 `sessions_spawn` 子代理（v3:117, 566），但文档自己承认“OpenClaw 子代理能力仅部分确认”（v3:749）；关键中间产物传递和 Bash 可用性仍在“实现阶段验证”，说明可行性尚未闭环。
- 现有实现与 v3 架构存在代差：`skills/soul-extractor/scripts/extract.py` 仍是串行单主流程（extract.py:441-453），并未体现 v3 所述多子代理路由。
- “回炉改造=增量提取”是关键承诺（v3:437-447），但当前技术描述没有给出增量算法（diff 边界、知识失效判定、卡片级 provenance 合并策略），而现有提取脚本是全量提取路径。

### 建议
- 在用户旅程里新增显式 Phase：`冲突消解`、`失败降级`、`版本回退`。
- 给 Phase 3 增加“可行性闸门”：若子代理能力检测失败，自动退化到单代理串行模式。
- 定义增量提取协议：`baseline snapshot + change set + affected cards + recompile scope`，并先在 1 个领域做回归验证。

## 维度 3: 技术架构
### 判断: 存疑

### 论据
- “确定性编排 + 子代理”是合理方案，但不是已证明最优解（v3:303-310）。在 MVP 阶段，单代理 + 确定性工具链也可能更低复杂度、更易观测。
- Knowledge Compiler 的价值被研究支持（v3:307, AI 消费研究:22-24, 109），但“必须有”与“可后置”的边界未定义：若去掉它，哪些质量指标会下降多少没有量化。
- Soul Lego Bricks 的“可预提取稳定”只对 WHY 层较成立，UNSAID 层衰减更快（积木研究:31-35, 189-190）；v3 将其整体性表述为稳定锚点，存在泛化过度。
- “这是调度问题，不是预算问题”（v3:628）过强。预算依然是硬约束，尤其在大型仓库里“代码片段 ≤20K”如何选取仍未回答（v3:630）。

### 建议
- 把架构论断改成“分阶段最优”：MVP 用单代理/小并发验证，达标后再扩子代理编排。
- 给 Knowledge Compiler 制定去留实验：有/无编译层 A/B，对比任务成功率、幻觉率、token 成本。
- 积木分层保鲜：WHY 年级更新、UNSAID 季度更新，并强制版本标签参与路由。
- 增补代码片段选择策略：热力图 + 入口调用链 + 最近变更优先。

## 维度 4: 质量保证体系
### 判断: 不通过

### 论据
- 三层幻觉防护目前是结构性主张，不是实证性结论（v3:487-494）。文档未给每层独立贡献度，也未给组合后误差上限。
- A/B/C/D 置信度定义可读但不可操作到工程层面：缺少判定流程、冲突仲裁、标注一致性检查（v3:495-500）。
- “事实验证通过”标准不足。`repo_facts.json` 白名单只覆盖命令/skills/文件/配置键（extract_repo_facts.py:134-140），无法覆盖语义事实、行为因果、性能断言。
- v3 与现有实现存在标准不一致：现有校验仍以 0-1 `confidence` 为主（validate_extraction.py:138-149），与 v3 的 A/B/C/D 分级尚未对齐。

### 建议
- 建立可证伪 QA 指标：`幻觉率`、`证据覆盖率`、`规则可执行率`、`端到端任务成功率`，并设置发布阈值。
- 为 A/B/C/D 设计判定流水线：证据类型、最低证据数、冲突处理与人工复核条件。
- 扩展事实验证到“语义级”：API 形参、返回约束、依赖版本、失败模式。

## 维度 5: 竞争分析
### 判断: 存疑

### 论据
- “WHY + UNSAID = 行业空白”是强判断（v3:675-687），但当前竞品样本偏窄，且缺少持续更新机制；结论更像“当前样本内空白”，不是“全市场空白”。
- 飞轮链路中最脆弱环节是“更多使用 -> 更多有效反馈 -> 提取精度提升”（v3:691-700）。非技术用户反馈往往是结果导向而非可训练信号，未必能直接提升提取器。
- 若 LLM 基础能力持续提升，纯“提取与叙事”价值会被侵蚀；Doramagic 的长期价值将更依赖“可验证知识资产、真实运行集成、反馈数据闭环”，文档对此防御还不够具体。

### 建议
- 把竞争结论从“行业空白”降级为“现阶段差异化优势”，并建立季度竞品复盘机制。
- 在飞轮里加入“反馈可用性过滤层”（可训练反馈比例、噪声率）并独立度量。
- 护城河叙事从“模型能力差”转向“资产与流程差”：证据图谱、可追溯验证、真实工具执行历史。

## 维度 6: 被忽略的风险
### 判断: 不通过

### 论据
- 最大盲区是“评估设计缺位”：文档写了很多机制，但缺少一套能证明“机制真的带来用户价值”的统一实验协议。
- 多个关键前提未验证：
  - OpenClaw 关键能力在目标版本稳定可用（v3:749 仍是部分确认）。
  - 引用的核心难点文档在仓库中不可复现（v3 多处引用 `memory/doramagic-5-challenges.md`，当前仓库未找到）。
  - 增量提取可在不污染旧知识的前提下稳定更新。
  - 非技术用户能正确判断“道具质量是否可用”。
- 风险讨论偏技术，缺少合规与法律层：开源知识二次编译后的许可边界、外部数据源 ToS、领域高风险输出责任划分。

### 建议
- 把“评估协议”升级为 P0：先定义 benchmark、评分规则、失败阈值，再推进架构迭代。
- 增加“证据可复现”门槛：所有核心结论必须可追溯到仓库内可读文件。
- 增补合规与责任模型：许可审计、数据源合规、风险场景拒答与人工接管机制。

## 最重要的一个批评
v3 目前最大的问题不是“想法不够好”，而是**把“合理机制”当成了“已验证机制”**：核心结论大量建立在概念闭环上，而非实证闭环。没有可复现评估协议，就无法证明“WHY/UNSAID 提取 + 编译 + 置信度透传”真的在真实用户任务中持续优于更简单方案。

## 建议的优先改进清单
1. 先建立统一评估基线（离线 benchmark + 在线任务成功率 + 幻觉率 + 成本/时延），作为所有架构决策的唯一裁判。
2. 为 Phase 3 建立“双模执行”与降级策略（子代理失败时自动退化串行），避免平台能力不确定导致全链路失效。
3. 把置信度体系工程化：A/B/C/D 判定规则、证据最小集合、冲突仲裁与人工复核触发条件。
4. 明确定义增量提取协议（差分检测、受影响卡片重算、版本回退），并在一个领域做端到端回归。
5. 重写 OpenClaw First 论证为“阶段性最优”，补充可迁移内核与多平台路径的成本对比。
6. 扩展风险面到合规和责任：开源许可、数据源条款、高风险建议的安全闸门。
