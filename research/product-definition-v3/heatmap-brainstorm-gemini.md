# 依赖热力图头脑风暴：全景脉络与动态抄袭优先级

这是基于"抄作业"核心目标的头脑风暴记录。发散优先，关注那些能改变我们对代码和知识理解视角的独特维度。

## 1. 维度发现：抄作业的雷达 (超越引用计数)

除了谁引用谁，代码本身还散发着很多关于"我重不重要"的信号。

*   **[常规][直接] Git 变更频率 (Git Blame / Churn)**：频繁变动的代码，说明设计决策仍在演化，坑多且深，是提炼"为什么这么改" (WHY) 的黄金区。
*   **[常规][直接] 注释与 TODO 密度**：作者觉得需要写大段注释解释的地方，或者留下 TODO/HACK 的地方，正是"代码无法自解释"的复杂业务逻辑或隐性知识的藏身处。
*   **[有趣][直接] 圈复杂度与决策分支密度 (AST 分析)**：If-else 或 Switch 的森林。决策点越多，意味着业务规则和边界情况处理越密集。
*   **[有趣][直接] "防线"厚度 (错误处理与断言密度)**：抛出自定义异常种类极多、Try/Catch 密集的模块，或者单元测试中断言 (Assert) 密集的模块，往往是系统最脆弱或最重要的"契约"地带。
*   **[疯狂][间接] 命名熵 (Naming Entropy)**：`calc()` vs `calculate_risk_adjusted_return_with_slippage()`。作者在变量/方法命名上耗费的脑力 (长度、词汇独特性) 往往与该模块在业务上的核心程度正相关。
*   **[疯狂][间接] 纯文本压缩率**：对单个文件进行通用算法压缩。压缩率低说明代码模式重复少、信息量/逻辑密度大；压缩率高说明可能是一大堆模板化、低价值的 Boilerplate (如纯数据类或重复配置)。

## 2. 跨项目信号：上帝视角的作业本

Doramagic 可以跳出一个项目的局限，通过全网开源数据建立"知识星座"。

*   **[有趣][直接] 黄金共生链 (Ecosystem Symbiosis)**：发现常在一起搭配使用的框架/库。如发现目标项目用了 `FastAPI` + `Pydantic`，即便它没有显式地把 `SQLAlchemy` 写在核心依赖里，我们也大概率知道它所处的架构生态位，能直接挂载相关的"全家桶积木"。
*   **[有趣][直接] 情绪风向标 (Issue Label Sentiment)**：分析一个库 GitHub Issues 的标签分布。全是 `bug` 标签的库，重点提取经验避坑知识；全是 `question` 标签的库，说明文档极差，重点提取操作步骤。
*   **[疯狂][直接] 迁移轨迹向量 (Migration Vectors)**：监控开源社区的架构迁移潮 (如 `Flask -> FastAPI` 或 `Webpack -> Vite`)。这些迁移相关的 PR 和讨论，是提炼"架构变迁的 WHY"和"选型对比的 DECISION"的最强高信噪比数据。

## 3. 备件库 (积木) 设计：活的知识生态

积木不是死板的说明书，应该是能够互相反应的化学物质。

*   **[有趣][直接] 场景组合图谱 (Scenario Graph)**：积木不是扁平的列表，而是带边的图。节点间存在：`推荐组合` (Django + Celery)、`互斥方案` (SQLAlchemy vs Django ORM)。当抽取到某个特征时，顺藤摸瓜点亮相关子图。
*   **[有趣][间接] 知识保鲜期 (TTL Metadata)**：积木自带生命周期。React Hooks 积木的 TTL 可能是半年，Unix 哲学的 TTL 是一万年。过期积木优先触发降权或更新流程。
*   **[疯狂][直接] "反面教材"积木 (Anti-pattern Bricks)**：不仅记录正确做法，更记录灾难做法。用于在扫描项目时进行"反向雷达匹配"。如果发现项目里出现了这种反模式，就可以当作鲜活的案例 (WAR STORIES) 提取出来教育用户。

## 4. 隐喻与类比的启示

*   **[有趣][间接] 基因转录因子 (生物学类比)**：有些接口类或基础配置文件，自身很少被调用，但就像转录因子一样，控制着下游海量模块的形态。热力图应该识别并高亮这些系统的"基因开关"。
*   **[疯狂][直接] 边际收益递减 (经济学类比)**：如果一个项目中包含几十个结构高度相似的 CRUD 模块。一旦前几个文件被提取，剩下的文件对知识增量的贡献急剧下降。热力图需要有"视觉疲劳机制"，当识别到重复模式时，主动拉低剩余相似模块的权重，节约 Token 预算。

## 5. 反直觉的发现

*   **[有趣][直接] 最少引用的文件反而可能最重要**：如 `ARCHITECTURE.md`、或者定义抽象接口和协议规范的纯基类文件。它们在代码执行层面上是"冷"的，但在心智模型构建层面上是"最热"的。
*   **[有趣][待验证] 明星项目的诅咒**：去抄 GitHub 8 万 Star 项目的作业，可能抄到一堆为了兼容极其罕见边缘情况而产生的冗长代码 (妥协的产物)。反而那些 3000 Star，专注单一垂直场景的小而美项目，其业务逻辑和设计哲学更加纯粹、易于提取。
*   **[疯狂][待验证] 热力图不应止步于 Stage 0**：热力图不应该只是一次性的静态产物。Stage 1 的 Agent (Opus) 在阅读少量核心代码后，可能会发现："咦，这个之前静态分析认为不重要的小模块，其实承载了最核心的状态机逻辑"。它应该有权**动态修改后续阶段的热力图分布**。

## 6. 与其他组件的神奇协同

*   **[有趣][直接] 热力图 × 积木 = 偏离地图 (Deviation Map)**：积木代表了框架的"标准作业流程"，热力图和代码代表了项目的"实际解法"。两者的 Diff，就是该项目最独特的创新或妥协之处，这才是提取的真正黄金区。
*   **[疯狂][直接] 热力图 × 用户的当前意图 = 聚光灯效应**：在需求挖掘阶段 (Phase 1)，如果用户说"我要一个极度注重**性能**的工具"，那么热力图在生成时，立刻给项目中的并发控制、缓存管理、C 扩展模块等施加巨大权重乘数，实现基于意图的"偏心"抄作业。
*   **[有趣][直接] 热力图 × Knowledge Compiler = 预算救星**：当最终提取的内容爆炸，面临 Token 预算硬约束时。Knowledge Compiler 直接根据源知识所在模块的原始热力图分值进行优先级排序，低分的知识点被无情压缩或剪裁。
