# 道具更新机制研究：如何让用户便捷更新 Doramagic 生成的工具

## 背景

Doramagic 是运行在 OpenClaw 上的 AI 道具制造器。它本身是一个 OpenClaw Skill，它生成的道具也是 OpenClaw Skill。

**核心产品哲学：不教用户做事，给他工具。**

Doramagic 是一个新产品，迭代会很快——提取引擎升级、积木仓库扩展、Knowledge Compiler 改进、新的证据验证机制上线。每次 Doramagic 自身升级后，之前生成的道具不会自动变好。

## 问题

存在两个层面的更新需求：

### 1. Doramagic 自身的更新
Doramagic 作为 OpenClaw Skill 的更新。这相对简单，走 OpenClaw 平台的 Skill 更新机制即可。

### 2. Doramagic 生成的道具的更新（核心问题）
- Doramagic 提取引擎升级了，旧道具应该受益
- 积木仓库扩展了，旧道具可以获得更好的框架知识锚定
- Knowledge Compiler 改进了，旧道具的知识格式可以优化
- 用户使用过程中积累了反馈（MEMORY.md），重新锻造可以利用这些反馈

### 与"回炉改造"的区别
- **回炉改造**：用户主动想改功能（"帮我加港股支持"）
- **版本更新**：Doramagic 变强了，旧道具应该用新能力重新锻造

## 约束条件

1. **不教用户做事**：用户不应该需要了解 Doramagic 的内部版本或技术升级细节
2. **OpenClaw 原生**：方案必须在 OpenClaw 平台的能力范围内实现
3. **用户控制权**：更新不能自动发生（道具可能已被用户自定义），必须用户确认
4. **增量优先**：不是每次都从零重新提取，尽可能利用已有的知识资产

## 你需要回答的问题

请从**第一性原理**出发，提供**具体的解决方案**：

1. **道具版本管理机制**：Doramagic 如何追踪它生成的每个道具（版本号、锻造时间、使用的引擎版本、积木版本、知识来源）？这些元数据存在哪里？

2. **更新触发机制**：什么时候提示用户道具可以更新？谁来判断"旧道具应该更新了"？是 Doramagic 主动检查还是被动等待？

3. **更新执行策略**：更新时是全量重新锻造还是增量更新？如何保留用户在使用中积累的个性化数据（MEMORY.md 中的反馈）？

4. **用户体验设计**：更新提示怎么呈现？用户如何一键更新？更新前后的差异如何展示？如何处理用户拒绝更新的情况？

5. **参考其他产品的更新机制**：iOS App Store 的更新提示、Chrome 扩展的自动更新、VS Code 扩展的更新——哪些模式可以借鉴？哪些不适用于 AI 道具？

6. **风险**：更新后道具行为变化导致用户不适应怎么办？能否回滚？
