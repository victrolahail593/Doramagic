# 置信度体系设计研究：从"打分"到"举证"？

## 背景

Doramagic 是一个 AI 知识提取引擎，从开源项目中提取 6 层知识（事实/操作/经验/决策/设计哲学/隐性知识），编译成 AI 可消费的格式注入 OpenClaw Skill。

核心架构原则：**代码说事实，AI 说故事**
- 事实层：确定性脚本提取（文件结构、依赖、API 签名），零模型自由度（AI 不许编）
- 解读层：LLM 生成（设计哲学、心智模型、决策理由），有约束的模型自由度（AI 可以解读但不能瞎编）

**硬性约束：所有设计必须从第一性原理推导，不依赖类比或惯例。**

## 问题

在 v3 产品定义中，我们设计了置信度 A/B/C/D 分级：
- A：创作者明确说明（README/文档/注释）
- B：社区证据支持（Issues/讨论/博客）
- C：代码模式推断（代码结构推断但无外部证据）
- D：纯 LLM 推理（模型自行推断）

Codex 在审查中判"不通过"，批评是：**定义可读但不可操作——缺少判定流程、冲突仲裁、标注一致性检查。**

更根本的问题是：**让 LLM 给自己的输出打置信度，就像让学生给自己的考试打分——不知道自己错的人不会给自己打低分。** LLM 对自己的幻觉没有自知之明。

## 我们的替代方案草案

从"打分"变成"举证"——不评"有多可信"，而是标"有什么证据"：

```
当前方案（主观评级）：
  "Django 用 ORM 是为了可移植性" → confidence: B

替代方案（证据标签）：
  "Django 用 ORM 是为了可移植性"
    → [DOC] Django FAQ 中提到
    → [COMMUNITY] Stack Overflow 3 个讨论
    → [CODE] settings.py 支持多数据库切换
```

标签类型：
- [CODE] — 代码中直接可观察
- [DOC] — 项目文档明确说明
- [COMMUNITY] — 社区讨论中提到
- [INFERENCE] — LLM 从代码模式推断，无外部证据

Knowledge Compiler 用证据标签做过滤：有 CODE/DOC 的进能力清单，有 COMMUNITY 的进踩坑故事，只有 INFERENCE 的降权或标注。

## 你需要回答的问题

请从**第一性原理**出发，提供**具体的解决方案**（不只是批评）：

1. **A/B/C/D 分级 vs 证据标签，哪个方案更好？为什么？** 从信息论、LLM 行为特性、工程可行性三个角度分别论证。

2. **你是否有第三种方案？** 不局限于我们提出的两个选项。如果你认为有更好的方式来解决"LLM 提取的知识可信度如何表达"这个问题，请详细描述。

3. **无论选哪种方案，给出可操作的实现设计：**
   - 判定流程：谁标注？怎么标注？标注依据是什么？
   - 冲突处理：同一条知识从不同源得到不同标注怎么办？
   - 消费端处理：Knowledge Compiler 和最终 Skill 如何利用这些标注？
   - 验证方法：如何验证标注系统本身的准确性？

4. **这个系统在不同规模项目中表现是否一致？** 高 Star 项目（LLM 预训练数据中大量存在，可能"记忆污染"）vs 低 Star 项目（LLM 几乎没见过，更依赖代码推理）——同一套方案能否适用？
