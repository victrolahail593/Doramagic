# 依赖热力图头脑风暴：如何让 Doramagic 更好地"抄作业"

## 背景

Doramagic 是一个 AI 道具制造器。它的核心能力是**抄作业**——从开源项目中提取智慧，组装成用户可直接使用的 AI 工具。

**关键认知**：Doramagic 不是知识的创造者，是知识的**组装者**。就像装配线上的熟练工——每个零件都不是他造的，但他知道怎么拼成产品。LLM 的知识可以认为是无限的，但散落各处；用户的认知是有限的。Doramagic 的价值 = 把 LLM 散落的知识正确组装成用户能用的道具。

**积木（Soul Lego Bricks）**：预提取的框架级知识。积木的价值不是"补充 LLM 不知道的"，而是**让 LLM 的知识被正确组装**——锚定、排序、去噪、格式化。

## 当前的"依赖热力图"概念

我们在 v3 产品定义中提出了一个概念：**依赖热力图**——在 Stage 0（确定性分析阶段）产出一张地图，告诉后续的 AI 提取引擎"哪里值得深挖、哪里可以跳过"。

当前版本非常粗糙，只用了引用计数：

```json
{
  "heatmap": [
    {"node": "core.api.authenticate()", "references": 14, "type": "integration_hub"},
    {"node": "utils.cache.get()", "references": 8, "type": "infrastructure"},
    {"node": "helpers.format_date()", "references": 2, "type": "utility"}
  ]
}
```

这远远不够。

## 核心洞察：脉络

项目创始人 Tang 对"脉络"的理解：**足够细小单元组成的逻辑关系**。

举例：
- 一个金融 API（如 akshare）被 30 个项目依赖 → 这 30 个项目共享一套金融数据处理的共识知识
- Django 被 60 万个项目使用 → ORM 的设计哲学经过海量验证
- 全世界都在用的东西，我们用大概率不会犯错

所以"脉络"不只是一个项目内部的代码依赖图，它至少有两层：
1. **项目内部脉络**：一个项目内部，哪些模块是核心、哪些是配角
2. **跨项目脉络**：整个开源世界中，哪些知识/模式/库是被广泛验证的共识

两者共同服务于一个目标：**更好地抄作业**。

## 你的任务：发散性头脑风暴

请你**尽情发挥想象力**，围绕以下大方向进行**发散性探索**。不需要收敛到一个方案，不需要考虑实现难度，先把可能性空间打开。

### 大方向：热力图应该是什么？

热力图本质上是一张**"抄作业优先级地图"**。面前有一份作业（一个开源项目），哪些部分是精华要仔细抄？哪些部分是套路可以快速过？哪些部分是这份作业独特的创新要重点学？

请从以下角度发散思考（但不限于这些角度）：

1. **维度发现**：除了"引用计数"，还有哪些维度可以衡量代码/知识的"值得提取程度"？越多越好，越意外越好。想想其他领域有没有类似的优先级排序问题。

2. **跨项目信号**：如果 Doramagic 能获取跨项目的信息（比如 GitHub 的依赖图、npm/pip 的下载量、Stack Overflow 的问答频率），这些信号怎么用？能用来做什么以前做不到的事？

3. **备件库设计**：积木仓库（预提取的框架知识）应该怎么设计？什么决定积木的优先级？积木之间有没有"脉络"（逻辑关系）？积木仓库本身是一个扁平列表还是一个有结构的图？

4. **热力图与知识密度**：我们定义了"知识密度"这个概念——某段代码/某个模块背后蕴含的设计决策数量和独特性。有没有办法用确定性方法（不依赖 LLM）估算知识密度？

5. **隐喻和类比**：从其他领域借灵感。城市规划中的核心商圈和居民区？生物学中的基因调控网络？供应链管理中的关键零件？学术论文的引用网络？这些类比能给热力图带来什么启发？

6. **反直觉想法**：有没有什么违反常识但可能正确的想法？比如"最不被引用的代码反而最值得提取"？或者"热力图根本不应该在 Stage 0 做"？

7. **与 Doramagic 其他组件的交互**：热力图怎么跟积木系统、置信度系统、Knowledge Compiler、子代理架构互动？有没有意想不到的协同效应？

8. **动态性**：热力图是一次性产出还是会演化？用户使用道具的反馈能否反哺热力图？
