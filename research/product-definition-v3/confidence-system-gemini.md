# 置信度体系设计研究：从"主观打分"到"客观举证"的范式转移

## 1. 推荐方案：证据链标签 (Evidence-Chain Tagging, ECT)

**明确选择：证据标签方案。** 

不应让 LLM 充当"法官"（判定 A/B/C/D），而应让其充当"检察官"（提交证据链）。最终的置信度判定权应由 **Knowledge Compiler (确定性逻辑)** 掌握，或者透传给用户做最后的价值判断。

---

## 2. 推荐理由：第一性原理分析

### 2.1 信息论视角：减少熵增
*   **A/B/C/D 分级**：将复杂的证据关系压缩为一个标量（等级），这是一种**有损压缩**。LLM 在压缩过程中会引入自身的主观偏见（幻觉），导致信息的熵增加。
*   **证据标签 (ECT)**：保持了证据的**原始分布**。`[DOC]`、`[CODE]`、`[COMMUNITY]` 是离散的维度，它们描述了知识的**来源多样性**。保留来源信息意味着保留了信息的"真值溯源"能力，减少了虚假信息的混入。

### 2.2 LLM 行为特性：对抗"过度自信" (Overconfidence)
*   LLM 擅长**关联推理**，但不擅长**自我评价**。要求 LLM 打分会触发其"顺从性"（总是倾向于打高分以显得专业）。
*   要求 LLM **"举证"**（检索并引用特定代码行、文档段落或社区链接）是一种**强制性的慢思考 (System 2 Thinking)**。如果 LLM 无法提供具体的代码行号或文档引用，它就无法打上 `[CODE]` 或 `[DOC]` 标签。这种约束将 LLM 的行为从"自由创作"拉回到"证据检索"。

### 2.3 工程可行性：确定性判定的基础
*   等级分 (A/B/C/D) 难以进行冲突仲裁（两个模型打分不一致怎么办？）。
*   证据标签可以进行**布尔代数运算**。
    *   `[CODE] AND [DOC]` = 核心真相 (Core Truth)
    *   `[COMMUNITY] AND NOT [DOC]` = 隐性知识 (Unsaid Knowledge)
    *   `[INFERENCE] ONLY` = 实验性推论 (Experimental Inference)
    这种逻辑操作是确定性脚本（Knowledge Compiler）最擅长的。

## 3. 完整实现设计

### 3.1 判定流程：检察官与法官模式
1.  **Stage 1-3 (提取 Agent - 检察官)**：
    *   在提取每一条知识卡片时，必须附带 `evidence_pool`。
    *   `evidence_pool` 包含：代码片段索引、文档引用 URL/文本、社区讨论摘要。
    *   Agent 根据 `evidence_pool` 的内容打上初步标签。
2.  **Stage 3.5 (确定性验证 - 形式审查)**：
    *   脚本验证 `[CODE]` 标签对应的文件路径是否存在，代码片段是否真的在 `repo_facts.json` 中。
    *   脚本验证 `[DOC]` 标签引用的文件是否在项目目录中。
3.  **Knowledge Compiler (最终裁定 - 法官)**：
    *   根据标签的组合逻辑，计算最终的置信度权重。

### 3.2 冲突处理：证据叠加原则
当同一条知识从不同源或不同模型得到不同标注时：
*   **非竞争性冲突**：如果 A 模型找到 `[CODE]`，B 模型找到 `[DOC]`，最终结果为 `[CODE, DOC]`（取并集，增强置信度）。
*   **竞争性冲突**（如：A 说是 `[CODE]`，B 说是 `[INFERENCE]` 且认为代码中没有）：触发 **"反证 Agent"**。专门指派一个子代理寻找该知识的**证伪证据 (Counter-evidence)**。如果找到证伪证据，该知识标记为 `[CONTROVERSIAL]` 或直接丢弃。

### 3.3 消费端处理：按需路由
*   **`[CODE/DOC]` 级**：作为 `CRITICAL RULES` 或 `API CAPABILITY` 写入 `SKILL.md` 的核心部分。
*   **`[COMMUNITY]` 级**：作为 `WAR STORIES` 或 `TIPS` 写入，并注明 "社区普遍反馈"。
*   **`[INFERENCE]` 级**：仅作为注释出现在代码旁，或在 `PHILOSOPHY` 章节中以 "AI 推测的设计意图" 形式呈现。

### 3.4 验证方法：证据一致性抽检
*   **自动化测试**：提供 10% 的负样本（故意篡改的虚假代码/文档）。如果系统仍然打上 `[CODE/DOC]` 标签，说明举证系统失效。
*   **用户反馈闭环**：生成的道具中，每条知识后附带一个 `(source)` 链接。用户点击可查看证据原文。用户纠错行为直接降低该类证据的权重。

## 4. 跨项目规模的适应性分析

### 4.1 高 Star 项目：警惕"记忆寄生"
*   **现象**：LLM 可能会凭记忆说出 `[DOC]` 内容，即使当前项目版本已删除该文档。
*   **对策**：强化 `[CODE]` 的权重。对于高 Star 项目，强制要求 `[CODE]` 与 `[DOC]` 必须同时存在且互不矛盾。

### 4.2 低 Star 项目：依赖"推理验证"
*   **现象**：完全没有 `[DOC]` 和 `[COMMUNITY]`。
*   **对策**：启用 **"模式匹配积木 (Lego Bricks)"**。虽然项目本身没被 LLM 见过，但其使用的模式（如标准的 Django 目录结构）是被见过的。标签演变为 `[PATTERN_MATCH]` + `[CODE]`。

---

## 5. 风险与局限
*   **举证成本**：要求 LLM 详细举证会增加 Token 消耗和响应延迟。
*   **证据误植**：LLM 可能会为了证明自己的结论而张冠李戴（将 A 文件的代码作为 B 结论的证据）。这需要 Stage 3.5 的**确定性脚本**进行严丝合缝的校验。

---

## 6. 一句话总结

**置信度不应由 AI "感觉"出来，而应由 AI 从代码、文档和社区中"挖掘"出来的证据链条编织而成。**
