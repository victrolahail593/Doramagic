# 依赖热力图头脑风暴 — Claude

## 1. 维度发现——"值得抄"的信号

### 常规维度
- **引用计数**（已有）— 被多处 import/调用
- **git blame 变更频率** — 频繁改的代码说明设计决策还在演化，WHY 在变
- **注释密度** — 作者认为需要解释的地方，知识密度高的间接信号
- **文件大小与圈复杂度** — 复杂代码背后的设计决策更多

### 有趣维度
- **"偏离框架默认"的程度**（新颖度：有趣 / 关联度：直接）
  - 如果积木说 Django 默认用 `ModelForm`，但项目自己写了 `FormFactory`，偏离本身就是高价值提取目标
  - 偏离 = 项目独特的设计决策 = 精华
  - 可确定性检测：拿积木的"标准做法列表"跟项目实际代码做 diff
  - **与积木系统的协同**：积木不只是提取的锚点，还是偏离检测的基准线

- **"被测试覆盖的密度"**（新颖度：有趣 / 关联度：直接）
  - 作者写了大量测试的模块 = 他认为这里容易出错、行为复杂、值得保护
  - 测试密度 ∝ 知识密度
  - 确定性可计算：统计 test 文件中对各模块的引用频率

- **"配置表面积"**（新颖度：有趣 / 关联度：直接）
  - 一个模块暴露了多少可配置项（环境变量、配置文件参数、命令行参数）
  - 高配置表面积 = 高决策密度（用户需要做很多选择 = 需要 Doramagic 替用户做判断）

- **"错误处理密度"**（新颖度：有趣 / 关联度：直接）
  - try/catch、error code、retry 逻辑的密度
  - 防御性代码多 = 作者知道这里有很多坑 = 踩坑经验的富矿

### 疯狂维度
- **"命名努力"**（新颖度：疯狂 / 关联度：间接）
  - 变量名/函数名的平均长度和描述性
  - `calculate_risk_adjusted_portfolio_return()` vs `calc()`
  - 作者花心思命名的地方 = 他认为这个概念重要到值得用精确的名字

- **"TODO/FIXME/HACK 密度"**（新颖度：疯狂 / 关联度：直接）
  - 这些注释是作者承认"这里有问题但我知道"的信号
  - HACK = 隐性知识的直接表达

- **"接口-实现比"**（新颖度：疯狂 / 关联度：间接）
  - 公共接口（exported functions/classes）vs 内部实现的比值
  - 高比值 = 被设计为供外部使用的抽象 = 设计决策更有意义


## 2. 跨项目信号——全新的可能

### 常规
- GitHub dependents 数量
- PyPI/npm 下载量
- Stack Overflow 标签频率

### 有趣
- **"组合热力图"**（新颖度：有趣 / 关联度：直接）
  - 不只看单个库的使用量，看**库的组合**的使用频率
  - Django + Celery + Redis 是一个被验证的组合模式
  - 如果用户的项目用了 Django + Celery，Doramagic 知道 Redis 大概率在栈里
  - 超越单库维度，进入**技术栈模式**层面

- **"Issues 的知识信号"**（新颖度：有趣 / 关联度：直接）
  - `bug` 标签多 = 坑多 = 踩坑经验价值高
  - `question` 标签多 = 文档不够清晰 = Doramagic 可以补充的空白

- **"迁移信号"**（新颖度：有趣 / 关联度：间接）
  - 从库 A 迁移到库 B 的讨论频率
  - 迁移路径本身是高价值知识

### 疯狂
- **"反向依赖知识图"**（新颖度：疯狂 / 关联度：直接）
  - 不是"这个库依赖谁"，是"谁依赖这个库，他们怎么用它"
  - 30 个项目都依赖 akshare，分析 import 模式：都用了 `stock_zh_a_spot_em()` 但没人用 `stock_hk_spot()`？
  - 前者 = 经过验证的用法，后者 = 可能有坑

- **"Star 增长曲线的导数"**（新颖度：疯狂 / 关联度：间接）
  - 不是看绝对 star 数，而是增长速率
  - 快速增长 = 社区正在发现价值 = 提前做积木的窗口期


## 3. 备件库（积木仓库）设计

### 有趣
- **积木之间的"组合图谱"**（新颖度：有趣 / 关联度：直接）
  - 积木不是扁平列表，而是有边的图
  - django-core → django-orm 是"包含"关系
  - django-orm → sqlalchemy-core 是"替代"关系
  - django + celery 是"组合"关系
  - 当用户需求涉及 Django + Celery 时，知道要加载哪些积木组合

- **"消耗频率驱动的库存管理"**（新颖度：有趣 / 关联度：直接）
  - 借鉴供应链 ABC 分析法
  - A 类（高频）：django, react, express → 优先制作、优先更新
  - B 类（中频）：celery, redis, sqlalchemy → 按需制作
  - C 类（低频/长尾）：不做积木，靠 LLM 通用知识 + 代码推理

- **"积木的保质期"**（新颖度：有趣 / 关联度：直接）
  - 知识衰减速度不同：React hooks 最佳实践每半年变；Unix 哲学几十年不变
  - 保质期写入积木元数据，驱动更新优先级

### 疯狂
- **"用户反馈驱动的积木进化"**（新颖度：疯狂 / 关联度：直接）
  - 道具使用中的问题反馈到积木里
  - 积木从"专家编写一次"变成"社区持续打磨"
  - 积木仓库变成活的知识库

- **"反积木"**（新颖度：疯狂 / 关联度：直接）
  - 不只记录"应该怎么做"，也记录"千万别这么做"
  - `django-antipatterns`：N+1 查询、循环里 `.save()`、`eval()` 动态查询等已知反模式
  - 直接服务于踩坑经验提取


## 4. 热力图与知识密度的确定性估算

### 有趣
- **"决策点计数"**（新颖度：有趣 / 关联度：直接）
  - if/switch/配置读取点数量
  - 每个条件分支 = 一个"作者做了选择"的信号
  - AST 确定性提取

- **"抽象层数"**（新颖度：有趣 / 关联度：间接）
  - 从入口到最终执行经过了多少层封装
  - 层数多 = 抽象设计复杂 = WHY 更值得提取

- **"接口稳定性"**（新颖度：有趣 / 关联度：间接）
  - 通过 git log 分析公共 API 变更频率
  - API 稳定 = 设计决策成熟
  - API 频繁变更 = 设计还在摸索 = WHY 在演化中

### 疯狂
- **"信息熵估算"**（新颖度：疯狂 / 关联度：待验证）
  - 把代码当文本，计算每个模块的信息熵
  - 高熵 = 高多样性 = 可能包含更多不同类型的逻辑
  - 低熵 = 重复模式 = 可能是 CRUD boilerplate
  - 纯数学方法，零 LLM


## 5. 类比启发

- **学术引用网络** → 高被引论文 = 领域基石。但最有突破性的论文往往初期被引不高。对应：被广泛使用的库 vs 新兴但有突破性的库（新颖度：常规 / 关联度：间接）

- **城市规划** → 核心商圈（高流量节点）需要最好的基础设施。但城市的灵魂往往在小巷子里（独特的文化）。对应：高引用模块需要基础积木，低引用但独特的模块需要深度 WHY 提取（新颖度：有趣 / 关联度：间接）

- **基因调控网络** → 转录因子（一个基因控制一大批下游基因的表达）。一个关键模块的设计哲学可能"调控"整个项目的行为模式。找到这些"转录因子"级的模块（新颖度：有趣 / 关联度：直接）

- **供应链** → 关键零件（缺了整条线停工）vs 可替换零件。对应：项目中不可替换的核心模块 vs 可以换成任何替代品的工具模块（新颖度：常规 / 关联度：直接）


## 6. 反直觉想法

- **"没人引用的代码可能最值得提取"**（新颖度：疯狂 / 关联度：直接）
  - `design_decisions.md` 或 `ARCHITECTURE.md` 从不被代码 import，但它是设计哲学的直接表达
  - 热力图不能只看代码引用，要看文档

- **"热力图不应该只在 Stage 0"**（新颖度：有趣 / 关联度：直接）
  - 初始热力图基于静态分析
  - Stage 1 的结果可能修改热力图："这个看起来不重要的 utils 模块其实承载了整个项目的错误处理哲学"
  - 动态更新 → Stage 2-3 调整优先级

- **"最好的作业可能不是 star 最高的"**（新颖度：有趣 / 关联度：直接）
  - star 最高的项目被 LLM 预训练学透了，存在记忆污染风险
  - 700 star 的项目可能有更干净的提取环境 + 更独特的设计决策


## 7. 与其他组件的协同

- **热力图 × 积木 = "偏离地图"** — 积木定义框架标准做法，热力图标出项目实际做法。两者的 diff = 项目独特之处 = 提取的黄金区域（新颖度：有趣 / 关联度：直接）

- **热力图 × 置信度 = 动态验证优先级** — 热力图高分但置信度低的知识 → 优先交叉验证（因为重要但不确定）（新颖度：有趣 / 关联度：直接）

- **热力图 × Knowledge Compiler = 智能压缩** — 热力图低分的知识在 token 预算紧张时优先被压缩/丢弃（新颖度：常规 / 关联度：直接）


## 8. 动态性

- **道具使用反馈 → 热力图修正** — 用户频繁使用道具的某个能力 = 对应知识高价值 → 回炉改造时提高该区域的热力图权重（新颖度：有趣 / 关联度：直接）

- **积木版本更新 → 热力图重算** — 新版积木改变了"框架标准做法"，偏离地图需要重新计算（新颖度：常规 / 关联度：直接）

- **跨用户聚合** — 100 个用户都让 Doramagic 做 Django 相关道具，使用数据聚合 = 最高价值的 Django 知识排行榜 → 反哺积木仓库（新颖度：疯狂 / 关联度：直接）
