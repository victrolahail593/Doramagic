## 二、核心理念

### 2.1 一句话定义

> **Doramagic = 超级 Copy Cat —— 站在巨人肩膀上快速抄作业**

### 2.2 核心理念

| 维度 | 说明 |
|------|------|
| **不是** | 帮用户从 0 开发一个功能 |
| **而是** | 从开源项目中找到成熟的解决方案，复制 + 调整 |
| **本质** | 用 AI 大模型替代人肉抄作业，自动完成知识提取和能力匹配 |

### 2.3 初心追溯

在与 Tangsir 的讨论中，我们进行了深刻的第一性原理拷问：

> **「没有 Doramagic，用户会死吗？」**  
> **「有了 Claude，用户为什么还需要你？」**

**Tangsir 的回答**：

- 用 SupPower 时觉得好用，但不适合自己
- 以前只能忍受，AI 时代可以调整
- 长期记忆技术是这种范式的典型

这揭示了核心洞察：**不是「好」，而是「适合」**。

## 三、范式转变

### 3.1 核心洞察：「好」vs「适合」

| 之前（工具时代） | 现在（AI 时代） |
|------------------|------------------|
| 用户适应工具 | 工具适应用户 |
| 忍受不喜欢的功能 | AI 动态调整 |
| 学习成本：高 | 使用成本：零 |

### 3.2 范式对比

| 旧范式 | 新范式（Doramagic） |
|--------|---------------------|
| 用户适应工具 | **工具适应用户** |
| 学习技能才能用 | **直接给成品** |
| 忍受不喜欢的功能 | **AI 动态调整到适合** |
| 搜索 → 学习 → 配置 → 使用 | **需求 → 成品** |

### 3.3 竞品差异化

| 产品 | 定位 | 比喻 |
|------|------|------|
| **Cursor/Copilot** | 建筑师 | 从 0 开始构建 |
| **Doramagic** | 抄作业专家 | 复制 + 调整 |
| **SupPower** | 工具集合 | 准备好一堆工具，用户自己选 |

## 四、产品定位

### 4.1 定位声明

**Doramagic 是 AI 时代的「抄作业专家」—— 让用户快速获得能用的成品，而不是教用户开发技能。**

### 4.2 目标用户

| 用户类型 | 特征 | 需求 |
|----------|------|------|
| **Vibe Coding 新手** | 用 Cursor/Claude Code 写代码 | 快速解决「配置问题/版本 bug/踩坑」 |
| **效率至上者** | 不想学只想用 | 直接要成品，不要学习成本 |
| **跨领域探索者** | 多个领域浅尝辄止 | 需要快速进入新领域，不深入学习 |

### 4.3 核心价值

| 价值点 | 说明 |
|--------|------|
| **零学习成本** | 不用说「教我开发」，直接说「我要」 |
| **个性化适配** | 记住用户的「适合」，动态调整 |
| **踩坑经验** | 整合 GitHub Issues/Discussions 的真实踩坑记录 |
| **快速动起来** | 不是给你渔具，是直接把鱼给你 |

## 五、三个核心挑战

### 5.1 挑战清单

| 挑战 | 优先级 | 说明 |
|------|--------|------|
| **噪音过滤** | ⭐⭐⭐ 最高 | 准确识别真正的知识和经验，是产品成立的前提 |
| **关联推理** | ⭐⭐ 很高 | 用户表达可能模糊/缺失，如何准确映射到开源项目 |
| **时效性** | ⭐ 次要 | 知识是沉淀的，不需要实时性 |

### 5.2 噪音过滤策略

> **本质问题**：开源社区充满噪音——重复问题、无效讨论、过时答案

**解决思路**：

1. **来源分级**：官方文档 > 官方 Issue > 社区讨论 > 第三方博客
2. **质量评分**：基于 upvotes、是否被官方采纳、解决方案验证
3. **LLM 过滤**：用 LLM 判断内容是否为「有效知识」

### 5.3 关联推理策略

> **本质问题**：用户说「我想管 Trello」→ 映射到哪个开源项目？哪个能力？

**解决思路**：

1. **置信度机制**：70% 阈值，高于直接执行，低于追问
2. **选择题优先**：不让用户自己想选项，而是给选项
3. **上下文匹配**：结合用户已有的 Skills、记忆、模型做个性化推荐

## 七、用户交互设计

### 7.1 自适应交互深度

> **核心洞察**：不是所有用户都需要深度交互

| 用户类型 | 需求清晰度 | 需要的交互 |
|----------|------------|------------|
| 能手型 | 说得很清楚 | 0 轮，直接进入选择 |
| 普通型 | 大概知道 | 1-2 轮选择题 |
| 新手型 | 说不清楚 | 3-5 轮追问 |

### 7.2 置信度机制

| 参数 | 值 | 说明 |
|------|-----|------|
| **阈值** | 70% | 行业标准（Dialogflow、LivePerson 均采用） |
| **高于阈值** | 直接呈现 | 进入能力积木阶段 |
| **低于阈值** | 选择题追问 | 最多 3 轮 |
| **达到上限** | 强制进入 | 提供保守方案 |

### 7.3 选择题设计原则

| 原则 | 实现 |
|------|------|
| **选择题优先** | 不让用户自己组织语言 |
| **最多 3 轮** | 超过后主动提供保守方案 |
| **提前终止** | 用户一旦选了就立刻推进 |
| **复述确认** | 「你是说 xxx 对吧？」形成闭环 |

### 7.4 完整交互流程

```
用户输入
    ↓
【第一阶段：自适应交互】
    ↓ 置信度判断
【第二阶段：意图理解 + 资源检索】← 子代理并行
    ↓
【第三阶段：能力积木呈现】
    ↓
【第四阶段：用户选择 + 配置】
    ↓
【第五阶段：Skill 生成 + 交付】
```

## 十、技术架构

### 10.1 架构选型历程

| 阶段 | 方案 | 问题 |
|------|------|------|
| **最初** | LangChain + Qdrant + BGE + Mem0 + Neo4j | 太重，不是 Skill |
| **最终** | 纯 LLM + OpenClaw Skills + 记忆文件 | 轻量、纯 Skill |

### 10.2 最终轻量架构

```
用户 → Doramagic Skill
         ↓
┌─────────────────────────────────────┐
│           纯 LLM 能力               │
│  • 意图理解（Prompt）               │
│  • Skill 规划（LLM）                │
│  • 上下文读取（记忆文件）            │
│  • 结果组装生成                     │
└─────────────────────────────────────┘
         ↓
   Skill 调用层
   ┌──────────────┐
   │ Skills        │ ← OpenClaw 已有
   │ LLM           │ ← OpenClaw 已配置
   │ 记忆          │ ← 记忆文件
   └──────────────┘
```

### 10.3 零外部依赖

| 资源 | 来源 |
|------|------|
| LLM | OpenClaw 已配置 |
| Skills | OpenClaw 已有 |
| 知识存储 | 记忆文件 |
| 灵魂提取 | 已有 soul-extractor Skill |

### 10.4 关键技术组件

| 组件 | 功能 | 技术方案 |
|------|------|----------|
| 意图识别 | 理解用户想要什么 | LLM + Prompt Engineering |
| 置信度评分 | 判断需求是否清晰 | 四维度加权模型 |
| 选择题生成 | 生成合理的选项 | LLM 生成 + 知识库检索 |
| 对话状态 | 管理多轮对话状态 | 内存状态机 |
| 灵魂提取 | 从开源项目提取知识 | soul-extractor Skill（已存在） |

### 10.5 置信度评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 实体完整度 | 35% | 用户输入中关键实体是否完整 |
| 意图匹配度 | 25% | LLM 对意图判断的确定性 |
| 模糊词检测 | 20% | 检测「随便」「都行」等模糊词 |
| 上下文连贯性 | 20% | 与历史对话的连贯程度 |

## 十一、长期记忆系统

### 11.1 核心方向

记住用户的「适合」—— 动态调整，而非固定选项。

### 11.2 记忆维度

| 维度 | 说明 | 示例 |
|------|------|------|
| 喜欢什么方式 | 交互风格偏好 | 简洁 vs 详细 |
| 跳过什么建议 | 不关注的选项 | 跳过配置建议 |
| 选择什么配置 | 最终决策模式 | 总是选高级选项 |
| 关注什么信息 | 注意力焦点 | 关注性能而非易用性 |

### 11.3 记忆工作原理

**用户 A 示例**：

1. 说「帮我做个 Trello 管理」
2. Doramagic 给出 4 个选项：A/B/C/D
3. 用户 A 选了「简洁模式」
4. → 长期记忆：**A 选择了简洁模式，不想要复杂配置**
5. 下次用户 A 说「帮我做个日历管理」
6. → Doramagic 直接默认给简洁模式，跳过复杂选项

**用户 B 示例**：

1. 每次都选 D（高级自定义）
2. → 长期记忆：**B 喜欢深度定制**
3. 下次给更多高级选项

### 11.4 技术实现（待细化）

- **存储**：OpenClaw 记忆文件
- **更新**：每次用户决策后更新
- **检索**：查询时加载用户偏好
- **融合**：在选择题生成时融入偏好

## 十二、用户场景

### 场景 1：告警规则不生效

| 方案 | 回答 |
|------|------|
| **通用 LLM** | 告诉你「在 alertmanager.yml 里加 group_by 参数」 |
| **Doramagic** | 代码层面：为什么要这么配 + 社区层面：GitHub Issue 里的 bug report、踩坑经验、最新 workaround |

- 用 Cursor/Claude Code 写代码
- 遇到问题和实际跑起来的结果不一样
- 需要快速定位是「配置问题」还是「版本 bug」还是「踩过的坑」

### 场景 3：跨领域探索

- 用户想做「每日任务提醒」
- 传统方式：搜索 → 学习 → 配置 → 使用
- Doramagic 方式：需求 → 成品（直接给一个可用的 Skill）
