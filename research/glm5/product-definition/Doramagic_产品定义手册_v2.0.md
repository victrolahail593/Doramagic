# Doramagic 产品定义手册

> 版本：v2.0 | 日期：2026-03-11 | 状态：讨论中

---

## 一、核心理念

**不是「帮你做」，而是「帮抄作业」** —— 快速动起来。

Doramagic 的核心能力是**提取灵魂**：
- 洞察开源项目做了什么
- 为什么要做
- 终局要解决什么问题

**学习引擎从人类变成 AI 大模型** —— 用 AI 去学习和提炼，而不是靠人肉抄作业。

---

## 二、产品定位

### 旧版（产品手册 v1.0）
- 解决「AI 对开源项目一知半解」的问题
- 双半魂模型：代码半魂 + 社区半魂
- 准确率 42% → 96%

### 新版（本次讨论确定）
- **超级 copy cat** —— copy 的信息来源是开源项目的 GitHub + 用户踩坑经验
- **不帮从 0 开发**，而是**站在巨人肩膀上快速抄作业**
- 核心是**准确识别知识和经验**，并作用于未来帮助用户最简单的实践

## 三、范式转变

| 旧范式 | 新范式 |
|--------|--------|
| 学习成本：高 | 使用成本：零 |
| 用户适应工具 | 工具适应用户 |
| 教用户开发 | 直接给成品 |

### 竞品定位差异

| 产品 | 定位 |
|------|------|
| Cursor/Copilot | 建筑师 - 从 0 构建 |
| Doramagic | 抄作业专家 - 复制 + 调整 |

---

## 四、三个核心挑战

| 挑战 | 优先级 | 说明 |
|------|--------|------|
| **噪音过滤** | ⭐⭐⭐ 最高 | 非常重要且核心。能否准确识别知识和经验，是产品成立的前提 |
| **关联推理** | ⭐⭐ 很高 | 用户表达经常是缺失的，如何将用户需求准确映射到开源项目 |
| **时效性** | ⭐ 次要 | 知识是沉淀的，不是新闻。不需要追求实时性 |

## 五、用户场景

### 场景 1：告警规则不生效
- **用户问题**：「为什么我的告警规则不生效？」
- **通用 LLM**：告诉你「在 alertmanager.yml 里加 group_by 参数」
- **Doramagic**：
  - 代码层面：为什么要这么配
  - 社区层面：GitHub Issue 里的 bug report、别人的踩坑经验、最新的 work-around

### 场景 2：Vibe Coding 新手
- 用 Cursor/Claude Code 写代码
- 遇到问题和实际跑起来的结果不一样
- 需要快速定位是「配置问题」还是「版本 bug」还是「踩过的坑」

## 六、长期记忆系统

### 核心方向
记住用户的「适合」—— 动态调整，而非固定选项。

### 记忆维度
| 维度 | 说明 |
|------|------|
| 喜欢什么方式 | 用户偏好什么交互风格 |
| 跳过什么建议 | 用户不关注哪些信息 |
| 选择什么配置 | 用户最终决策模式 |
| 关注什么信息 | 用户注意力焦点 |

### 实现目标
每次交互都在学习「什么是适合这个用户的 Skill」，而非每次从零开始。
