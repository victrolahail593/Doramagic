# 提示词设计评审报告

研究日期: 2026-03-10 | 模型: GLM-5

## 现状分析

Soul Extractor v0.8.0 采用 6 阶段流水线设计，每个阶段有独立的 prompt 文件：

| Stage | 文件 | 核心任务 | Prompt 长度 |
|-------|------|---------|------------|
| 0 | SKILL.md | 准备与编排 | ~1.5KB |
| 1 | STAGE-1-essence.md | 灵魂发现（7问） | ~3.4KB |
| 2 | STAGE-2-concepts.md | 概念卡+工作流卡 | ~2.4KB |
| 3 | STAGE-3-rules.md | 规则卡+社区陷阱 | ~5.0KB |
| 3.5 | STAGE-3.5-review.md | 验证与审查 | ~2.2KB |
| 4 | STAGE-4-synthesis.md | 专家叙事合成 | ~6.4KB |

整体设计亮点：
- **反合理化检查表**：Stage 3 和 Stage 4 都有"你的想法 vs 真相"表格，预判 agent 的偷懒行为
- **Hard Gate 机制**：每个阶段都有明确的阻断条件
- **验证脚本**：Stage 3.5 有 Python 脚本做硬校验，检查字段完整性、sources 可追溯性
- **示例驱动**：Stage 1 和 Stage 4 都有完整的 python-dotenv 示例

---

## 问题发现

### 1. Stage 1 灵魂发现：Q6/Q7 质量保证不足

**问题 1.1：Q6/Q7 的 Hard Gate 是"软约束"**

```
- Q6 不能是功能描述的换一种说法，必须是设计信念
- Q7 不能是项目简介，必须是能让人"顿悟"的心智模型
```

这两条约束是**声明式的**，没有给出可执行的检查方法。Agent 可以写出"这个项目的设计哲学是让开发者更方便地管理环境变量"这种换皮的功能描述，而 Hard Gate 无法检测。

**问题 1.2：Q1-Q5 与 Q6-Q7 的关系不明确**

Q1-Q5 收集的是事实信息（解决什么问题、替代方案、核心承诺等），但 prompt 没有明确说明这些信息如何帮助回答 Q6/Q7。Agent 可能把 Q1-Q5 当成独立任务，而不是 Q6/Q7 的推理素材。

**问题 1.3：缺少"反例"来校准质量**

示例只展示了"好的 Q6/Q7"，没有展示"坏的 Q6/Q7"及其修正过程。Agent 缺乏对"什么是功能描述换皮"的具体感知。

---

### 2. Stage 2 概念卡/工作流卡：模板不完整

**问题 2.1：工作流卡缺少完整示例**

概念卡有完整的 `CC-001` 示例，但工作流卡只描述了"步骤(含file:line) + 流程图 + 失败模式"，没有给出具体的 YAML frontmatter 和 markdown 结构。Agent 可能产出格式不一致的工作流卡。

**问题 2.2：Evidence 的 file:line 格式未强制校验**

验证脚本 `validate_extraction.py` 只检查文件路径是否存在，不检查行号是否有效。Agent 可以写 `src/main.py:999999` 而通过校验。

**问题 2.3：概念卡和工作流卡的数量硬编码为 3**

```
提取 3 张概念卡 + 3 张工作流卡
```

不同复杂度的项目可能需要不同数量的卡片。简单项目可能只有 1-2 个核心概念，复杂项目可能需要 5+ 个。硬编码数量会导致：
- 简单项目：强行拆分，产出低质量卡片
- 复杂项目：遗漏重要概念

---

### 3. Stage 3 规则卡：反合理化检查有效但可加强

**问题 3.1：反合理化检查表是"一次性"的**

反合理化检查表只在 prompt 中出现一次，Agent 读完后可能"忘记"。建议在验证脚本中加入语义检查，例如：
- 检测 DR-100~ 卡片的 sources 是否只引用代码（没有 Issue/CHANGELOG）
- 检测规则卡的"真实场景"是否过于抽象（缺少人名、时间、具体操作）

**问题 3.2：社区陷阱卡的真实性验证不足**

验证脚本只检查 Issue 编号是否出现在 `community_signals.md` 中，但不检查：
- Issue 是否真的与规则相关
- reactions 数是否被正确引用
- 规则描述是否与 Issue 内容一致

**问题 3.3：severity 的定义模糊**

```
severity: CRITICAL/HIGH/MEDIUM/LOW
```

没有给出每个级别的具体定义。Agent 可能把所有规则都标为 HIGH，或随意分配严重度。

---

### 4. Stage 3.5 验证与审查：软审查依赖自评

**问题 4.1：软审查的三个维度是自评**

```
| 维度 | 好的标准 | 差的标准 |
```

Agent 需要自己判断"具体性"、"真实性"、"可执行性"。但：
- Agent 可能对自己的产出有"确认偏误"
- 没有外部校准机制

**问题 4.2：审查产出格式是"汇报式"而非"修正式"**

```
DR-001: pass (具体性 ✓, 真实性 ✓, 可执行性 ✓)
DR-002: warn (真实场景太抽象，已改进)
```

这个格式鼓励 Agent 写 `pass` 而不是真正改进。建议改为：
- 必须列出具体的改进点
- warn/fail 必须附带修改后的卡片片段

---

### 5. Stage 4 专家叙事合成：拼接检测不足

**问题 5.1：反合理化检查表不够具体**

```
| "我直接把规则卡拼起来就行" | 那是数据库导出，不是专家叙事。重写。 |
```

但没有给出"拼接"的具体检测方法。Agent 可能只是把规则卡的标题连起来，加上一些过渡词。

**问题 5.2：缺少"叙事质量"的量化指标**

1500 字限制是长度约束，不是质量约束。建议加入：
- "踩坑故事"必须包含人、时间、场景、操作、结果、根因 6 要素
- "为什么这样设计"必须有因果链（因为 X → 所以 Y → 这就是为什么 Z）

**问题 5.3：Q6/Q7 直接复制可能继承 Stage 1 的质量问题**

```
设计哲学和心智模型从 00-soul.md 直接取，不要重写或稀释
```

如果 Stage 1 的 Q6/Q7 质量不高，Stage 4 会直接继承问题。建议在 Stage 4 加入 Q6/Q7 的质量校验。

---

### 6. Hard Gate 机制：强制力不足

**问题 6.1：Hard Gate 是文本声明，不是代码检查**

除了 Stage 3.5 的验证脚本，其他阶段的 Hard Gate 都是 prompt 中的文本声明。Agent 可以"假装"满足条件。

**问题 6.2：缺少阶段间的依赖检查**

Stage 4 声称要读取 Stage 1-3 的产出，但没有验证这些文件是否存在、是否完整。如果 Stage 2 跳过了某些卡片，Stage 4 可能产出不完整的叙事。

**问题 6.3：验证脚本的覆盖范围有限**

`validate_extraction.py` 只检查：
- 字段完整性
- severity 枚举值
- sources 格式
- 卡片数量

不检查：
- 内容质量（具体性、真实性、可执行性）
- Q6/Q7 是否是"设计信念"而非"功能描述"
- 专家叙事是否是"拼接"

## 改进建议

### P0（阻塞性问题）

#### 1. Stage 1：增加 Q6/Q7 质量校验

**方案 A：增加反例校准**

```markdown
## Q6/Q7 质量校准

### ❌ 错误的 Q6（功能描述换皮）
"python-dotenv 的设计哲学是让开发者更方便地管理环境变量，通过读取 .env 文件自动加载配置。"

这是功能描述，不是设计信念。它说的是"做什么"，不是"为什么这样做"。

### ✅ 正确的 Q6（设计信念）
"python-dotenv 的核心信念是'模拟手动 export 行为'。所有看似奇怪的默认行为——不覆盖已有变量、遵循 shell 引号规则——都是因为 shell 的 export 就是这样工作的。"

这揭示了设计决策背后的信念。

### ❌ 错误的 Q7（项目简介）
"python-dotenv 是一个用于加载 .env 文件的 Python 库，支持变量插值和多行值。"

这是项目简介，不能让人"顿悟"。

### ✅ 正确的 Q7（心智模型）
"把 .env 想成一个在应用启动时静默执行的 shell 脚本。每行是一条 export 命令。理解了 shell 变量行为，你就理解了 python-dotenv。"

这是一个类比，能让人"顿悟"。
```

**方案 B：增加自检问题**

```markdown
## Q6/Q7 自检清单

在提交 Q6 之前，问自己：
- [ ] 这句话能解释"为什么这样设计"而不是"设计了什么"吗？
- [ ] 如果我把项目名换成另一个项目，这句话还能成立吗？（如果成立，说明太泛）
- [ ] 这句话能让一个新贡献者理解"原来如此"吗？

在提交 Q7 之前，问自己：
- [ ] 这是一个类比或一句话概括吗？
- [ ] 理解了这句话，能推导出项目 80% 的行为吗？
- [ ] 这句话是"顿悟式"的吗？
```

#### 2. Stage 2：补全工作流卡模板

```markdown
### 工作流卡完整示例

---
card_type: workflow_card
card_id: WF-001
repo: python-dotenv
title: "load_dotenv() — Main Entry Point"
---

## 触发条件
用户调用 `load_dotenv()` 无参数时触发此工作流。

## 步骤

1. **文件发现** (src/dotenv/main.py:L45-L52)
   - 调用 `find_dotenv()` 向上搜索 .env 文件
   - 找不到时返回空字符串，不报错

2. **解析** (src/dotenv/parser.py:L23-L67)
   - 逐行读取，处理注释、引号、转义
   - 产出 `(key, value)` 元组流

3. **插值** (src/dotenv/variables.py:L12-L45)
   - 如果 `interpolate=True`，展开 `${VAR}` 引用
   - 递归解析嵌套变量

4. **注入** (src/dotenv/main.py:L78-L85)
   - 将键值对写入 `os.environ`
   - 如果 `override=False`，跳过已存在的键

## 流程图

find_dotenv() → parse() → interpolate() → os.environ
     ↓              ↓            ↓              ↓
  找文件        解析行       展开变量        注入环境

## 失败模式
| 失败 | 原因 | 解决 |
|------|------|------|
| 找不到 .env | 文件不在搜索路径 | 显式传 `dotenv_path` |
| 值被截断 | 多行值未加引号 | 用双引号包裹 |
| 变量未展开 | `interpolate=False` | 默认开启，检查是否被关闭 |
```

#### 3. Stage 3.5：将软审查维度加入验证脚本

```python
# 在 validate_extraction.py 中增加

def check_narrative_quality(body, card_type):
    """Check narrative quality dimensions."""
    warnings = []
    
    if card_type == "decision_rule_card":
        # 检查"真实场景"是否包含具体元素
        scenario_match = re.search(r'## 真实场景\n+(.+?)(?=\n##|\Z)', body, re.DOTALL)
        if scenario_match:
            scenario = scenario_match.group(1)
            # 检查是否有人称代词（具体性指标）
            if not re.search(r'(你|我|他|她|他们|开发者|用户)', scenario):
                warnings.append("真实场景缺少人称代词，可能过于抽象")
            # 检查是否有具体操作（可执行性指标）
            if not re.search(r'(运行|执行|调用|配置|设置|修改)', scenario):
                warnings.append("真实场景缺少具体操作，可执行性不足")
    
    return warnings
```

## 总结

Soul Extractor 的 prompt 设计整体思路清晰，反合理化检查表是一大创新。但存在以下核心问题：

1. **质量保证依赖 Agent 自律**：Hard Gate 多为文本声明，缺少代码级校验
2. **Q6/Q7 是瓶颈**：作为整个流水线最重要的产出，其质量保证不足
3. **软审查流于形式**：三个维度的自评容易产生确认偏误
4. **拼接检测不足**：Stage 4 的"专家叙事"可能只是规则卡的重新排列

建议优先实施 P0 级改进，特别是 Q6/Q7 的质量校验和工作流卡模板补全。这两项改进成本低、影响大，能显著提升产出质量。
