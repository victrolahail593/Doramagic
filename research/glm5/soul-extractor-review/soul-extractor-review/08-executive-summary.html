Soul Extractor 改进研究摘要

研究日期: 2026-03-10 | 研究者: G大脑袋 + 6个子代理 (GLM-5)

一、研究概述

通过 6 个子代理并行研究，对 soul-extractor skill 进行了全面评审，覆盖：流程编排、工具调用、提示词设计、鲁棒性、社区反馈、竞品分析。

发现问题：18 个 P0（阻塞）、15 个 P1（重要）、10 个 P2（改进）

核心结论：设计理念优秀，工程化不足。

二、关键发现

2.1 流程编排

  • ❌ 无中断续跑机制，中断后需完全重来
  • ❌ 验证失败无自动修复，需手动解读错误
  • 💡 Stage 2/3 可并行化，节省 30% 时间

2.2 工具调用

  • ❌ 缺少依赖预检，用户看到 cryptic 错误
  • ❌ 无仓库大小/网络超时限制，大仓库卡死
  • ❌ Windows 不兼容，Bash 脚本需 Git Bash/WSL

2.3 提示词设计

  • ❌ Stage 1 Q6/Q7 无执行检查，质量无保证
  • ❌ Hard Gate 是声明式约束，可被绕过
  • 💡 需新增反例校准和自检清单

2.4 鲁棒性

  • ❌ 大仓库截断（150KB），关键代码丢失
  • ❌ 无社区信号时死循环，非 GitHub 项目不可用
  • ❌ MIN_COMMUNITY_RULES=3 对新项目无法满足

2.5 社区反馈

  • ❌ SOUL.md 不被加载/被忽略（核心痛点）
  • ❌ Personality Drift（人格漂移）
  • 💡 用户期望：自动化提取、稳定性保障、更好模板

2.6 竞品差距

  • Repomix：Tree-sitter 压缩 70% token
  • GPTMe：Lessons 机制动态注入
  • Cursor Rules：globs 文件模式匹配 + 懒加载
  • ❌ soul-extractor 缺少：配置化、Token 预算、关键词触发

三、优先改进建议

P0 必须修复（约 20h）

# 改进项 工作量
1 中断续跑机制（.progress.json） 4h
2 依赖预检 + 跨平台支持 6h
3 验证失败自动修复（--auto-fix） 5h
4 无社区信号降级处理 2h
5 Q6/Q7 质量保证增强 3h

P1 重要改进（约 20h）

# 改进项 工作量
6 Stage 2/3 并行化 4h
7 配置化提取策略（soul-extractor.yaml） 8h
8 Token 预算控制 4h
9 工作流卡模板补全 2h
10 动态注入机制 2h

P2 体验优化（约 10h）

# 改进项
11 进度反馈量化
12 交互式反馈（skip/pause）
13 社区模板集成

四、架构建议

4.1 配置驱动

extraction:
  max_tokens: 100000
  community_rules: 3
output:
  formats: [markdown, json]

4.2 中断续跑

{
  "current_stage": "stage-3",
  "completed_stages": ["stage-0", "stage-1", "stage-2"]
}

4.3 验证增强

  • 新增 --auto-fix 自动修复格式问题
  • 输出具体修复命令

五、总结

优势：核心理念准确、流程清晰、社区信号有价值

不足：缺少容错机制、配置化、跨平台支持、验证强制力

建议路径：先修复 P0 → 再实现配置化 → 最后优化体验

预计工作量：P0 20h + P1 20h + P2 10h = 50h
