## 1. 概述

### 1.1 Doramagic 与 Prompt 工程的关系

Doramagic 是一个**纯 LLM 驱动的 Skill**，核心能力完全依赖 Prompt 实现。这意味着：

- **无硬编码逻辑**：所有决策和流程都通过 Prompt 引导 LLM 完成
- **动态生成**：根据用户输入动态生成响应，而非预定义回复
- **分层架构**：通过多层 Prompt 实现复杂的工作流程

### 1.2 Prompt 设计原则

基于 Doramagic 实践，总结出以下核心原则：

| 原则 | 说明 | 示例 |
|------|------|------|
| **角色锚定** | 明确 LLM 的身份和视角 | "你是一名知识提取专家" |
| **结构化输出** | 指定精确的输出格式 | 使用 YAML frontmatter + Markdown |
| **证据驱动** | 要求标注信息来源 | "证据等级：E1-E4" |
| **边界明确** | 说明"不是什么" | "边界：它不是什么" |
| **渐进式引导** | 分步骤完成任务 | 鹰眼→深潜→铸造 |

## 2. 意图理解 Prompt

### 2.1 核心挑战

意图理解是 Doramagic 的第一步，面临以下挑战：

- 用户表达往往模糊、不完整
- 同一需求有多种表达方式
- 需要判断用户真正想要什么

### 2.2 设计策略

#### 策略一：核心三问法

通过三个基本问题快速构建需求画像：

```markdown
## 需求挖掘 - 核心三问

请回答以下三个问题：

1. **你最想解决什么问题？**（核心痛点）
   - 描述你面临的最大挑战

2. **你希望最终得到什么？**（成功画面）
   - 描述理想的结果状态

3. **你的技术背景是什么？**
   - 描述你的技术经验水平
```

#### 策略二：场景细化追问

根据用户回答，追问 3-5 个关键维度：

```markdown
## 场景细化

基于你的回答，我需要了解更多细节：

1. **预算范围**：你愿意投入多少资源？
2. **时间线**：你希望多久完成？
3. **交付方式**：你想要代码、文档，还是完整服务？
4. **使用场景**：谁会使用这个方案？
5. **约束条件**：有什么必须满足的限制？
```

#### 策略三：置信度判断

```markdown
## 意图分析

分析用户输入，评估以下维度：

| 维度 | 评估标准 | 置信度 |
|------|----------|--------|
| 需求清晰度 | 用户是否明确说明想要什么？ | 高/中/低 |
| 领域明确性 | 是否确定所属领域？ | 高/中/低 |
| 技术可行性 | 需求是否可实现？ | 高/中/低 |

**输出要求**：
- 如果置信度 ≥ 80%，继续执行
- 如果置信度 50-80%，返回确认问题
- 如果置信度 < 50%，返回澄清请求
```

### 2.3 处理模糊表达的技巧

```markdown
## 模糊表达处理规则

当用户输入模糊时：

1. **识别关键词**：提取技术栈、场景、目标
2. **假设补全**：基于常见模式推断完整需求
3. **确认推进**："我假设你的意思是...，对吗？"
4. **选项引导**：提供 2-3 个选项让用户确认

示例处理：

| 模糊输入 | 处理方式 | 澄清输出 |
|----------|----------|----------|
| "我想做个网站" | 追问场景 | "个人博客还是商业网站？" |
| "帮我分析这个" | 追问目标 | "你想了解技术细节还是使用方式？" |
| "有没有开源方案" | 追问约束 | "对编程能力有要求吗？" |
```

## 3. Skill 规划 Prompt

### 3.1 核心挑战

- LLM 需要理解复杂的多阶段工作流
- 需要准确输出 JSON 格式的技能调用
- 多技能组合时的依赖关系处理

### 3.2 设计策略

#### 策略一：阶段锚定

```markdown
## Skill 规划框架

你正在规划一个多阶段任务。当前阶段：{phase}

### 已完成阶段
{completed_phases}

### 当前任务
{task_description}

### 可用技能
{available_skills}

### 规划要求

1. **选择技能**：从可用技能中选择最合适的
2. **确定顺序**：明确技能调用顺序
3. **处理依赖**：说明前序技能的输出如何影响后续
4. **输出格式**：JSON 格式

输出 JSON：
```json
{
  "skill_chain": [
    {
      "skill": "skill-name",
      "reason": "为什么选择这个技能",
      "input": { /* 传递给技能的输入 */ },
      "depends_on": null
    }
  ],
  "confidence": 0.85,
  "reasoning": "规划理由"
}
```
```

#### 策略二：JSON 格式强制

```markdown
## JSON 输出规范

**重要**：你的输出必须是有效的 JSON，不要包含任何额外文本。

### 输出约束

1. 使用双引号包裹字符串
2. 不允许尾随逗号
3. 键名必须使用双引号
4. 布尔值使用 true/false（不是 True/False）
5. null 表示空值（不是 None 或 null）

### 验证步骤

输出 JSON 前，自检：
- [ ] 所有字符串用双引号
- [ ] 无尾随逗号
- [ ] JSON 语法正确
- [ ] 包含所有必需字段

### 错误处理

如果无法生成有效 JSON：
- 保留字段设为 null
- 在 reasoning 字段说明原因
- 不要输出任何非 JSON 内容
```

#### 策略三：多技能组合

```markdown
## 多技能组合规划

任务：{task_description}

### 技能依赖图

```
[技能A] → 输出X → [技能B] → 输出Y → [技能C]
                    ↓
              [技能D](并行)
```

### 组合规则

1. **串行执行**：后一个技能依赖前一个的输出
2. **并行执行**：多个技能可以同时运行
3. **聚合执行**：并行技能的结果聚合后传递给下游

### 输出格式

```json
{
  "execution_plan": {
    "stages": [
      {
        "stage": 1,
        "parallel": false,
        "skills": ["技能A"],
        "inputs": { /* 初始输入 */ }
      },
      {
        "stage": 2,
        "parallel": true,
        "skills": ["技能B", "技能D"],
        "inputs_from": [1],
        "aggregation": "merge"
      },
      {
        "stage": 3,
        "parallel": false,
        "skills": ["技能C"],
        "inputs_from": [2]
      }
    ]
  }
}
```
```

## 4. 结果生成 Prompt

### 4.1 「哆啦A梦」风格设计

Doramagic 的核心特色是「哆啦A梦」风格——像从万能口袋中掏出宝物一样交付解决方案。

```markdown
## 角色设定

你是 Doramagic，像哆啦A梦一样从万能口袋里掏出解决方案。

### 风格要求

1. **亲切开场**：用"嘿！"、"看我的！"等活泼开场
2. **宝物展示**：用"我有几个宝贝可以帮你"引出方案
3. **逐个介绍**：像介绍道具一样说明每个方案
4. **使用说明**：像说明道具用法一样给出使用指南
5. **鼓励结尾**："有了这些，你一定能成功！"

### 示例

"嘿！看我的！我从万能口袋里掏出了三个宝贝：

**宝贝一：XXX**
- 这是什么：一个能帮你...的神奇道具
- 怎么使用：只需要...
- 注意啦：有个小陷阱要小心...

..."
```

### 4.2 选择题格式实现

```markdown
## 选择题输出格式

当需要用户做决定时，使用选择题格式：

### 格式模板

```
🔮 我发现了几个可能的方案，你想要哪一个？

**方案 A：{名称}**
- 优点：{优点1}、{优点2}
- 缺点：{缺点1}
- 适合：{目标用户}
- 推荐指数：⭐⭐⭐⭐⭐

**方案 B：{名称}**
- 优点：{优点1}、{优点2}
- 缺点：{缺点1}
- 适合：{目标用户}
- 推荐指数：⭐⭐⭐⭐

**方案 C：{名称}**
- 优点：{优点1}
- 缺点：{缺点1}、{缺点2}
- 适合：{目标用户}
- 推荐指数：⭐⭐⭐

请回复 A、B 或 C 告诉我你的选择！
```

## 5. 输出格式控制

### 5.1 JSON 输出保障

```markdown
## JSON 输出保障策略

### 策略一：结构化提示

在 Prompt 中明确指定 JSON Schema：

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "skill_name": {
      "type": "string",
      "description": "技能名称"
    },
    "parameters": {
      "type": "object",
      "properties": {
        "project_url": {"type": "string"},
        "focus_areas": {"type": "array", "items": {"type": "string"}}
      }
    },
    "confidence": {"type": "number", "minimum": 0, "maximum": 1}
  },
  "required": ["skill_name", "confidence"]
}
```

### 策略二：双重输出

```markdown
## 输出格式

请同时输出两种格式：

1. **JSON 格式**（供程序解析）
```json
{
  "skill": "extractor",
  "params": {}
}
```

2. **人类可读格式**（供用户理解）
- 技能名：extractor
- 用途：提取代码库知识

### 策略三：容错处理

当 JSON 解析失败时：
1. 提取 JSON 代码块中的内容
2. 尝试修复常见语法错误
3. 如果仍然失败，返回降级响应
```

### 5.2 幻觉处理

```markdown
## 幻觉防范策略

### 识别幻觉信号

| 信号 | 特征 | 处理方式 |
|------|------|----------|
| 过度具体 | 给出过于详细的实现细节 | 要求标注"推测" |
| 证据缺失 | 声称来自源码但无引用 | 要求提供引用 |
| 绝对化 | "完美"、"唯一"、"最佳" | 改为"通常"、"常见" |

### 提示词技巧

```markdown
## 证据标注要求

每条陈述必须标注来源：

| 陈述 | 证据等级 | 来源 |
|------|----------|------|
| "该项目使用 React" | E1 | README.md |
| "性能优于 X" | E3 | 社区讨论 |
| "作者建议使用 Y" | E2 | Issue #123 |

**禁止**：
- 未经标注的绝对性陈述
- 无法提供来源的技术细节
- 超出证据范围的推论
```

### 5.3 结构化输出模板

```markdown
## 知识卡片模板

### YAML Frontmatter

```yaml
---
id: CC-<项目名>-<编号>
type: concept_card
evidence_level: E1-E4
source_files:
  - path/to/file1.ts
  - path/to/file2.py
tags:
  - 核心概念
  - 架构
---
```

### Markdown 内容

```markdown

## 定义
[这个概念是什么]

## 为什么重要
[为什么需要理解它]

## 边界
[它不是什么]

## 用户解释
[用通俗语言解释]

## 参考资料
- [文件1] (行号)
- [文件2] (行号)
```

## 6. Prompt 优化技巧

### 6.1 核心优化技巧

| 技巧 | 说明 | 示例 |
|------|------|------|
| **few-shot 学习** | 提供示例引导输出 | 给出 3 个问答示例 |
| **思维链** | 要求 LLM 展示推理过程 | "先思考，再回答" |
| **约束强化** | 强调必须遵守的规则 | "重要：必须使用 JSON" |
| **角色扮演** | 设定具体身份增强聚焦 | "你是一名资深架构师" |
| **分步引导** | 将复杂任务拆解为步骤 | "Step 1: ... Step 2: ..." |

回答问题前，请按以下步骤思考：

2. **识别关键**：有哪些关键信息需要考虑？

### 6.3 常见问题与解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 输出格式不稳定 | Prompt 太宽松 | 增加格式约束 + 示例 |
| 幻觉严重 | 缺少证据标注 | 强制要求标注来源 |
| 理解偏差 | 角色不清晰 | 强化角色设定 |
| 过度冗长 | 无长度限制 | 指定输出长度 |
| 遗漏关键点 | 无检查清单 | 添加 checklist |

## 7. 测试与迭代

### 7.1 测试方法

```markdown
## Prompt 测试框架

### 测试用例设计

**正向测试**：验证 Prompt 能正确处理正常输入
- 清晰的需求描述
- 完整的技术背景
- 明确的交付要求

**负向测试**：验证 Prompt 能正确处理边界情况
- 模糊的需求描述
- 缺失的关键信息
- 冲突的要求

**压力测试**：验证 Prompt 能处理复杂场景
- 多技能组合需求
- 大规模项目分析
- 特殊约束条件

### 评估指标

| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| 格式正确率 | ≥ 95% | JSON 解析成功率 |
| 意图准确率 | ≥ 90% | 人工评估匹配度 |
| 响应时间 | < 5s | 日志统计 |
| 用户满意度 | ≥ 4.5/5 | 反馈收集 |
```
