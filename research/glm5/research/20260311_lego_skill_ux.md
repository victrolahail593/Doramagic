## 一、研究背景与目标

### 1.1 终局场景

Doramagic 的愿景是让普通用户能够像玩乐高一样，为 OpenClaw 创建定制化的 Skill。这是一种「授人以渔」的理念——用户不需要懂编程，只需要表达自己想要什么，就能获得一个可用的自动化能力。

### 1.2 研究问题

本研究聚焦以下六个核心问题：

1. "玩乐高"的体验本质是什么？
2. 这种体验如果应用到 Skill 创建，交互流程应该是什么样的？
3. 如何让用户描述"我想要 XXX 功能"，然后帮他组合出需要的 Skill？
4. 如何设计"验证"环节，让用户知道拼出来的 Skill 能工作？
5. 如何处理用户的错误选择？提供"撤销"或"调整"的体验？
6. 参考现有的 No-Code/Low-Code 平台，有什么最佳实践？

## 二、"玩乐高"的体验本质分析

### 2.1 乐高体验的核心要素

通过研究乐高的用户体验设计原则，我们可以提炼出以下关键要素：

| 核心要素 | 描述 | 对应 UX 原则 |
|---------|------|-------------|
| **模块化（Modularity）** | 标准化积木块，可无限组合 | 可组合性（Composability） |
| **即时反馈（Immediate Feedback）** | 拼上去就知道合不合适 | 可供性（Affordance） |
| **可视化（Visualization）** | 所见即所得，看到成品样子 | 映射（Mapping） |
| **可逆性（Reversibility）** | 拼错了拔下来重拼 | 容错性（Error Tolerance） |
| **渐进式复杂度（Progressive Complexity）** | 从简单到复杂，门槛低上限高 | 渐进式披露（Progressive Disclosure） |
| **成就驱动（Achievement）** | 拼完有成就感，愿意分享 | 目标梯度效应 |

### 2.2 乐高体验的用户心理模型

玩乐高时，用户的心理模型是：

```
1. 想象目标 → 2. 识别可用块 → 3. 尝试拼接 → 4. 验证匹配 → 5. 调整或确认
```

这个模型的关键特征是：

- **循环迭代**：不是线性流程，而是不断循环"尝试-验证-调整"
- **低成本试错**：拔下重拼的代价很低，用户愿意多次尝试
- **即时满足**：每拼一块都有进展感，不需要等到完成才获得反馈

### 2.3 对 Skill 创建的启示

将乐高体验迁移到 Skill 创建，意味着：

| 乐高元素 | 对应 Skill 元素 |
|---------|-----------------|
| 积木块 | Skill 组件（Trigger、Action、Condition） |
| 拼接 | 组件之间的连接 |
| 说明书/图纸 | 预设模板/工作流 |
| 拼出来的作品 | 完整的自动化流程 |
| 拔下重拼 | 撤销/修改组件 |

## 三、Skill 创建的交互流程设计

### 3.1 总体流程框架

基于乐高体验的本质，我们设计如下六步交互流程：

```
┌─────────────────────────────────────────────────────────────────┐
│                      乐高式 Skill 创建流程                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐   │
│   │ 1. 描述  │───▶│ 2. 推荐  │───▶│ 3. 组合  │───▶│ 4. 验证  │   │
│   │ 想要什么 │    │ 智能建议  │    │ 拖拽拼接 │    │ 测试运行 │   │
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘   │
│                                             │                 │
│                                             ▼                 │
│   ┌─────────┐    ┌─────────┐          ┌─────────┐           │
│   │ 6. 完成  │◀───│ 5. 调整  │◀────────│ (验证失败)│           │
│   │ 保存分享 │    │ 撤销重试 │          │ 返回调整  │           │
│   └─────────┘    └─────────┘          └─────────┘           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 各步骤详细设计

#### 步骤 1：描述想要什么

**目标**：让用户用自然语言表达需求

**设计要点**：

- **入口设计**：首页突出"创建 Skill"按钮，下方展示"热门 Skill"降低冷启动焦虑
- **输入方式**：支持自然语言描述（如"当我收到 Gmail 邮件时，自动提取附件保存到 Dropbox"）
- **智能提示**：输入时实时显示候选意图，帮助用户明确需求
- **示例引导**：提供 3-5 个典型示例，让用户知道可以怎么表达

**界面原型概念**：

```
┌────────────────────────────────────────────────────────┐
│  🎯 创建新 Skill                                      │
├────────────────────────────────────────────────────────┤
│                                                        │
│  "我想要当我___________，自动___________"              │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │ 当我收到 Gmail 新邮件时，自动...               │ │
│  └──────────────────────────────────────────────────┘ │
│                                                        │
│  💡 试试这样描述：                                     │
│  • "每天早上 8 点给我播报天气"                        │
│  • "有人在我的 GitHub 项目发 issue 时通知 Slack"     │
│  • "把收到的发票自动整理到表格里"                     │
│                                                        │
└────────────────────────────────────────────────────────┘
```

#### 步骤 2：智能推荐组件

**目标**：基于用户描述，推荐合适的组件组合

**设计要点**：

- **组件分类**：将可用能力分为 Trigger（触发）、Action（动作）、Condition（条件）三类
- **可视化展示**：用卡片/积木形式展示推荐组件，带图标和简短说明
- **可选性**：用户可以接受推荐，也可以自己从组件库选择
- **透明度**：告诉用户"我为什么推荐这个"

**组件库结构示例**：

```
Skill 组件库
├── 📥 触发器 (Trigger)
│   ├── Gmail 新邮件
│   ├── 日历 事件开始
│   ├── GitHub Issue 创建
│   ├── Webhook 调用
│   └── ⏰ 定时触发
│
├── ⚡ 动作 (Action)
│   ├── 📧 发送邮件
│   ├── 💬 发送 Slack 消息
│   ├── 📝 写入 Google Sheets
│   ├── 📁 保存文件到 Drive
│   └── 🔔 发送手机通知
│
└── 🔀 条件 (Condition)
    ├── ⛔ 过滤（满足条件才执行）
    ├── 🔀 分支（根据条件选择路径）
    └── ⏳ 延迟（等待一段时间）
```

#### 步骤 3：组合与连接

**目标**：可视化地把组件拼在一起

**设计要点**：

- **画布式编辑**：像 Make.com 那样，用可视化画布连接组件
- **拖拽交互**：组件可以拖进画布，连接线可以拖出来
- **即时反馈**：连线时显示数据类型匹配提示
- **自动布局**：提供"整理布局"按钮，自动对齐组件

**界面概念**：

```
┌─────────────────────────────────────────────────────────────┐
│  Skill: 邮件附件自动保存                          [测试] [保存] │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────┐         ┌─────────────┐                  │
│   │ 📥 Gmail    │────────▶│ ⚡ 保存到    │                  │
│   │ 新邮件附件  │         │ Dropbox     │                  │
│   └─────────────┘         └─────────────┘                  │
│        │                                               │
│        ▼                                               │
│   ┌─────────────┐                                       │
│   │ 🔔 发送通知 │                                       │
│   └─────────────┘                                       │
│                                                             │
│   [+ 添加组件]                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 步骤 4：验证与测试

**目标**：让用户确认 Skill 能正常工作（详见第四章）

#### 步骤 5：调整与优化

**目标**：处理错误，提供撤销/修改能力（详见第五章）

#### 步骤 6：完成与分享

**目标**：保存 Skill，允许分享

**设计要点**：

- **命名**：让用户给 Skill 起个名字
- **描述**：自动生成描述，也可手动编辑
- **分享**：可以设为私有、分享链接、或提交到社区模板库
- **展示**：完成后展示"你已经创建了 X 个 Skill"，增强成就感

## 四、验证环节设计

### 4.1 验证的重要性

在乐高体验中，"拼上去就知道合不合适"是核心反馈机制。在 Skill 创建中，用户需要知道：
- 组件之间的连接是否正确？
- 数据能否正确传递？
- 整体流程能否跑通？

### 4.2 验证的三层设计

#### 第一层：语法验证（即时）

**时机**：用户在编辑时实时进行

**检查内容**：
- 必填字段是否填写
- 数据类型是否匹配
- 必需的配置是否完成

**反馈方式**：
- 红色边框/图标标识问题
- 悬停显示错误提示
- 阻止进入下一步直到修复

```
┌─────────────────────────────────────┐
│ ⚠️ 验证失败                          │
│ ─────────────────────────────────── │
│ • Gmail 触发器：需要选择标签过滤     │
│ • Dropbox 动作：需要选择目标文件夹  │
└─────────────────────────────────────┘
```

#### 第二层：流程验证（保存前）

**时机**：用户点击"测试"或"保存"时

**检查内容**：
- 完整流程是否有断点
- 所有分支是否都有出口
- 循环引用检测

**反馈方式**：
- 流程图高亮问题节点
- 给出修复建议

#### 第三层：运行验证（测试执行）

**时机**：用户点击"测试运行"时

**设计要点**：

- **模拟运行**：不真正执行，只模拟数据流
- **真实测试**：用最近的真实数据或模拟数据跑一遍
- **分步展示**：像调试器一样展示每一步的输入输出
- **可重试**：测试失败可以修改后重试

**界面概念**：

```
┌─────────────────────────────────────────────────────────────┐
│  🧪 测试运行                                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: Gmail 新邮件触发器                                 │
│  ├─ 输入: {subject: "发票", from: "vendor@..."}            │
│  └─ 输出: ✓ 触发成功                                       │
│                                                             │
│  Step 2: 提取附件                                          │
│  ├─ 输入: [invoice.pdf]                                    │
│  └─ 输出: ✓ 提取成功                                       │
│                                                             │
│  Step 3: 保存到 Dropbox                                   │
│  ├─ 输入: /invoices/invoice.pdf                           │
│  └─ 输出: ✗ 权限不足 - 需要 OAuth 授权                     │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 🔧 需要授权 Dropbox 访问权限                        │   │
│  │ [立即授权] [查看权限说明]                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 验证的心理学设计

- **乐观设计**：默认显示绿色，直到发现问题
- **渐进披露**：不一次性显示所有错误，而是按步骤
- **可操作性**：每个错误都有明确的修复路径
- **学习机会**：错误提示要解释"为什么"，不只是"什么错了"

## 五、错误处理与调整体验

### 5.1 错误类型的分类

| 错误类型 | 描述 | 示例 |
|---------|------|------|
| 选择错误 | 选错了组件 | 选错了触发条件 |
| 配置错误 | 组件配置不正确 | 邮箱地址格式错误 |
| 逻辑错误 | 流程设计有漏洞 | 死循环、无限等待 |
| 权限错误 | 缺少必要的授权 | OAuth 未授权 |

### 5.2 撤销/调整设计原则

#### 原则 1：无处不在的撤销

- **快捷键**：`Cmd/Ctrl + Z` 撤销，`Cmd/Ctrl + Shift + Z` 重做
- **撤销历史**：显示可撤销的操作列表，支持任意回退
- **批量撤销**：可以一次撤销多步操作

#### 原则 2：组件级编辑

- **点击编辑**：点击组件打开配置面板
- **拖拽删除**：把组件拖出画布即可删除
- **热插拔**：可以在流程中间替换组件，数据流自动重连

#### 原则 3：渐进式调整

- **最小修改**：调整一个参数不需要重新配置整个流程
- **预览变更**：修改前显示变更将带来的影响
- **保存点**：允许用户保存多个版本，便于比较

### 5.3 调整交互设计

```
┌─────────────────────────────────────────────────────────────┐
│  [⟲ 撤销] [↩ 重做]  │  已修改 3 处  │  [放弃修改] [保存]   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   修改历史：                                                │
│   ─────────                                                │
│   • 14:32 - 将 "保存到 Dropbox" 改为 "保存到 Google Drive"  │
│   • 14:30 - 添加了 "发送 Slack 通知" 组件                  │
│   • 14:28 - 修改了 Gmail 过滤条件                          │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐   │
│   │ [撤销到此处]                                         │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 错误恢复策略

| 场景 | 策略 |
|------|------|
| 保存后发现问题 | 版本历史回退 |
| 测试运行失败 | 保留测试数据，定位问题后重试 |
| 运行时出错 | 发送通知 + 提供错误诊断 + 提供手动修复入口 |
| 组件失效（API 变更） | 检测并提示 + 提供迁移建议 |

## 六、No-Code/Low-Code 平台最佳实践

### 6.1 行业代表平台分析

| 平台 | 核心特点 | 值得借鉴的设计 |
|------|---------|---------------|
| **Zapier** | 简单直接，Zap 概念清晰 | 命名即描述，降低理解门槛 |
| **Make (Integromat)** | 视觉化流程图，模块化 | 实时数据流可视化，错误诊断强 |
| **n8n** | 开源+本地部署，节点式 | 自然语言创建工作流（AI Workflow Builder） |
| **Glide** | 移动优先，预设布局 | 渐进式复杂度，从模板开始 |
| **Bubble** | 高度可定制，复杂功能 | 可视化编辑器，属性面板 |

### 6.2 最佳实践清单

#### 1. 从模板/预设开始

**实践**：不要让用户从零开始

**案例**：
- Zapier 的 "Zap Templates" 提供 3000+ 预设
- Make 的 "Templates" 展示热门场景

**借鉴**：Doramagic 应提供"Skill 市场"，用户可以从模板一键安装，也可以基于模板修改

#### 2. 渐进式复杂度

**实践**：初学者用简单模式，高级用户用专业模式

**案例**：
- Bubble 提供 "Bubble for beginners" 和高级编辑器
- Glide 预设布局，用户只需填数据

**借鉴**：Doramagic 应有"简单模式"（填空题）和"高级模式"（完全自定义）

#### 3. 自然语言 + 可视化结合

**实践**：让用户描述需求，AI 生成初稿，人工调整细节

**案例**：
- n8n 的 AI Workflow Builder：用自然语言描述目标，生成工作流草稿
- Cursor：用自然语言生成代码片段

**借鉴**：Doramagic 应是"对话式创建 + 可视化调整"的混合模式

#### 4. 实时反馈与验证

**实践**：编辑时即时显示数据流和潜在问题

**案例**：
- Make 的 Scenario 运行日志详细到每一步
- Zapier 的 "Replay" 功能可重跑失败任务

**借鉴**：Doramagic 需要强大的调试/测试模式

#### 5. 社区与模板生态

**实践**：用户创建的自动化可以分享给他人

**案例**：
- Zapier 的 "Shared Zaps"
- Make 的 "Community Templates"

**借鉴**：Doramagic 应有"Skill 商店"，鼓励用户分享自己创建的 Skill

#### 6. 错误处理的人性化

**实践**：错误不仅是报错，更是教学机会

**案例**：
- Zapier 在失败时显示"什么是错的"和"怎么修"
- Make 提供错误步骤的详细日志

**借鉴**：每个错误提示都应该是"诊断 + 修复建议"的组合

### 6.3 对比表格：各平台 UX 亮点

| 维度 | Zapier | Make | n8n | 借鉴建议 |
|------|--------|------|-----|---------|
| 创建入口 | 对话式 | 画布式 | AI 对话 | 混合模式 |
| 学习曲线 | 较平缓 | 较陡峭 | 中等 | 提供渐进式引导 |
| 模板生态 | ★★★★★ | ★★★★☆ | ★★★☆☆ | 优先建设模板 |
| 错误处理 | ★★★☆☆ | ★★★★☆ | ★★★★☆ | 重点投入 |
| 协作功能 | 基础 | 基础 | 开源扩展 | 后续考虑 |

## 七、综合建议与设计原则

### 7.1 核心设计原则

基于本研究，我们提出 Doramagic Skill 创建的六大核心设计原则：

#### 原则 1：所见即所得（WYSIWYG）

- 可视化画布，拖拽即拼接
- 实时显示数据流和连接状态

#### 原则 2：低成本试错（Low-Cost Experimentation）

- 无限撤销，不害怕犯错
- 测试运行不影响真实数据
- 错误是最好的学习机会

#### 原则 3：渐进式披露（Progressive Disclosure）

- 初学者：填空题模式，选项都已准备好
- 进阶用户：自定义模式，完全控制
- 通过"显示更多选项"逐步展开复杂度

#### 原则 4：对话式创建 + 可视化调整

- 自然语言描述需求
- AI 理解后生成初稿
- 用户在画布上调整细节

#### 原则 5：即时验证与反馈

- 语法错误：编辑时即时提示
- 流程错误：保存前完整检查
- 运行错误：详细日志 + 修复建议

#### 原则 6：成就与社区

- 创建完成有庆祝动画
- 可分享到社区
- 可安装他人分享的 Skill

### 7.2 分阶段实施建议

#### 第一阶段（MVP）

- 支持 3-5 个常用触发器和动作
- 简单的线性流程（无分支）
- 基础模板市场
- 基本错误提示

#### 第二阶段（完善）

- 增加条件分支和循环
- 完整的版本历史
- 增强的测试模式
- 社区分享功能

#### 第三阶段（生态）

- AI 辅助优化建议
- 复杂工作流支持
- 多用户协作
- 开放组件市场

### 7.3 关键成功指标

| 指标 | 描述 | 目标（参考） |
|------|------|-------------|
| 创建完成率 | 用户开始创建后成功保存的比例 | > 60% |
| 首次成功运行率 | 新建 Skill 首次测试通过的比例 | > 70% |
| 模板使用率 | 用户使用模板而非从零创建的比例 | > 50% |
| 错误恢复时间 | 从错误到修复的平均时间 | < 5 分钟 |
| 分享率 | 用户分享自己创建 Skill 的比例 | > 20% |

## 八、结论

将"玩乐高"的体验应用到 Skill 创建，核心是将复杂的技术实现转化为直观、可视、可调整的交互过程。通过：

1. **自然语言输入**降低表达门槛
2. **智能推荐**减少选择困难
3. **可视化画布**让组合过程清晰可见
4. **三层验证**确保质量
5. **无处不在的撤销**鼓励试错
6. **模板生态**降低启动成本

我们有信心设计出让普通用户也能轻松创建定制化 Skill 的体验，让 Doramagic 成为真正的"乐高式 AI 能力组装平台"。
