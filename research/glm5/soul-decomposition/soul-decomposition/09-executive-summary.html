灵魂拆解（Soul Decomposition）研究摘要

研究日期: 2026-03-10 | 研究者: G大脑袋 + 8个子代理 (GLM-5)

一、研究背景

问题：大型项目（Linux、Kubernetes、VS Code）包含数百个模块，当前 soul-extractor 只能提取"整体灵魂"，导致： 1. 上下文过大，AI 无法有效加载 2. 粒度太粗，无法定位具体模块知识 3. 增量更新困难

目标：实现"灵魂拆解"能力——识别模块边界 → 提取每个模块的灵魂 → 形成模块化知识图谱 → 支持按需加载

二、核心研究成果

2.1 模块识别方法（4层框架）

层级 方法 核心技术
L1 代码结构分析 目录、命名空间、构建配置
L2 依赖图分析 Louvain/Leiden 社区检测
L3 职责边界识别 LCOM 度量、耦合度量化
L4 语义增强 注释、文档、命名分析

关键洞察：控制流和知识流方向相反——上游提供知识，下游消费知识

2.2 粒度控制（五层模型）

层级 名称 LOC 范围 Token 范围
L1 系统 100K+ 50K+
L2 子系统 10K-100K 5K-50K
L3 模块 1K-10K 500-10K ⭐
L4 组件 100-1K 50-500
L5 函数 <100 <50

黄金平衡区间：单模块灵魂 500-10,000 tokens

2.3 模块间关系建模（8种依赖）

关系类型 知识流向 强度权重
调用 上游→下游 0.30
继承 父→子 0.25
组合 容器→组件 0.20
接口 实现→接口 0.15
事件 发布→订阅 0.10

四维耦合公式：

Coupling = 0.30×Structural + 0.30×Behavioral + 0.25×Data + 0.15×Temporal

2.4 知识图谱结构

4类节点：Module、Concept、Rule、Code

4类边：DEPENDS_ON、CONTAINS、REFERENCES、CALLS

存储方案：Neo4j 主存储 + JSON 导出 + SQLite 缓存 + Markdown 归档

2.5 按需加载机制

5类触发器：关键词、文件路径、API调用、错误栈、组合触发

双层架构：描述预加载（常驻）+ 内容按需加载（懒加载）

Token 效率：Cursor 动态上下文减少 47% tokens

2.6 增量更新策略

变更类型 处理策略
单文件修改 局部更新该模块
核心接口变更 全量更新下游模块
超过 20% 文件变更 全量重新提取

2.7 现有工具调研

关键洞察：设计意图提取是业界空白

工具类型 代表工具 能力 缺失
代码分析 SonarQube 质量门禁 设计意图
语义分析 CodeQL 数据流分析 心智模型
知识图谱 Neo4j 图存储 自动提取

2.8 原型设计

架构：

源代码 → 模块识别器 → 灵魂提取器 → 知识图谱 → 索引文件

技术选型： - AST 解析：Tree-sitter - 社区检测：python-louvain - 知识图谱：Neo4j - 向量索引：FAISS

四、关键决策

决策点 选择 理由

五、风险与缓解

风险 缓解措施

模块边界不准 人工校准 + 配置覆盖
Token 超限 动态压缩 + 分层输出
知识图谱一致性 事务机制 + 自动修复

六、总结

灵魂拆解 是 soul-extractor 的自然延伸，解决大型项目的知识模块化问题。

核心价值： 1. 让 AI 能够"理解"大型项目的模块化结构 2. 支持按需加载，控制上下文大小 3. 增量更新，降低维护成本
