## 一、变更检测机制

### 1.1 Git Diff 检测
```bash
# 检测变更文件
git diff --name-only HEAD~1 HEAD

# 检测变更内容
git diff HEAD~1 HEAD --stat
```

### 1.2 文件监控
- **watchman**：Facebook 开发的文件监控工具
- **fsevents**：macOS 原生文件事件 API
- **inotify**：Linux 文件监控

### 1.3 变更类型分类
| 类型 | 影响 | 处理策略 |
|------|------|----------|
| 新增文件 | 可能新模块 | 触发模块识别 |
| 删除文件 | 可能删除模块 | 更新知识图谱 |
| 修改文件 | 内容变更 | 增量提取灵魂 |
| 重命名 | 模块边界变化 | 重新分析依赖 |

## 二、影响分析算法

### 2.1 依赖传播计算
```
变更文件 → 直接依赖者 → 间接依赖者 → 影响范围
```

### 2.2 影响评估公式
```
Impact = Σ(DirectDeps × 0.5 + IndirectDeps × 0.2)
```

### 2.3 算法流程
1. 解析变更文件的依赖关系
2. 计算下游影响范围
3. 标记受影响的模块灵魂
4. 确定需要重新提取的范围

---

## 三、增量提取策略

### 3.1 局部更新 vs 全量更新
| 场景 | 策略 |
|------|------|
| 单文件修改 | 局部更新该模块 |
| 核心接口变更 | 全量更新下游模块 |
| 配置文件变更 | 触发全量重新分析 |
| 超过 20% 文件变更 | 全量重新提取 |

### 3.2 增量提取流程
```
1. 检测变更 → 2. 影响分析 → 3. 标记受影响模块 
→ 4. 并行提取 → 5. 更新知识图谱 → 6. 验证一致性
```
