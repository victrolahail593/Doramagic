# 已知限制 (LIMITATIONS)

版本: calorie-tracker v0.1.0
评估日期: 2026-03-18

---

## 精度限制

### L-01: AI 营养估算误差 15-30%
- **描述**: LLM 估算食物营养数据的误差范围。常见单一食物（苹果、鸡蛋、米饭）约 10-15%；复合菜品（宫保鸡丁、麻辣烫）约 25-30%。
- **根因**: LLM 训练数据中的营养信息本身是统计平均值，且不知道具体烹饪方式和用料比例。
- **缓解**: 每条记录标注 confidence 等级；用户可修正数值；重量缩放机制让修正更便捷。
- **来源**: ACC (UNSAID-001), ONT (UNSAID-05), COM/LobsterLair (80-90%)

### L-02: 份量估算是精度瓶颈
- **描述**: 当用户不提供克重时，"一碗米饭"的默认估算可能偏差 50-100%（150g vs 300g）。
- **根因**: 碗/份/盘等容器大小因地区和个人差异巨大。
- **缓解**: 鼓励用户在有条件时使用厨房秤；照片可辅助估算但仍有限。
- **来源**: COM/LobsterLair

### L-03: MET 运动消耗估算偏差 20-30%
- **描述**: MET 值是群体平均值，同一运动因强度/体能/环境不同差异巨大。
- **根因**: 2011 Compendium 的 MET 值测量样本有限，且不考虑个体差异。
- **缓解**: 仅作为配额调整参考，不作为"可以多吃多少"的精确依据。
- **来源**: ONT (UNSAID-02)

### L-04: Mifflin-St Jeor 公式适用人群有限
- **描述**: 对 18-65 岁、BMI 18.5-30 的一般成人较准确。
- **不适用**: 运动员（肌肉量高 → BMR 被低估）、孕妇、18 岁以下、极端体重（BMI > 40）、老年人（>75 岁）。
- **缓解**: 在 profile 设置时如检测到可能不适用的情况（如 BMI < 18 或 > 35），给出提示。
- **来源**: ACC (K-004), ONT (RULE-01)

## 功能限制

### L-05: 无外部食物数据库集成
- **描述**: v0.1 完全依赖 LLM 估算，不接入 Open Food Facts、USDA FDC 等外部 API。
- **影响**: 品牌包装食品（如"可口可乐 330ml"）的精确营养数据需要外部数据库才能准确获取。
- **未来方向**: v0.2 可考虑接入 OFF API 用于条码扫描和品牌食品查找。
- **来源**: FY (CONCEPT-02), ONT (CONCEPT-02) — 两个项目都实现了多数据库融合

### L-06: 无条码扫描
- **描述**: 当前版本不支持条码扫描识别食物。
- **影响**: 包装食品需要用户手动描述，无法利用条码直接获取精确数据。
- **来源**: FY (WORKFLOW-01), ONT (RULE-04)

### L-07: 仅追踪宏量营养素 (P/C/F)
- **描述**: v0.1 只追踪卡路里、蛋白质、碳水、脂肪四项。不追踪维生素、矿物质、纤维、钠、水分等微量营养素。
- **理由**: 用户需求明确为"记录卡路里"，微量营养素增加复杂度但对目标用户价值有限。
- **来源**: FY (CONCEPT-04) 提供了 43 字段完整模型可供未来参考

### L-08: 无食谱管理
- **描述**: 不支持创建和复用食谱。每次需要重新描述。
- **影响**: 经常吃同一道菜的用户需要反复描述。
- **未来方向**: v0.2 可加入"常吃食物"快捷记录。
- **来源**: FY (CONCEPT-05) Recipe.unpack 提供了完整参考

### L-09: 无趋势分析和异常检测
- **描述**: 只有单日数据查看，无跨日/周/月趋势分析。
- **影响**: 用户无法看到"本周平均蛋白质摄入不足"类洞察。
- **来源**: COM 社区空白分析指出这是差异化机会

### L-10: 无多用户支持
- **描述**: profile.json 为单例，只支持一个用户。
- **来源**: ACC (UNSAID-004) 同样是单例 profile

## 平台限制

### L-11: 日志文件无并发控制
- **描述**: JSON 日志采用"读-改-写"整文件操作，多会话并发可能导致数据覆盖。
- **概率**: 低（单用户场景少有并发）。
- **缓解**: 无。v0.1 接受此风险。

### L-12: cron 依赖 OpenClaw 平台支持
- **描述**: 午餐/晚餐提醒和每日总结依赖 OpenClaw 的 cron 调度能力。如果平台不支持或配置不当，提醒功能失效。
- **缓解**: 用户可手动触发"每日总结"。

### L-13: 多模态能力依赖底层模型
- **描述**: 照片识别食物依赖 OpenClaw 底层 LLM 的多模态能力。如果使用不支持图片的模型，照片功能不可用。
- **缓解**: 始终支持文字输入作为 fallback。

## 数据安全限制

### L-14: 数据未加密存储
- **描述**: v0.1 的 JSON/Markdown 文件以明文存储在用户文件系统上。
- **对比**: FY 使用 NutrientValue 领域加密，ONT 使用 AES-256 Hive 加密，ACC 使用 OpenKBS 平台加密。
- **缓解**: 依赖用户设备自身的安全性（磁盘加密等）。

### L-15: 无数据导出/导入标准
- **描述**: 不支持 CSV 导出或从其他应用导入。
- **来源**: FY (UNSAID-04) 的 CSV header 设计暗示了互操作意图
