# 灵魂积木粒度研究计划

## 背景

Doramagic 的 Soul Extractor 从开源项目代码中提取隐性知识（WHY + UNSAID）。当前每次提取时，LLM 需要同时推断框架级知识和项目级知识，导致：
- 幻觉高发（LLM 可能"自信地记错"框架知识）
- 弱模型表现差（没有参考锚点，纯靠自由生成）
- 重复劳动（每个 Django 项目都重新推断 Django 的设计哲学）

**解决方案假设**：预提取热门框架/生态的"灵魂积木"（Lego Bricks），作为提取过程中的 RAG 参考知识，缩小 LLM 自由生成范围。

## 核心研究问题

**灵魂积木的最优粒度是什么？**

三个候选粒度：
- **框架级**（Django、React、Express）→ 通用性最高，可能太泛
- **框架+模式级**（Django ORM、Django REST Framework、React Hooks）→ 更精准
- **框架+领域级**（Django 用于电商、Django 用于 CMS）→ 最精准但组合爆炸

## 研究子问题拆解

### Q1: 开源生态的知识结构是什么？
- 框架、库、插件、模式之间的知识边界在哪里？
- 开发者在使用开源项目时，哪些知识是框架级的（通用的），哪些是项目级的（特有的）？
- 有没有自然的知识分层？（语言 → 框架 → 库 → 模式 → 项目）

### Q2: 现有 AI 开发工具如何处理框架知识？
- Cursor、GitHub Copilot、Sourcegraph Cody、Continue.dev 等工具是否有框架级知识的预置？
- RAG 系统中的 chunking 策略和粒度选择有哪些最佳实践？
- 知识图谱领域对"节点粒度"有什么研究？

### Q3: 什么粒度的参考知识最能减少幻觉？
- 学术研究中，RAG 的 chunk size 对生成质量的影响？
- 太粗（整个框架文档）vs 太细（单个函数签名）的 tradeoff？
- 有没有"知识密度最优点"的研究？

### Q4: 哪些框架/生态最值得优先预提取？
- 按 GitHub stars、npm/PyPI 下载量、Stack Overflow 问题数排序
- 哪些框架的 WHY 层知识最丰富（设计哲学明确）？
- 哪些框架的 UNSAID 层知识最多（社区踩坑密集）？

### Q5: 不同粒度的维护成本和可行性？
- 框架级积木数量（~50-100）vs 模式级（~500-1000）vs 领域级（~5000+）
- 框架更新频率对积木鲜度的影响
- 积木之间的依赖和组合问题

## 三方研究分工

### Track A: Codex — 工程与架构视角
重点：现有 AI 开发工具的框架知识架构、RAG 粒度工程实践、知识索引技术
产出：`reports/codex-report.md`

### Track B: Gemini — 产品与生态视角
重点：开源生态知识结构分析、热门框架价值排序、知识分层理论、维护成本模型
产出：`reports/gemini-report.md`

### Track C: Claude — 技术深度视角
重点：学术研究（RAG chunk 粒度、知识图谱节点粒度）、Doramagic 现有实验数据分析（哪些幻觉可被积木预防）、积木格式设计
产出：`reports/claude-report.md`

## 研究产出要求

每份报告必须包含：
1. **事实发现**（带来源链接）— 不要推测，要有据可查
2. **粒度推荐**（带理由）— 框架级/模式级/领域级，或混合方案
3. **Top 20 优先积木清单**（带排序理由）
4. **积木格式草案**（一个积木应该包含什么结构）
5. **风险和未知**（这个方案可能失败的原因）
