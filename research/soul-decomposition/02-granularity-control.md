研究日期: 2026-03-10 | 模型: GLM-5

## 一、粒度层级定义

### 1.1 五层架构模型

| 层级 | 名称 | 特征 | 代码规模 | 典型边界 |
|------|------|------|----------|----------|
| L1 | **系统 (System)** | 独立部署的整体应用 | 10万+ LOC | 仓库/部署单元 |
| L2 | **子系统 (Subsystem)** | 业务领域划分 | 1-10万 LOC | 目录/命名空间 |
| L3 | **模块 (Module)** | 单一职责的功能单元 | 1千-1万 LOC | Package/文件夹 |
| L4 | **组件 (Component)** | 可复用的构建块 | 100-1000 LOC | 类/结构体 |
| L5 | **函数 (Function)** | 原子操作单元 | 10-100 LOC | 方法/函数 |

### 1.2 各层级定义

**L1 系统 (System)**
- 定义：一个完整的、可独立运行和部署的软件实体
- 特征：拥有独立的生命周期、配置、数据存储
- 边界识别：独立仓库、独立部署流水线、独立数据库
- 灵魂特征：整体架构哲学、核心设计理念、技术选型原则

**L2 子系统 (Subsystem)**
- 定义：系统内按业务领域或技术层次划分的大型分区
- 特征：相对独立的数据模型、可独立测试、有清晰的接口边界
- 边界识别：顶层目录、DDD Bounded Context、微服务边界
- 灵魂特征：领域模型、业务规则、核心工作流

**L3 模块 (Module)**
- 定义：实现单一职责的代码集合
- 特征：高内聚、低耦合、可独立理解
- 边界识别：Package、文件夹、单一职责边界
- 灵魂特征：设计模式、实现策略、关键算法

**L4 组件 (Component)**
- 定义：可复用的代码构建块
- 特征：封装性、可测试性、接口清晰
- 边界识别：类、结构体、接口定义
- 灵魂特征：状态管理、协作模式、API契约

**L5 函数 (Function)**
- 定义：最小可执行单元
- 特征：单一输入输出、无副作用或副作用明确
- 边界识别：方法签名、代码块
- 灵魂特征：算法逻辑、边界处理、性能优化

### 1.3 层级间的关键关系

```
系统 (L1)
├── 子系统A (L2) ─── Bounded Context
│   ├── 模块A1 (L3) ─── 单一职责
│   │   ├── 组件A1.1 (L4)
│   │   │   └── 函数 (L5)
│   │   └── 组件A1.2
│   └── 模块A2
└── 子系统B (L2)
    └── 模块B1
```

**跨层级规则**：
1. 上层可以调用下层，下层不应直接调用上层
2. 同层级之间通过接口通信
3. 跨层级调用需要明确的契约

## 二、粒度权衡分析

### 2.1 过粗粒度的问题

| 问题 | 影响 | 案例 |
|------|------|------|
| **上下文溢出** | 超出 LLM 有效上下文窗口 | Linux 内核整体提取 → 300万+ tokens |
| **知识稀释** | 核心概念被噪音淹没 | Kubernetes 整体 → 真正的设计理念只占 5% |
| **定位失效** | 无法精确关联代码变更 | 修改一个函数却加载整个系统灵魂 |
| **增量困难** | 任何改动触发全量更新 | 改 10 行代码需要重新提取整个模块 |

**经验阈值**：
- 单模块灵魂 < 10,000 tokens（约 7000 字）
- 超过此阈值需要拆分为子模块

### 2.2 过细粒度的问题

| 问题 | 影响 | 案例 |
|------|------|------|
| **知识碎片化** | 概念被割裂，无法形成整体理解 | 按函数拆分 → 设计意图丢失 |
| **上下文重建成本** | 需要大量 tokens 重新组装 | 加载 100 个函数灵魂 > 加载 1 个模块灵魂 |
| **关系爆炸** | 模块间关系数量指数增长 | n 个模块 → n² 个可能关系 |
| **维护负担** | 小改动影响多个模块灵魂 | 重构一个类需要更新 20 个灵魂文件 |

**经验阈值**：
- 单模块灵魂 > 500 tokens（约 350 字）
- 低于此阈值应该合并到上层模块

### 2.3 黄金平衡点

**Sweet Spot 理论**：

```
粒度最佳区间 = [最小独立理解单元, 最大有效上下文]

最小独立理解单元：
- 一个开发者可以在 30 分钟内完全理解的代码单元
- 对应约 500-2000 LOC 或 500-2000 tokens 的灵魂

最大有效上下文：
- LLM 可以保持高质量推理的最大 tokens
- 当前最佳实践：4K-8K tokens（Claude/GPT-4 实测有效范围）
- 注意：虽然有 100K+ 上下文窗口，但质量在 10K 后显著下降
```

**粒度决策矩阵**：

| 因素 | 倾向更粗 | 倾向更细 |
|------|----------|----------|
| 内聚性 | 高（职责单一） | 低（职责分散） |
| 耦合度 | 低（独立性强） | 高（依赖复杂） |
| 变更频率 | 低（稳定） | 高（频繁改动） |
| 团队规模 | 小（统一理解） | 大（并行开发） |
| AI 使用场景 | 理解整体架构 | 定位具体实现 |

### 2.4 粒度自适应策略

```
IF 模块灵魂 tokens > 10000 THEN
    拆分为子模块
ELSE IF 模块灵魂 tokens < 500 THEN
    合并到父模块或相邻模块
ELSE
    保持当前粒度（最佳区间）
```

## 三、项目类型策略

### 3.1 Monorepo 粒度策略

**特征**：
- 多项目共享同一仓库
- 强依赖管理（Yarn Workspaces, Bazel）
- 统一的工具链和规范

**粒度策略**：

```
仓库根目录 (L1)
├── apps/                    # 应用层
│   ├── web-app/            # 独立系统 (L1)
│   └── api-server/         # 独立系统 (L1)
├── packages/               # 共享层
│   ├── ui-components/      # 模块 (L3)
│   ├── utils/              # 模块 (L3)
│   └── types/              # 模块 (L3)
└── libs/                   # 基础层
    └── core/               # 子系统 (L2)
```

**灵魂提取策略**：
1. **Package 级**：每个 `packages/*` 独立灵魂
2. **App 级**：每个 `apps/*` 独立灵魂
3. **跨包关系**：记录依赖图，不重复提取共享代码

**粒度决策点**：
- Package > 50 个文件 → 考虑拆分子 package
- Package < 5 个文件 → 考虑合并

### 3.2 Microservice 粒度策略

**特征**：
- 每个服务独立部署
- 服务间通过 API/事件通信
- 强调 Bounded Context

**粒度策略（DDD 驱动）**：

```
Bounded Context (L2) ─────→ 一个微服务
├── Aggregate (L3) ───────→ 服务内模块
│   ├── Entity (L4) ──────→ 领域对象
│   └── Value Object (L4) → 值对象
└── Domain Service (L3) ──→ 领域服务
```

**灵魂提取策略**：
1. **服务级**：每个微服务一个系统灵魂
2. **聚合级**：每个 Aggregate 一个模块灵魂
3. **事件级**：跨服务事件契约单独提取

**粒度决策点**：
- 服务代码 > 5万 LOC → 检查是否应该拆分服务
- 服务代码 < 1000 LOC → 检查是否应该合并

**关键洞察**：
> 一个微服务 ≈ 一个 Bounded Context ≈ 一个独立灵魂单元

### 3.3 Monolith 粒度策略

**特征**：
- 单一部署单元
- 模块间通过函数调用通信
- 可能演进为微服务

**粒度策略（分层架构）**：

```
应用 (L1)
├── 表现层 (L2)
│   ├── Controllers/ (L3)
│   └── Views/ (L3)
├── 业务层 (L2)
│   ├── Services/ (L3)
│   └── Domain/ (L3)
├── 数据层 (L2)
│   ├── Repositories/ (L3)
│   └── Models/ (L3)
└── 基础层 (L2)
    ├── Utils/ (L3)
    └── Config/ (L3)
```

**灵魂提取策略**：
1. **分层提取**：按架构层次拆分
2. **领域优先**：业务层优先，基础设施层次之
3. **演进感知**：标记可能拆分为微服务的边界

**粒度决策点**：
- 模块间耦合度 > 0.3 → 考虑重新划界
- 模块代码 > 1万 LOC → 强制拆分

### 3.4 策略对比总结

| 维度 | Monorepo | Microservice | Monolith |
|------|----------|--------------|----------|
| **L1 边界** | 整个仓库 | 单个服务 | 整个应用 |
| **L2 边界** | App/Package | Bounded Context | 架构层 |
| **L3 边界** | Package | Aggregate | 模块目录 |
| **灵魂数量** | 最多 | 中等 | 最少 |
| **提取复杂度** | 高（依赖复杂） | 中（边界清晰） | 低（结构简单） |
| **推荐粒度** | Package 级 | Service 级 | Layer 级 |

## 四、Token 预算模型

### 4.1 Token 预算分配公式

```
总预算 = 上下文窗口 × 有效利用率

有效利用率：
- Claude: 60-70% (100K 窗口, 有效约 60-70K)
- GPT-4: 50-60% (128K 窗口, 有效约 64-77K)
- 本地模型: 40-50%

模块灵魂预算 = 总预算 - 预留空间

预留空间：
- 系统提示词: ~2K tokens
- 用户查询: ~1K tokens
- 输出空间: ~4K tokens
- 缓冲区: ~1K tokens
─────────────────────
总计预留: ~8K tokens
```

### 4.2 粒度与 Token 关系模型

**基本公式**：

```
T(n) = T_base × n^α

其中：
- T(n) = n 个模块的总 token 数
- T_base = 基础开销（每个模块的固定 token）
- α = 粒度指数（0.8-1.2）

α < 1：粗粒度更省 token（知识复用）
α ≈ 1：线性关系
α > 1：细粒度更省 token（减少噪音）
```

**实际测算**：

| 项目规模 | 粗粒度(系统级) | 中粒度(模块级) | 细粒度(组件级) |
|----------|----------------|----------------|----------------|
| 小型(<1万LOC) | 5K tokens | 8K tokens | 15K tokens |
| 中型(1-10万LOC) | 50K tokens | 35K tokens | 80K tokens |
| 大型(>10万LOC) | 500K tokens | 150K tokens | 400K tokens |

**关键发现**：
- 小型项目：粗粒度最优
- 中大型项目：模块级粒度最优
- 超大型项目：需要分层策略

### 4.3 按需加载的 Token 优化

**懒加载模型**：

```
初始加载 = 核心模块灵魂（~5K tokens）

按需加载触发：
IF 用户查询涉及特定模块 THEN
    加载该模块灵魂（~2K tokens）
    
最大同时加载：
- 主模块: 1 个
- 依赖模块: ≤3 个
- 总计: ≤15K tokens（确保质量）
```

**预热策略**：

```
高频模块预加载：
- 统计用户查询模式
- 预加载最常访问的 3-5 个模块
- 缓存热点模块灵魂
```

### 4.4 Token 效率评估指标

```
效率 = 有效信息量 / 总 token 数

有效信息量定义：
- 核心概念: 每个概念 100 tokens
- 关键规则: 每条规则 50 tokens
- 设计决策: 每个决策 200 tokens
- 示例代码: 每个示例 300 tokens

目标效率：
- 核心模块: > 0.5（信息密度高）
- 工具模块: > 0.3
- 基础设施: > 0.2（信息密度低，可考虑不提取）
```

## 五、决策框架

### 5.1 粒度决策流程图

```
                    ┌─────────────────────┐
                    │ 识别代码单元边界     │
                    └──────────┬──────────┘
                               ▼
                    ┌─────────────────────┐
                    │ 评估内聚性 (LCOM)   │
                    │ LCOM < 2 → 高内聚   │
                    └──────────┬──────────┘
                               ▼
              ┌────────────────┴────────────────┐
              ▼                                  ▼
    ┌─────────────────┐               ┌─────────────────┐
    │ 高内聚单元       │               │ 低内聚单元       │
    │ 可作为独立模块   │               │ 需要拆分         │
    └────────┬────────┘               └────────┬────────┘
             ▼                                  ▼
    ┌─────────────────┐               ┌─────────────────┐
    │ 评估耦合度 (CBO) │               │ 按职责拆分       │
    │ CBO < 5 → 低耦合 │               │ 重新评估内聚性   │
    └────────┬────────┘               └─────────────────┘
             ▼
    ┌─────────────────────┐
    │ 计算 token 预算      │
    │ 估算灵魂大小         │
    └──────────┬──────────┘
               ▼
    ┌─────────────────────┐
    │ 500 < tokens < 10000 │
    │       ↓ 是           │
    │ 确定粒度合适         │
    │       ↓ 否           │
    │ 调整粒度（合并/拆分）│
    └─────────────────────┘
```

### 5.2 快速决策检查表

**P0 - 必须拆分**：
- [ ] 模块代码超过 10,000 行
- [ ] 预估灵魂超过 15,000 tokens
- [ ] LCOM > 5（严重缺乏内聚）
- [ ] CBO > 10（严重耦合）
- [ ] 多个独立业务领域混在一起

**P1 - 建议拆分**：
- [ ] 模块代码 5,000-10,000 行
- [ ] 预估灵魂 8,000-15,000 tokens
- [ ] LCOM 3-5
- [ ] CBO 5-10
- [ ] 变更频率差异大的代码在同一模块

**P2 - 可选拆分**：
- [ ] 模块代码 1,000-5,000 行
- [ ] 预估灵魂 3,000-8,000 tokens
- [ ] 有清晰的子结构但耦合较低
- [ ] 预期未来会独立演进

**保持现状**：
- [ ] 模块代码 < 1,000 行
- [ ] 预估灵魂 < 3,000 tokens
- [ ] 高内聚低耦合
- [ ] 单一职责清晰

### 5.3 项目类型适配

```yaml
Monorepo:
  默认粒度: Package 级
  拆分触发: Package > 50 文件 或 tokens > 10K
  合并触发: Package < 5 文件 或 tokens < 500
  特殊处理: 共享包需要记录被引用关系

Microservice:
  默认粒度: Service 级 + Aggregate 级
  拆分触发: 服务内聚合边界不清晰
  合并触发: 服务代码 < 1000 行
  特殊处理: 跨服务事件需要独立提取

Monolith:
  默认粒度: Layer 级
  拆分触发: 层内模块职责混乱
  合并触发: 层内模块强依赖
  特殊处理: 标记未来微服务拆分边界
```

## 六、优先级建议

### 6.1 P0 - 核心能力（必须实现）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0-1 | **Token 估算器** | 在提取前预估灵魂大小，自动判断粒度 |
| P0-2 | **内聚性分析** | 计算 LCOM，识别低内聚模块 |
| P0-3 | **边界识别器** | 基于目录、依赖、职责自动识别模块边界 |
| P0-4 | **粒度调整器** | 自动拆分过粗模块、合并过细模块 |

### 6.2 P1 - 重要能力（推荐实现）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P1-1 | **耦合度分析** | 计算 CBO，识别高耦合模块 |
| P1-2 | **项目类型检测** | 自动识别 monorepo/microservice/monolith |
| P1-3 | **变更影响分析** | 基于 git diff 分析哪些模块灵魂需要更新 |
| P1-4 | **按需加载引擎** | 根据用户查询动态加载相关模块灵魂 |

### 6.3 P2 - 增强能力（可选实现）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P2-1 | **粒度可视化** | 展示模块粒度分布，帮助人工决策 |
| P2-2 | **历史优化建议** | 基于使用历史优化粒度策略 |
| P2-3 | **跨项目知识复用** | 相似模块的灵魂可以复用 |
| P2-4 | **粒度演进追踪** | 记录粒度变化历史，支持回滚 |

### 6.4 实施路线图

```
Phase 1 (MVP - 2周):
├── P0-1 Token 估算器
├── P0-3 边界识别器（目录级别）
└── 手动粒度调整

Phase 2 (完善 - 2周):
├── P0-2 内聚性分析
├── P0-4 粒度调整器
└── P1-2 项目类型检测

Phase 3 (智能 - 3周):
├── P1-1 耦合度分析
├── P1-3 变更影响分析
└── P1-4 按需加载引擎

Phase 4 (优化 - 2周):
├── P2-1 粒度可视化
└── P2-2 历史优化建议
```

## 附录：关键指标参考值

### A. 代码度量指标

| 指标 | 公式 | 健康值 | 警戒值 |
|------|------|--------|--------|
| LCOM (Lack of Cohesion) | 不连通分量数 | < 2 | > 5 |
| CBO (Coupling Between Objects) | 依赖类数量 | < 5 | > 10 |
| LOC (Lines of Code) | 代码行数 | 500-5000 | > 10000 |
| CC (Cyclomatic Complexity) | 圈复杂度 | < 10 | > 20 |

### B. Token 预算参考

| 场景 | 推荐预算 | 上限 |
|------|----------|------|
| 单模块灵魂 | 2-5K tokens | 10K |
| 多模块加载 | 10-15K tokens | 20K |
| 完整系统概览 | 5-8K tokens | 15K |

### C. 常见项目粒度示例

| 项目 | 类型 | 推荐粒度 | 预估模块数 |
|------|------|----------|------------|
| CLI 工具 | Monolith | 功能模块 | 3-5 |
| Web 应用 | Monolith | 层+模块 | 10-20 |
| 微服务系统 | Microservice | 服务级 | N 个服务 |
| 前端 Monorepo | Monorepo | Package 级 | 10-50 |
| 大型后端 | Monorepo | 服务+Package | 50-200 |
