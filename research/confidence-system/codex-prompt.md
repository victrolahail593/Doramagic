# Codex 研究指令：知识可信度系统设计

## 背景

Doramagic 是一个运行在 OpenClaw 上的知识提取系统。它的核心引擎 Soul Extractor 从开源项目中提取六层知识（领域知识、操作知识、经验知识、决策知识、设计哲学 WHY、隐性知识 UNSAID），最终编译成一个 AI 道具（OpenClaw Skill）。

**核心矛盾**：提取的知识中，事实层（文件结构、依赖、命令）可以确定性验证，但解读层（设计哲学、决策理由、社区经验）由 LLM 生成，可能包含幻觉。用户（尤其是非技术用户）无法区分"真知识"和"LLM 编的"。

## 问题

我们最初设计了 A/B/C/D 置信度分级：
- A：创作者明确说明
- B：社区证据支持
- C：代码模式推断
- D：纯 LLM 推理

在第一性原理审查中，这个方案被指出"不可操作"：
1. LLM 给自己打置信度不可靠（不知道自己错的人不会给自己打低分）
2. 缺少判定流程、冲突仲裁、标注一致性检查
3. 主观评级无法工程化验证

我们提出了一个替代方向："从打分变成举证"——不评"有多可信"，而是标"有什么证据"：
- `[CODE]` — 代码中直接可观察
- `[DOC]` — 项目文档中明确说明
- `[COMMUNITY]` — Issues/讨论/博客中提到
- `[INFERENCE]` — LLM 从代码模式推断

## 你的任务

用第一性原理方法，为 Doramagic 设计一个**可落地的知识可信度系统**。

具体回答以下问题：

### 1. 评估现有方案
- A/B/C/D 分级方案的根本缺陷是什么？有没有办法修复？
- "证据标签"方案是否更好？它的缺陷和盲区是什么？
- 有没有第三种方案？

### 2. 工程实践调研
- 现有 AI 系统（RAG、知识图谱、搜索引擎）如何处理知识可信度？
- 有没有可借鉴的工程模式？（如 Google Knowledge Graph 的置信度、Wikipedia 的引用机制、学术论文的证据等级）
- 开源 AI 工具（Cursor、Copilot、Continue）有没有做知识可信度标注？

### 3. 设计可落地方案
- 给出你推荐的方案（可以是修改版、混合版或全新方案）
- 方案必须满足：
  - **不依赖 LLM 自评**（LLM 不能给自己打分）
  - **可工程化验证**（能写自动化检查）
  - **消费端可用**（Knowledge Compiler 能基于它做过滤规则）
  - **用户端可理解**（非技术用户能感知到"这条靠谱/这条不确定"）
- 给出具体的 schema、判定规则、过滤策略

### 4. 与 Doramagic 架构的协调
- 你的方案如何嵌入现有管线？（Stage 0 → Stage 1-3 → 验证关卡 → Knowledge Compiler → Stage 4）
- 哪些步骤是确定性的？哪些需要 LLM？
- 实施优先级和验证策略

## 约束

- **第一性原理**：所有设计必须从根本事实推导，不依赖类比或惯例
- **可落地**：不要给理论框架，给可实现的工程方案
- **简单优先**：能用简单方案解决就不要复杂化
